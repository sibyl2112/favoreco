# レビュー観点（事前レビュー / 精査用チェックリスト）

> **役割**: 仕様書（②-A/②-B）と実装指示書（③）を、実装前に精査するためのチェックリスト。主に Codex の精査役が使う。
> **使い方**: 実装に入る前にこのリストを通す。実装後レビューでも再利用してよい。
> **最終更新**: （新規アプリ作成時に更新）

> 📌 ねらい: 過去に手戻りを生んだ論点を毎回チェック項目として通し、AI間の往復を減らす。

---

## A. ロジック・データ（②-A仕様の精査）

- [ ] 状態の定義は網羅的か（取りうる状態がすべて挙がっているか）
- [ ] **状態の正本が1つに定まっているか**（複数箇所から推測していないか）
- [ ] **集計の母数が明確か**（何を数え、何を除外するか）
- [ ] **異なる母数を混ぜていないか**
- [ ] 割合（成功率など）の分母・分子が定義されているか。0除算の扱いは決まっているか
- [ ] 並び順が一意に決まるか（同値時の2次・3次キーまで定義されているか）
- [ ] データ移行ルールがあるか（既存データに新フィールドが無い場合の初期値）

## B. デザイン（②-B仕様の精査）

- [ ] 使う色・フォントがすべて既存のデザインシステム定義に紐づいているか（新規色を勝手に増やしていないか）
- [ ] 各状態に対する見え方（強調/抑制）が定義されているか
- [ ] 空状態（0件）の見せ方・文言が決まっているか
- [ ] アニメーション/ジェスチャーが他のシステム動作（画面端スワイプ等）と干渉しないか

## C. 構造設計（②-Cの精査）

- [ ] 「育つ場所」が予測され、分割単位が事前に切られているか
- [ ] View / ロジック / データ の境界が引かれているか（集計を View に直書きしていないか）
- [ ] 同じロジック・同じUIを2箇所以上に持たせていないか
- [ ] 1つの型が「表示+集計+状態+保存」を抱えすぎていないか
- [ ] 将来 500〜600行を超えそうなファイルに分割方針があるか

## D. 指示書（③の精査）

- [ ] 「やること」と同じ密度で「やらないこと」が書かれているか
- [ ] 対象ファイルが必要最小限に絞られているか（広げすぎていないか）
- [ ] 受け入れ条件が検証可能な形か（なぞって確認できるか）
- [ ] 改修かリファクタか明示されているか（挙動を変えるか変えないか）

## E. テスト・検証

- [ ] テストすべきロジックが洗い出されているか（母数・割合・並び順・空/0件/1件分岐）
- [ ] そのロジックが View から分離され、テスト可能な形になっているか
- [ ] 実機でしか確認できない項目（見た目・ジェスチャー）が分けて挙げられているか

## F. データ安全性（永続化を伴う場合）

- [ ] スキーマ変更を含むか。含むなら移行ルールとロールバック手段があるか
- [ ] **ストア自動削除系のフォールバックを増やしていないか**（スキーマ不一致や初期化失敗時に既存データを消さない）
- [ ] ユーザー確認なしに記録データを消す経路がないか

## G. 性能

- [ ] 一覧はサムネイル前提か（フルサイズ画像を毎回同期デコードしていないか）
- [ ] 重い処理（リモートfetch・画像生成等）を画面表示の同期パスで実行していないか

> ※ F / G はプロジェクトの技術スタックに合わせて観点を追加・削除する。

## 意見が割れたときの扱い

精査役と実装役（または自分）で判断が割れた場合は、**採否と理由を `docs/project-log.md` に決定ログとして残す**。同じ論点で再度往復しないため。
