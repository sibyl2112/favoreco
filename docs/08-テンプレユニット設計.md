# テンプレユニット設計（草案・②-A仕様の前段）

> 作成日: 2026-07-05
> ステータス: 草案（ユーザーレビュー待ち）
> 方針（2026-07-05 ユーザー指示）: **テンプレは項目ユニットの組み合わせで構成し、カスタムカテゴリ作成時に同じユニットを再利用できるようにする。基本テンプレート6種もユニットの組み合わせとして定義する。**

---

## 1. 設計原則

1. **スキーマはスーパーセット固定・動的スキーマは作らない**。全ユニットのフィールドをVisit/Eventが最初から持ち、RecordCategoryは「どのユニットを表示・入力するか」の構成データだけを持つ。SwiftDataで動的スキーマは危険であり、CloudKit互換（docs/07）も保てる
2. **入力していないユニットは表示に出さない**（空欄を並べない）。Mystoriumの「存在＝入力済み」原則（docs/04 §3-1）をUIにも適用する。これが「美しく記録する」の実装面の担保——記録画面も閲覧画面も、その人が使う項目だけで構成される
3. **統計・検索はユニット単位で実装する**。同じユニットを使うカテゴリ同士は同じ統計が自動で成立し、カテゴリ横断統計（「今年の全体験」）が自然に出る
4. 完全自由のカスタム項目は「スペック表ユニット」（ラベル＋値のペア配列）で受ける。ユニット追加＝アプリ更新、項目追加＝ユーザー操作、と役割を分ける
5. **日付・期間の入力はカレンダー式ピッカー**（graphical）。日付ホイール（ダイヤル）は使わない。期間（会期・販売期間等）は同一カレンダー上で開始日→終了日をタップして範囲選択する（2026-07-05 ユーザー指定）

## 2. 土台（全カテゴリ共通・外せない）

| 項目 | 備考 |
|---|---|
| タイトル | |
| 日付（開始・終了） | 日跨ぎ対応（Mystorium V8踏襲）。ステータス（気になる/予定/済み）は日付から自動推測 |
| 写真（複数） | ファイルパス正本 |
| **アイキャッチ** | **全カテゴリ標準**（2026-07-05 ユーザー指示）。写真から選択 or 単独設定。**表示サイズ・縦横比はカテゴリごとに設定**（テンプレにプリセット付属・カスタムでも比率セットから選択）。比率セット案: 1:1 / 3:4縦 / 2:3縦（書影・ポスター） / A4縦1:√2（チラシ） / 4:3横 / 16:9横 |

| **SNSトグル** | X / Instagram / Threads を各サービス別トグルでオンオフ追加できる。2026-07-05確定。観劇の「公式X/insta」を一般化し全テンプレ共通の土台項目に |
| メモ（文字数無制限） | 05リサーチ「無制限自由記述」の実証パターン |
| 総合評価 | RESONANCE型（詳細は②-Aで） |

| 観劇 | 美術展 | ライブ | 映画 | 酒 | テーマパーク | 書籍 |
|:-:|:-:|:-:|:-:|:-:|:-:|:-:|
| A4縦（チラシ） | A4縦（チラシ） | 1:1正方形（ジャケ写・2026-07-05変更） | 2:3縦（ポスター） | 3:4縦（ボトル） | 4:3横 | 2:3縦（書影） |

> ※比率はトリミングで選べる（初期値の話）。「スマホ縦撮り＝3:4では」という指摘（2026-07-05）に対し、**プリセットは対象物比率のまま維持**と決定——チラシ/ポスター/書影を取り込む前提の比率。自分で撮った写真は3:4だが、トリミング枠で対象物比率に合わせて収める運用。

## 4-10. 参照DBの配信方式と課金（2026-07-06・方針転換／課金枠は要確定）

> 御朱印・博物館系・城・船（および施設）の場所DBは、**アプリ同梱プリセットをやめ、「テンプレをONにしたらそのカテゴリの参照DBをダウンロード」する方式**に変更（ユーザー指示）。ここを**課金の柱**にしたい意向。

- **配信＝CloudKit公開データベース（Public Database）**：別サーバー不要（Apple内・アプリのCloudKitコンテナの公開DB）。テンプレON時に取得→端末キャッシュ。サーバー側にスポットを追加すれば端末が再取得＝**あとから増やせる**。同期(v2)で使うprivate DBとは別枠。
  - **原則からの逸脱を明示**：「完全ローカル・サーバーなし」から一歩踏み出す（Apple hostedの静的参照データ）。純ローカル原則の例外として扱う。
- **守る線（絶対）**：**「記録する」機能は無料**。Apple Maps POIでどのスポットも検索・記録できる。課金するのは参照DBの便利さ（プリセット/コンプ/エンリッチ）だけ。「記録は買い切りで全部入り／端末内データは守る」と矛盾させない。
- **課金で売る価値**：①有名どころが入って選ぶだけ ②コンプ機構（100名城 ◯/100、御船印参加社 ◯社、都道府県マップ塗り＝スイログの全国制覇の楽しさ）③将来のエンリッチ（由緒・御朱印の種類・授与情報）。
- **課金枠＝要確定（A/B/C）**：(A)サブスク込み（継続更新＝継続課金に整合）／(B)カテゴリDBパック単体販売（御朱印DB・100名城DB・御船印DB のアドオン買い切り）／(C)買い切り全部入りに含める。**現状の推奨は(A)＋(B)併用**、ユーザー確認待ち。
- **正直なコスト**：増え続けるDBは「コンテンツを作り続ける仕事」（100名城・御船印参加社・寺社・博物館の整備・更新）。サーバー代はほぼゼロでも整備が継続コスト＝**運営を続ける覚悟が要る軸**。課金はそれを賄う位置づけ。
- **スコープ**：v1＝Apple Maps POIで記録（DB無しでも全スポット可）＋**場所マスターに「DL可能な参照DB」の下地だけ通す**（後付けを重くしない）。CloudKit公開DB＋コンプ＋課金の実装は**v1.x/v2**。

**残り1テンプレ（書籍）のヒアリングは未保存分の復元が必要**（前セッションでISBN取得等を議論したが保存されず消失）。テーマパークは4-7「おでかけ施設」に統合済み。
