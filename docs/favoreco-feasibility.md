## 1. 主要エンティティ（素描）

| エンティティ | 持つ情報 | 主な関係 |
|------------|---------|---------|
| **RecordCategory** ★新規 | 名前・アイコン（SF Symbol）・カラー・並び順・テンプレート由来フラグ | 各マスターとEventの親。**Mystoriumの固定enumをこのモデルに置き換えるのが本質的な仕事** |
| Event（体験） | タイトル・カテゴリ参照・会場スナップショット・ステータス（日付から自動推測） | RecordCategory / Venue(nullify) / Visit(cascade) |
| Visit（回） | 日時・写真パス[String]・感想・成果・座席等のテンプレ別項目・タグ/同行者[String] | Event / AxisScore(cascade) |
| Venue / Organizer | 場所・主催のマスター（削除時nullify＋スナップショット表示） | Event |
| マスター群 | AxisDefinition（評価軸）/ OutcomeOption（成果）/ FormatMaster（形態）/ Tag / Companion | 親を enum → RecordCategory 参照に差し替えて流用 |
| **横断コレクション項目** ★新規 | 半券・印・カード・スタンプの写真＋メモ（Visit配下 or 独立） | Visit（設計は②-Aで確定） |

## 2. データ保存方針

- **ローカルファースト**（SwiftData正本・サーバーなし・運営コストゼロ）。Mystorium踏襲
- **写真は `PhotoBlob`（SwiftData・`@Attribute(.externalStorage)`）方式を最初から採用**（`docs/09-CloudKit写真ストレージ仕様.md` が正本。Mystorium V10実機検証済み）。画面側はrelativePath文字列のみ扱う（PhotoStorage API層で分離）。動画は `video_` プレフィックスでファイル保存・同期対象外。byteCountを別プロパティに持ち、存在チェック・容量集計でdataをフォールトさせない
- v1はローカルのみ＋zipバックアップ（`DataExporter` 流用。**externalStorageの `_SUPPORT` ディレクトリ同梱必須**・インポート後は再起動必須——09 §7）。**CloudKit同期はv2のサブスク機能**だが、v1スキーマをCloudKit互換3条件（全プロパティにデフォルト値／unique禁止・fetch-first upsert／リレーション全optional）で設計する（後付け不可のため）
- スキーマバージョニングはMystoriumのVersionedSchema＋lightweight migration運用をV1から採用

## 3. 技術的な山場

1. **カテゴリのデータ駆動化**: EventCategory（固定enum）→ RecordCategory（SwiftDataモデル）。カテゴリ別マスター4系統（形態・評価軸・成果・ticker設定）の親付け替え
2. テンプレートのプリセット注入（6種×評価軸・成果・形態のデフォルト一括生成。`SampleDataSeeder`/version-gated reconciliation が参考）
3. カテゴリ横断統計（「今年の全体験」ビュー。Mystoriumはカテゴリ内統計のみ）
4. 買い切りIAP（StoreKit 2。Mystorium Phase 2 実装を先行例に）

## 4. 技術スタック（選定理由）

- SwiftUI / SwiftData / MapKit / ImageIO / UserNotifications・外部ライブラリゼロ（Mystorium準拠）
- 既存資産に寄せた点: 画像パイプライン・シェアカード・統計・検索・カレンダー・エクスポート・マイグレーション運用をそのまま流用（docs/04 §5）
- 新規に増やす技術: なし（山場はすべて既存スタック内の設計問題）

## 4-2. 地図・場所入力の方針（2026-07-05 確定）

- **場所の入力は3系統**: ①アプリ内検索（MKLocalSearch＋補完） ②Google Mapsの共有URL（共有シート受け取り or ペースト） ③Apple Mapsの共有URL（同）
- Google URLからは**住所・座標を自動抽出しない**（Places規約: 取得データの永続保存は不可・Place IDのみ永続可。記録アプリの「一生保存」と根本的に非互換のため Places API 自体を採用しない）。名称はリンクのOGPタイトル（LinkPresentation）で取得し、**座標はApple側（CLGeocoder/MKLocalSearch）で解決**して保存する——「特定はGoogleの精度、保存はAppleデータ＋リンク」の分業
- Venueマスター拡張: 名称・住所・座標・GoogleMapリンク・AppleMapリンク・手動ピン調整
- 参照: Google Places API Policies（キャッシュ30日制限）・2025年3月の無料枠改定

## 4-3. カレンダー機能の方針（2026-07-05 範囲・時期とも確定）

**範囲＝A＋B（両方v1）。C（favoreco内での汎用予定作成）はやらない**（コンセプト§5に明記）。

- **A: 体験特化カレンダーのGoogle式表示**
  - 月表示: 日セルにカテゴリ色のイベントチップ（色＝`RecordCategory.colorHex`）
  - 週/日表示: **時間軸グリッドに予定ブロックが浮くGoogle式を完全再現**（2026-07-05確定・新規実装）。ブロックの高さは開演＋上演時間セレクトから導出した終演時刻で決まる
  - **終日・期間ものの描き分け（2026-07-05 モックで確定・Googleの「終日帯」積み上げを廃止）**:
    1. 期間の状態（販売期間・会期）＝**期間バー**（セル下部を走る高さ約11pxの連続バー・カテゴリ/種別色・**バー内に白抜きタイトル**を開始日と各週の左端に表示）。複数重なればバーを下に積む（最大2段＋あふれ「+n」）。グリッド・チップ領域を占有しない
    2. その日に起きる点（チケット発売日＝販売期間の開始日時）＝枠線チップ「発売 10:00」で開始点だけ強調。発売アラームはこの日時に発火
    3. 通常イベント＝カテゴリ色チップ・**タイトル2行まで**折り返し・入り切らない件数は「+n」→日別リスト
    4. 本当の終日体験（フェス全日・遠征）＝月表示は日付下の**グレー枠線1行チップ**、週/日表示は折りたたみピル1行（「終日 ▸ n件」）
    5. 参加済みの記録＝薄めのカテゴリ色チップ（✓）。「記録」レイヤーでon/off
    6. 月グリッドは7列完全均等（中身に列幅を引っ張らせない）
  - Liquid Glass基調（コンセプト§6）: 浮遊するビュー切替ピル・ガラスパネル・外部予定は低彩度/破線縁
  - **レイヤーのオン/オフ**: ①カテゴリごと（カテゴリ色）②種別（予定/記録(済み)/チケット発売・販売期間）③外部カレンダー（カレンダー単位）。「記録」レイヤーONで過去のカレンダーが思い出で埋まる＝アーカイブの見返しを兼ねる
- **B: 外部カレンダー重ね表示（EventKit一本でApple/Google両対応）**
  - iOS設定でGoogleアカウントを繋いだユーザーのGoogleカレンダーは標準カレンダーDBに同期されるため、EventKit読み取りで両方を**薄く**（低彩度・favorecoの体験が主役の階層を守る）重ね表示。OAuth・サーバー・Google API契約は不要、データは端末外に出ない
  - 逆方向（体験予定→標準カレンダーへの書き出し）も同じEventKitで対応
  - 権限: iOS 17以降のカレンダーアクセス許可フロー（読み取りはフルアクセス必要）

## 5. コード共有方式（要決定）

- 推奨（03の現実解）: **まずフォークでPoC → コンセプト検証後に共有コア（Swift Package）切り出しを判断**
- 理由: 共有コアの設計を最初にやると初速が落ちる。RecordCategory化という構造変更を試す段階では、Mystoriumのコードを自由に壊せるフォークが速い

## 6. 致命リスク（1つ）と検証方法

- **リスク**: RecordCategory化（enum→モデル参照）が、SwiftDataのリレーション・マイグレーション・既存UIパターン（ホームのカテゴリタブ・カテゴリ別マスター管理）と噛み合わない場合、Mystorium資産の流用前提が崩れ企画の工数見積りが崩壊する
- **検証方法（実装の最初のステップ＝PoC）**:
  1. RecordCategory＋Event＋マスター1系統（評価軸）だけの最小スキーマを組む
  2. テンプレ6種のプリセット注入 → ホームのカテゴリタブが動的に生える → 自作カテゴリの追加/リネーム/削除（nullify・記録は消さない）が破綻しないことを確認
  3. カテゴリ削除時の既存記録の扱い（「未分類」へ退避 等）をこの段階で設計確定
- **判定**: PoCで上記が素直に組めれば企画続行。組めなければ共有方式・スキーマ設計を再検討
