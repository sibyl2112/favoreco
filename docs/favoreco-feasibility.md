## 1. 主要エンティティ（素描）

| エンティティ | 持つ情報 | 主な関係 |
|------------|---------|---------|
| **RecordCategory** ★新規 | 名前・アイコン（SF Symbol）・カラー・並び順・テンプレート由来フラグ | 各マスターとEventの親。**Mystoriumの固定enumをこのモデルに置き換えるのが本質的な仕事** |
| Event（体験） | タイトル・カテゴリ参照・会場スナップショット・ステータス（日付から自動推測） | RecordCategory / Venue(nullify) / Visit(cascade) |
| Visit（回） | 日時・写真パス[String]・感想・成果・座席等のテンプレ別項目・タグ/同行者[String] | Event / AxisScore(cascade) |
| Venue / Organizer | 場所・主催のマスター（削除時nullify＋スナップショット表示） | Event |
| マスター群 | AxisDefinition（評価軸）/ OutcomeOption（成果）/ FormatMaster（形態）/ Tag / Companion | 親を enum → RecordCategory 参照に差し替えて流用 |
| **横断コレクション項目** ★新規 | 半券・印・カード・スタンプの写真＋メモ（Visit配下 or 独立） | Visit（設計は②-Aで確定） |
| **Brewery（酒蔵）** ★新規 | 酒蔵名・**所在県**・URL(公式/EC)・スナップショット | 酒テンプレのEvent（銘柄）。Venue(飲んだ店)とは別。所在県マップの集計元。Organizerのマスター型を流用 |
| **場所マスター（汎用・種別付き）** ★新規 | 名称・**種別**（施設=テーマパーク/水族館/動物園、印の場所=寺社/城/港…）・地域・**正確な住所/座標**・運営・URL・スナップショット | Brewery/施設/寺社城港を包む共通型。**参照DB（CloudKit公開DB・テンプレONでDL）＋地図集計＋コンプ**の共通部品。記録時にVisitへスナップショット（DB非依存）。08 §4-7/§4-8/§4-10 |

## 2. データ保存方針

- **ローカルファースト**（SwiftData正本・サーバーなし・運営コストゼロ）。Mystorium踏襲
- **写真は `PhotoBlob`（SwiftData・`@Attribute(.externalStorage)`）方式を最初から採用**（`docs/09-CloudKit写真ストレージ仕様.md` が正本。Mystorium V10実機検証済み）。画面側はrelativePath文字列のみ扱う（PhotoStorage API層で分離）。動画は `video_` プレフィックスでファイル保存・同期対象外。byteCountを別プロパティに持ち、存在チェック・容量集計でdataをフォールトさせない
- v1はローカルのみ＋zipバックアップ（`DataExporter` 流用。**externalStorageの `_SUPPORT` ディレクトリ同梱必須**・インポート後は再起動必須——09 §7）。**CloudKit同期はv2のサブスク機能**だが、v1スキーマをCloudKit互換3条件（全プロパティにデフォルト値／unique禁止・fetch-first upsert／リレーション全optional）で設計する（後付け不可のため）
- スキーマバージョニングはMystoriumのVersionedSchema＋lightweight migration運用をV1から採用

## 3. 技術的な山場

1. **カテゴリのデータ駆動化**: EventCategory（固定enum）→ RecordCategory（SwiftDataモデル）。カテゴリ別マスター4系統（形態・評価軸・成果・ticker設定）の親付け替え
2. テンプレートのプリセット注入（6種×評価軸・成果・形態のデフォルト一括生成。`SampleDataSeeder`/version-gated reconciliation が参考）
3. カテゴリ横断統計（「今年の全体験」ビュー。Mystoriumはカテゴリ内統計のみ）
4. 買い切りIAP（StoreKit 2。Mystorium Phase 2 実装を先行例に）

## 4. 技術スタック（選定理由）

- SwiftUI / SwiftData / MapKit / ImageIO / UserNotifications・外部ライブラリゼロ（Mystorium準拠）
- 既存資産に寄せた点: 画像パイプライン・シェアカード・統計・検索・カレンダー・エクスポート・マイグレーション運用をそのまま流用（docs/04 §5）
- 新規に増やす技術: なし（山場はすべて既存スタック内の設計問題）

## 4-1b. ラベルOCR（写真からのテキスト読み取り・2026-07-05 方針）

- **Vision（VisionKit）iOS標準の日本語OCR** を使用。外部サービス・API契約なし・オフライン・サーバー送信なし（ローカルファースト整合・外部ライブラリゼロ方針維持）
- 用途: 酒のラベル表裏（U16）・書籍の奥付等。撮影→認識テキストを候補提示→タップで銘柄名/スペック表/メモへ流し込む**半自動**
- 線引き: 定型の裏ラベル（精米歩合/日本酒度/酸度/アルコール分）は拾いやすい。装飾書体の銘柄名・筆文字は誤認識ありうるため**完全自動の項目振り分けはv2**。v1は「読み取り→候補→ユーザー確定」に留める（失敗しない設計）

## 4-1c. 酒蔵所在県マップ（2026-07-05 v1確定）

- **Brewery（酒蔵）マスターの所在県を集計し、日本地図を県別に塗り分け**（飲んだ数のヒートマップ）。「自分はどこの酒が好きか」の可視化＝酒テンプレのキラー機能
- MapKit資産＋県ポリゴン塗り分け（meguri型・訪問県塗り分けの応用）。所在県は列挙型（47都道府県）で持つので地図生成は軽い
- 将来: 酒に限らず「産地/所在県を持つカテゴリ」に転用可能な汎用マップ機能に育てる余地
- **2026-07-06 一般化確定**: この「場所マスター＋種別＋座標＋地図集計」は酒蔵・おでかけ施設・御朱印(寺社/城/港)で共通部品化。訪問施設マップ・都道府県マップ・城/寄港マップが同じ資産で成立（08 §4-7/§4-8/§4-10）

## 4-1d. 天気アイコン自動付与（2026-07-06 確定・WeatherKit）

- **出かける系テンプレ（観劇/美術展/ライブ/おでかけ施設）**で、**訪問日が過ぎたら日付＋会場座標から履歴天気を取得し、日付横にSF Symbolsの天気アイコンを自動表示**（酒・店・書籍・映画は除外）。カテゴリに「天気自動付与」フラグ（データ駆動）
- **WeatherKit（Apple純正）** を採用。外部契約・サーバーなし・Apple Developer Programで月50万コール無料＝方針適合。Yahoo天気は旧APIが終了/外部契約のため不採用
- **取得範囲＝2021年8月1日〜現在**（WeatherKit制約・出典: Apple Developer Forums）。それ以前・座標なし・海外はアイコンを出さないだけ（失敗しない設計）・手動上書き可。予報は出さない（過ぎるまで付かない＝実天気=思い出）
- 取得した天気＋気温はVisitに軽くキャッシュ。画面に「 Weather」帰属表示が必要（軽微）
- 古い日付も欲しい需要が強ければOpen-Meteo（1940年〜・APIキー不要）をオプション化する余地（外部送信になるため方針トレードオフ・要ユーザー判断）

## 4-1e. 場所参照DBの配信と課金（2026-07-06 確定）

- **場所DB（御朱印の寺社/城/港・おでかけ施設等）はアプリ同梱をやめ、テンプレONで CloudKit公開データベース（Public DB）からDL** →端末キャッシュ。別サーバー不要（Apple内）・後から増やせる。同期(v2)のprivate DBとは別枠。**「完全ローカル」からの明示的例外**（Apple hostedの静的参照データ）
- **記録は参照DBに依存しない＝スナップショット**：記録時に場所の名称/住所/座標をVisitへ焼き付け（Venueスナップショットと同型）。DBやスポットが消えても記録は無傷。参照DBは「選ぶ・コンプ・情報を足す」の便利レイヤー
- **守る線**：記録機能は無料（Apple Maps POIで全スポット検索・記録可）。課金するのは参照DBの便利さのみ（プリセット/コンプ/エンリッチ）
- **課金＝(A)＋(B)併用で確定**：(A)サブスクに参照DBアクセス＋更新／(B)カテゴリDBパック単体販売。(C)買い切り込みは不採用
- **増やすのは"幅"（新収集ドメイン）**：御酒印（場所=既存Brewery流用）/ダムカード（国交省）/マンホールカード（GKP）/道の駅きっぷ/インフラカード…各々が公式/オープンな配布リストを持ち、新(B)パックになる。※**カテゴリの括り方は要確定**（「収集」1カテゴリの種別 vs 印系/カード系を別カテゴリ）
- 実装スコープ: v1は下地（DL可能な参照DBのフック）だけ通し、CloudKit公開DB＋コンプ＋課金の実装はv1.x/v2

## 4-1f. 共有マスターによる横断リンク（2026-07-06 確定・下地はv1）

- **共有マスターを参照する記録は、そのマスターを"ハブ"に横断でつながる**。汎用則。
- **第一例＝酒蔵(Brewery)ハブで御酒印↔酒**：御酒印の対象(Event)と酒テンプレのEvent(銘柄)が**同じBreweryを参照**。御酒印の記録詳細に「この酒蔵で飲んだお酒一覧」（`Event.brewery==この酒蔵`）を表示、御酒印から「お酒を記録」で酒蔵をプリセットした酒記録を作れる（相互リンク）。08 §4-8
- **成立条件＝同一エンティティ参照**：両者が同じ酒蔵マスター（酒蔵DB）から選ぶこと。手入力は名寄せ（正規化してfetch-first upsertで1件に）＝②-Aの「同一対象判定」課題と同じ。
- **横断は"生きた参照"（relationship）を使う**。スナップショット（記録の耐久性・マスター削除時の表示）は別物で、参照が生きている間だけリンクが張れる。
- **Breweryを"酒専用でなく共有マスター"として設計する下地はv1から**（御酒印パック自体はv1圏外だが、後付けは重いため）。同型の将来例＝酒を飲んだ店(Venue)↔飲食店テンプレ。

## 4-2. 地図・場所入力の方針（2026-07-05 確定）

- **場所の入力は3系統**: ①アプリ内検索（MKLocalSearch＋補完） ②Google Mapsの共有URL（共有シート受け取り or ペースト） ③Apple Mapsの共有URL（同）
- Google URLからは**住所・座標を自動抽出しない**（Places規約: 取得データの永続保存は不可・Place IDのみ永続可。記録アプリの「一生保存」と根本的に非互換のため Places API 自体を採用しない）。名称はリンクのOGPタイトル（LinkPresentation）で取得し、**座標はApple側（CLGeocoder/MKLocalSearch）で解決**して保存する——「特定はGoogleの精度、保存はAppleデータ＋リンク」の分業
- Venueマスター拡張: 名称・住所・座標・GoogleMapリンク・AppleMapリンク・手動ピン調整
- 参照: Google Places API Policies（キャッシュ30日制限）・2025年3月の無料枠改定

## 4-3. カレンダー機能の方針（2026-07-05 範囲・時期とも確定）

**範囲＝A＋B（両方v1）。C（favoreco内での汎用予定作成）はやらない**（コンセプト§5に明記）。

- **A: 体験特化カレンダーのGoogle式表示**
  - 月表示: 日セルにカテゴリ色のイベントチップ（色＝`RecordCategory.colorHex`）
  - 週/日表示: **時間軸グリッドに予定ブロックが浮くGoogle式を完全再現**（2026-07-05確定・新規実装）。ブロックの高さは開演＋上演時間セレクトから導出した終演時刻で決まる。**日表示は週表示から1日分を抜き出した同構造**（時間グリッド1列・ヘッダー下の期間バー/終日ピルも同ルール。2026-07-05確定）
  - **終日・期間ものの描き分け（2026-07-05 モックで確定・Googleの「終日帯」積み上げを廃止）**:
    1. 期間の状態（販売期間・会期）＝**期間バー**（セル下部を走る高さ約8pxの細い連続バー・カテゴリ/種別色・**バー内に白抜き小フォントのタイトル**を開始日と各週の左端に表示。2026-07-05に「もう少し細く・小さく」の調整確定）。**バーの頭にアイコン**：販売期間＝チケットアイコン、会期などイベント期間＝カテゴリのアイコン（SF Symbol）。複数重なればバーを下に積む（最大2段＋あふれ「+n」）。グリッド・チップ領域を占有しない
    2. その日に起きる点（チケット発売日＝販売期間の開始日時）＝枠線チップ「発売 10:00」で開始点だけ強調。発売アラームはこの日時に発火
    3. 通常イベント＝カテゴリ色チップ・**タイトル2行まで**折り返し・入り切らない件数は「+n」→日別リスト
    4. 本当の終日体験（フェス全日・遠征）＝用事名のみ表示（「終日」の文言なし・1行・あふれたら切る）。**月＝日付のすぐ下にグレー枠線バー（期間バーと同じ太さ）／週＝ヘッダー下にグレー枠線ピル（一回り大きい従来サイズ）**。色付き＝期間の状態、グレー枠線＝終日の予定。**月表示で終日がある週は、日付直下の1段目を週全体で「終日スロット」として確保し、通常チップは2段目から始める（チップの高さがガタつかない・Googleの複数日バーと同じ整列規則）。終日がない週はスロットを詰める**（2026-07-05 ユーザー指摘で追加）
    5. 参加済みの記録＝薄めのカテゴリ色チップ（✓）。「記録」レイヤーでon/off
    6. 月グリッドは7列完全均等（中身に列幅を引っ張らせない）。**ビュー切替（月/週/日）は年月タイトルの左に置き縦幅を稼ぐ**
  - Liquid Glass基調（コンセプト§6）: 浮遊するビュー切替ピル・ガラスパネル・外部予定は低彩度/破線縁
  - **レイヤーのオン/オフ**: ①カテゴリごと（カテゴリ色）②種別（予定/記録(済み)/チケット発売・販売期間）③外部カレンダー（カレンダー単位）。「記録」レイヤーONで過去のカレンダーが思い出で埋まる＝アーカイブの見返しを兼ねる
- **B: 外部カレンダー重ね表示（EventKit一本でApple/Google両対応）**
  - iOS設定でGoogleアカウントを繋いだユーザーのGoogleカレンダーは標準カレンダーDBに同期されるため、EventKit読み取りで両方を**薄く**（低彩度・favorecoの体験が主役の階層を守る）重ね表示。OAuth・サーバー・Google API契約は不要、データは端末外に出ない
  - 逆方向（体験予定→標準カレンダーへの書き出し）も同じEventKitで対応
  - 権限: iOS 17以降のカレンダーアクセス許可フロー（読み取りはフルアクセス必要）

## 5. コード共有方式（要決定）

- 推奨（03の現実解）: **まずフォークでPoC → コンセプト検証後に共有コア（Swift Package）切り出しを判断**
- 理由: 共有コアの設計を最初にやると初速が落ちる。RecordCategory化という構造変更を試す段階では、Mystoriumのコードを自由に壊せるフォークが速い

## 6. 致命リスク（1つ）と検証方法

- **リスク**: RecordCategory化（enum→モデル参照）が、SwiftDataのリレーション・マイグレーション・既存UIパターン（ホームのカテゴリタブ・カテゴリ別マスター管理）と噛み合わない場合、Mystorium資産の流用前提が崩れ企画の工数見積りが崩壊する
- **検証方法（実装の最初のステップ＝PoC）**:
  1. RecordCategory＋Event＋マスター1系統（評価軸）だけの最小スキーマを組む
  2. テンプレ7種のプリセット注入 → ホームのカテゴリタブが動的に生える → 自作カテゴリの追加/リネーム/削除（nullify・記録は消さない）が破綻しないことを確認
  3. カテゴリ削除時の既存記録の扱い（「未分類」へ退避 等）をこの段階で設計確定
  4. **2階層マスターの検証（酒種別サブレイヤー）**：〈カテゴリ→酒種別→軸/スペック/OCR辞書〉が、Mystoriumの1階層カテゴリ別マスターの素直な一般化で組めるか。種別キーでプリセットを引く仕組みがCloudKit互換・スキーマ固定を保ったまま成立するか。★ユーザー強調の「下地」なので**PoC必須項目**（08 §4-6★★）。ここが崩れると酒の拡張（ビール/ウィスキー）が後で重いリファクタになる
- **判定**: PoCで上記が素直に組めれば企画続行。組めなければ共有方式・スキーマ設計を再検討
