## 0. このモデルが解く問題

Mystorium は `EventCategory` を**固定enum**で持っていた。favoreco の本質的な仕事は、これを**ユーザー定義の SwiftData モデル（RecordCategory）に一般化**すること（docs/03）。あわせて、Mystorium の「1回性のイベント」を、

- **Event（対象）＝繰り返しても同一と見なす対象**（この公演／この銘柄／この作品）
- **Visit（回）＝その対象に触れた各回**（この日観た／この日飲んだ）

の2粒度に分ける。リピート（再観劇・再飲）は「1つの Event にぶら下がる複数の Visit」で表現する。

## 1. モデル一覧（A1で定義する範囲）

| モデル | 役割 | 主な関係 |
|---|---|---|
| **RecordCategory** | ユーザー定義カテゴリ。どのユニットを表示・入力するかの構成を持つ | Event(多) / 各種マスター（A2） |
| **Event（対象）** | 対象の同一性を定義する属性を持つ器 | RecordCategory / Visit(多・cascade) / Venue系マスター |
| **Visit（回）** | その対象に触れた各回の属性を持つ器 | Event / PhotoBlob(多・cascade) / AxisScore(多) |
| **PhotoBlob** | 写真/動画バイナリ（externalStorage）。正本は 09 | Visit（cascade） |

> Venue / Organizer / Brewery / AxisDefinition / OutcomeOption / FormatMaster / Tag / Companion などのマスター群は **A2 と A3** で定義する。A1 は「その参照をどう持つか」だけ規定する。

## 2. Event / Visit の境界ルール ★レビュー確定（2026-07-06 再掲）

**判定基準**:
> **対象の同一性を定義する属性 = Event。繰り返しても対象が変わらない属性 = Visit。**

「この属性が変わったら別の対象になるか？」で振り分ける。タイトルが変われば別作品＝Event。会場が変わっても同じ公演のツアー＝Visit。

| 属性 | 所属 | 理由 |
|---|:-:|---|
| タイトル | **Event** | 変われば別対象 |
| シリーズ名（seriesName） | **Event** | 対象の帰属（例: 書籍のシリーズ、ツアー名） |
| 種別（subType・酒種別など） | **Event** | 対象の分類（A2）。日本酒⇔ビールは別対象 |
| 主催（Organizer 参照） | **Event** | 対象の帰属 |
| 酒蔵（Brewery 参照） | **Event** | 銘柄＝対象の造り手（同一性の一部） |
| URL（複数）・SNSトグル | **Event** | 対象に紐づく外部情報 |
| 代表アイキャッチ（default） | **Event** | 対象の顔（§5） |
| **会場（Venue 参照）** | **Visit** | 同じ公演でも回ごとに会場が変わりうる（ツアー） |
| **総合評価（overallRating）** | **Visit** | 回ごとに評価する |
| **座席** | **Visit** | 回ごと |
| **写真** | **Visit** | 回ごとの記録 |
| **成果（outcome）** | **Visit** | 回ごとの結果 |
| **タグ／同行者** | **Visit** | 回ごとの文脈 |
| **金額** | **Visit** | 回ごとの支出 |
| 回別アイキャッチ（override） | **Visit** | §5・二段構え |

> **なぜ会場・評価を Visit にしたか（レビューの主眼）**: 当初これらを Event 側に置くと、ツアー公演（同一 Event を複数会場で複数回）や再飲（同一銘柄を別の店で）で破綻する。「1対象＝1評価・1会場」の暗黙前提は、リピートを持つ体験記録では成り立たない。**回の属性は必ず Visit に置く**。

**単発（リピートなし）の扱い**: 1回しか触れない対象も内部的には Event 1 + Visit 1 で持つ。UI 上は1画面で両方を編集できる（登録フローは A7）。「対象を先に作る」感覚をユーザーに強制しない。

## 3. RecordCategory（ユーザー定義カテゴリ）

Mystorium の固定enumを置き換える中核。**テンプレ7種も「保存済みの RecordCategory プリセット」にすぎない**構造にする（08 §4）。

| プロパティ | 型 | 既定値 | 備考 |
|---|---|---|---|
| id | UUID | 生成 | 同期のupsertキー（`@Attribute(.unique)` は使わない・§4） |
| name | String | "" | カテゴリ名 |
| iconSymbol | String | "square.grid.2x2" | SF Symbol 名 |
| colorHex | String | 既定色 | カレンダー/チップ色。RecordCategory.colorHex（08カレンダー仕様の色源） |
| sortOrder | Int | 0 | ホームのカテゴリ並び順 |
| isBuiltIn | Bool | false | テンプレ由来フラグ（プリセット再注入の照合に使用） |
| templateKey | String | "" | 由来テンプレの識別子（"theater" 等・空=完全カスタム） |
| enabledUnitsRaw | String | "" | 表示・入力するユニットIDのCSV/JSON（例 "U1,U3,U7,U11"）。動的スキーマは作らず「どのユニットを出すか」だけを持つ（08 §1） |
| isArchived | Bool | false | 削除の代替（§6）。参照は残す |

- **カスタム作成フロー**: 名前・アイコン・色 → ユニットをトグル → 評価軸/成果/形態の選択肢を編集 → 保存（08 §4）。テンプレは同フローの保存済みプリセット。
- **enabledUnitsRaw** はスキーマを変えない構成データ。全ユニットのフィールドは Event/Visit が最初から持ち（スーパーセット固定・08 §1）、表示可否だけをここで切る。

## 4. CloudKit 互換3条件（全モデル必須）

CloudKit 同期公開は v2 のサブスク機能（07）だが、**スキーマは v1 から互換で設計する**（後付け不可）。全モデルが以下を守る:

1. **全プロパティにデフォルト値**を与える（`var x: Int = 0`）。CloudKit は非optional・デフォルト無しを許さない。
2. **`@Attribute(.unique)` を使わない**。一意性は **fetch-first upsert**（id で先に fetch → あれば更新／なければ挿入）で担保する。プリセット再注入・マスターの重複防止もこの方式。
3. **リレーションは全て optional**（`var event: Event?`）。CloudKit は必須リレーションを許さない。削除時 nullify との相性も良い。

> この3条件は A1〜A7 の全モデル定義で機械的に守る。②-A レビューの必須チェック項目（project-log 2026-07-05「レビュー観点にCloudKit互換を必ず入れる」）。

## 5. アイキャッチ二段構え ★レビュー確定（2026-07-06 再掲）

全カテゴリ標準のアイキャッチ（08 土台）を **Event / Visit の2段**で持つ:

- **Event.defaultEyecatchPath**（代表アイキャッチ）: 対象の顔。ポスター/書影/ボトル等。
- **Visit.eyecatchPath**（回別アイキャッチ・override）: その回の記録写真から選んだ1枚。

**表示解決**: ある Visit を表示するとき、`Visit.eyecatchPath` があればそれを使い、無ければ `Event.defaultEyecatchPath` にフォールバック。

- 縦横比はカテゴリ別プリセット（08 §2。観劇=A4縦、酒=3:4縦…）。比率は `RecordCategory` 側に持たせる（A3で確定）。
- パスは relativePath 文字列のみ扱う（実体は PhotoBlob・§7）。

> **なぜ二段か**: 「対象の顔（毎回同じポスター）」と「その回の思い出の1枚」は別物。1段だと、再訪のたびに代表画像が上書きされるか、逆に回の写真を代表に選べない。二段にして両立させる（ユーザー選択「二段構え(推奨)」）。

## 6. 削除・退避の設計

端末内データは絶対に壊さない（ルートCLAUDE.md／09の原則）。

- **カテゴリ削除**: 物理削除せず `isArchived=true`。ぶら下がる Event/Visit は「未分類」カテゴリへ退避（`Event.category` を nullify → 未分類にひも直し）。記録本体は消えない。
- **マスター削除**（Venue/Organizer/Brewery）: **nullify＋スナップショット表示**（Mystorium踏襲・docs/04 §3）。Event/Visit は削除時点の名称文字列（スナップショット）を別プロパティに保持し、参照が消えても表示は残る。
- **Event 削除**: ぶら下がる Visit は cascade（対象ごと消す明示操作のみ）。
- **Visit 削除**: PhotoBlob は cascade。

## 7. 写真・動画（PhotoBlob）

正本は **09-CloudKit写真ストレージ仕様**。A1 での要点のみ:

- 写真/動画は `PhotoBlob`（SwiftData・`@Attribute(.externalStorage)`）。**画面側は relativePath 文字列だけ扱う**（PhotoStorage API層で分離）。
- `byteCount` を別プロパティに持ち、存在チェック・容量集計で `data` をフォールトさせない。
- 動画は `video_` プレフィックスでファイル保存・**同期対象外**。
- サムネは非永続（AsyncImageDataView のダウンサンプリングで生成）。軽量タイル表示のため写真1枚につき縮小デコードを遅延実行。
- Visit の「写真（複数）」と、ユニット U11 コレクション（半券・印・ラベル等）・U16 ラベル表裏は、**用途で分離するが物理格納は同じ PhotoBlob**（相対パスの所属先が違うだけ・A3で確定）。

## 8. ユニット値の格納方式（unitFieldsRaw と個別プロパティの二本立て）

ユニット（08 §3・U1〜U17）の入力値をどう持つか。**「検索・統計・カレンダーで使う値」と「表示するだけの値」を分ける**:

### 8-1. 個別プロパティで持つ（クエリ対象）
検索・集計・カレンダー・地図の元になる値は、Event/Visit の**独立したプロパティ**として持つ（`#Predicate` や `FetchDescriptor` で引ける形）:

- 日付（開始・終了）／総合評価／成果（outcome）／タグ[String]／同行者[String]
- 座席（軽量文字列）／金額／シリーズ名（seriesName）
- 評価軸スコア（AxisScore・別モデル）／味覚マップ2軸座標（U17・数値2つ）
- 種別（subType・A2）／酒蔵所在県（Brewery経由・地図集計元）

### 8-2. unitFieldsRaw（JSON・表示専用の受け皿）
「表示するだけ・カテゴリごとに項目が違う」値は、**ユニットIDで名前空間を切った JSON 文字列**1本にまとめて持つ。スキーマを増やさずユニット追加に耐える:

```
Visit.unitFieldsRaw (String, 既定 "")  // 例:
{ "u3.kaien": "18:30", "u3.matinee": false,
  "u17.amakaraLabel": "やや辛口", "u9.rows": [{"k":"精米歩合","v":"50%"}] }
```

- キーは `u{番号}.{項目}` で衝突回避（例 `u3.kaien`, `u17.amakara`, `u9.rows`）。
- U9 スペック表（ラベル＋値ペア配列）はこの中に配列で持つ（②-A持ち越し「スペック表のCloudKit互換表現」への回答＝**JSON文字列1プロパティ**。ペア配列を別モデルにすると同期・移行が重くなるため）。
- **原則**: 8-2 に入れてよいのは「その値で検索も集計もしない」もの。少しでもクエリ・統計・カレンダーに使うなら 8-1 の個別プロパティへ昇格させる。

> なぜ全部を JSON にしないか: SwiftData の `#Predicate` は JSON 文字列の中身を引けない。統計・カレンダー・地図が成立しなくなる。**クエリする値＝列、しない値＝JSON** の線引きを A1 で固定する。

## 9. ユーザー確認事項（消失レビューの再確認）

以下は前セッションのレビューで確定したが記録が消えていた項目。清書で復元したが、取り違えがないか確認したい:

1. **Event/Visit 境界（§2）**: 会場・総合評価・座席・写真・成果・タグ・同行者・金額を **Visit** 側に置く（ツアー/再飲で破綻しないため）。この振り分けで合っているか。
2. **アイキャッチ二段構え（§5）**: Event に代表アイキャッチ、Visit に回別override、表示は Visit優先→Eventフォールバック。この方式で合っているか。
3. **スペック表の格納（§8-2）**: U9 を JSON 文字列（unitFieldsRaw 内の配列）で持つ方針。別モデル化しない。これで合っているか。

## 10. A2 への引き継ぎ

- 種別（subType）は Event の同一性属性（§2）。その**プリセット出し分け（味覚軸/スペック/OCR辞書）の2階層マスター**を A2 で定義する。
- 各マスター（AxisDefinition/OutcomeOption/FormatMaster）の**親付け替え**（enum → RecordCategory 参照、さらに subType キー）は A2・A3 で確定する。
