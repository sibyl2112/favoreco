# Favoreco 共通場所カタログ

Favoreco が CloudKit Public Database から配信する共通場所マスターの調査原稿です。会場だけでなく、寺社仏閣、文化施設、レジャー施設、史跡・ランドマーク等を同一形式で扱います。

## 登録方針

- 件数上限は設けず、各分類で利用・来訪・記録される機会が多い場所から順次追加する
- 地域別の件数均等化は目標にしない。偏りを許容し、実際の利用実績を確認できる場所だけを登録する
- 劇場・ライブハウスは収容人数だけで選ばず、大・中規模会場とは別に、長期運営、劇場群、登竜門、地域・インディー文化の拠点等の小規模な文化的重要会場を収録する
- 球場・スタジアム・アリーナは、公表された入場可能数・収容人数が1万人以上の現役施設を優先して全国収録する。プロリーグの現役本拠地はリーグ公式一覧で名称変更と利用実績を再確認する
- 野外音楽祭は場所そのものではないため定期イベントカタログへ分離し、苗場スキー場、蘇我スポーツ公園等の固定開催地だけを共通場所カタログへ紐付ける。開催時だけ設営される特設会場は`status=seasonal`とする
- 興行・イベント会場は、公式サイトの存在だけでは登録しない。公式の公演・イベント日程、貸館実績、または主催者等の公式発表により、原則として確認日から18か月以内の利用実績か今後の開催予定を確認する
- 会場一覧、業界団体への参加、知名度、過去の公演歴だけでは現役判定の根拠にしない。運営団体や劇団そのものも、固有の実会場を持たない場合はPlaceとして登録しない
- 架空・デモ・推測上の施設は共通カタログへ入れない。現役施設は公式の直近利用を確認し、閉館済みでも全国的知名度、文化的・地域的な重要性、世代横断の記録需要がある実在施設は`status=closed`として歴史施設枠へ収録する
- 施設名、所在地、公式URL、収容人数等は施設・運営者・自治体等の公式一次情報で確認する
- 利用者向け項目は、名称、検索用のよみ・別名、種別、住所・座標、公式URL、公表済みの収容人数程度に絞る
- 営業・開館時間、休館日、料金、最寄駅、アクセス経路、開催情報、混雑等は収録せず、最新情報は公式サイトへ任せる
- 入場者数等の公表値がない場合、根拠のない推定値は保存しない
- 説明文や画像は転載せず、検索・同定に必要な事実情報と出典URLのみを保存する
- 不明な値は空欄にし、`sourceURL` と `verifiedAt` を必須の確認情報として扱う
- 寺院の宗派と神社の御祭神は、場所本体へ推測値を混ぜず、`religious-place-details.csv`へ`placeID`で1対1に紐付けて順次収録する
- `prefecture`は例外なく47都道府県のいずれかを必須とし、空欄・表記揺れのある行はCloudKit投入対象にしない
- 改称、移転、閉館、建替えは既存IDを安易に再利用せず、履歴を追える状態にする
- `status=open`は「今日営業している」の意味ではなく、カタログ上で現役の施設であることを示す。臨時休業・改修期間・当日営業は管理しない
- `status=closed`は過去の訪問記録と検索用に保持し、未来の予定作成における通常候補からは除外する。閉館日を公式確認できる場合は`notes`へ保存する
- 文化的重要性は利用者向けランキングにせず、選定根拠を`selectionReason`へ残す。収容人数が公式非公表なら空欄にする
- 新宗教・新興宗教、その派生団体の本部、礼拝施設、関連施設は、知名度・規模・観光掲載・建築的価値の有無にかかわらず共通場所カタログへ収録しない。分類が不明確な候補も、対象外でないことを確認できるまで追加しない
- この除外はFavoreco共通カタログに対する運用ルールであり、利用者が私有の`PlaceMaster`へ手動登録した場所を自動削除・制限するものではない
- 寺社は、自治体・観光協会・DMOの公式観光案内、公式霊場会の参拝案内、または編集主体と所在地情報が明確な観光サイトに掲載されていることを共通候補の必須条件とする。寺社自身の公式サイトや神社庁・宗派名簿への掲載だけでは採用せず、観光・参拝目的の第三者掲載を1件以上確認する
- 出典は公的・公式観光案内を優先し、民間観光サイトまで候補確認範囲を広げる。ただし個人ブログ、SNS投稿、口コミだけのページ、出典不明の自動生成一覧は採用根拠にしない。社務所・授与所・御朱印案内は優先度を上げる補強情報とし、絶対条件にはしない
- 都道府県・自治体の公式観光入込客統計等で年間2万人以上の参拝・来訪が確認できる寺社は優先採用対象とする。初詣等の単独イベント人数だけでは年間値とみなさず、調査年、人数、単位、対象範囲を`temple-shrine-visitor-statistics.csv`へ保存する
- 御朱印の当日授与、受付時間、書置き／直書き、宮司不在日は変動情報として保存せず、利用時は公式案内へ委ねる

## 出典の優先順位

1. 施設・宗教法人・運営会社の公式サイト
2. 国・自治体・独立行政法人・指定管理者の公式ページ
3. 文化財・世界遺産等の公的データベース
4. 公式情報で不足する項目だけ、権利と再利用条件を確認した外部データ

## CSVの位置づけ

`place-catalog.csv` は収集・レビュー用の原稿です。アプリへ固定同梱する完成データではありません。`sourceURL`、`verifiedAt`、`selectionReason`、`notes`は保守・レビュー用で、通常の候補画面へそのまま表示しません。レビュー済みデータを CloudKit Public Database に投入し、アプリは差分取得してローカルキャッシュします。ユーザーが予定・記録で選択した時点の名称・住所等は個人側へスナップショット保存します。

投入前検証では、列数・ID重複・URL・管理状態に加え、`prefecture`が47都道府県の正規表記に一致することを必ず確認します。

## CloudKit投入用レコード

`generate-cloudkit-records.py`は、場所本体へ宗教施設詳細と巡礼所属を結合し、CloudKit管理処理へ渡すNDJSONを生成する。アプリ本体へ同梱するファイルではない。

```bash
python3 docs/data/place-catalog/generate-cloudkit-records.py \
  --output /tmp/favoreco-public-places.ndjson \
  --missing-readings-output /tmp/favoreco-missing-place-readings.csv
```

- レコード種別は`PublicPlace`、レコード名は安定した`placeID`とする
- `temple-shrine-selection-evidence.csv`が`confirmed`でない寺社は出力しない。2026-07-22時点の出力は3,057件で、追加確認中83件を除外する
- 読み仮名は3,140件中1,633件（52.0%）が入力済みで、1,507件は未補完。公開候補3,057件では読みあり1,550件、未入力1,507件。読みがなくても正式名称・別名・住所で検索できるが、ひらがな検索の網羅性を上げるため公式サイト等で確認できたものから補完する。固有地名・寺社名を機械変換だけで確定しない
- `place-reading-evidence.csv`の`method`は、公式がかなを明記した`official_kana`、公式日本語名称を検索用ひらがなへ変換した`official_name_normalization`、公式英字ブランドから検索用かなを作った`official_brand_transliteration`の3種とする。生成時に場所ID、読み、方法、出典URL、確認日の一致を検証する
- `updatedAt`はアプリの差分カーソルなので、公開項目を変更するたびに必ず進める
- 公開終了時はCloudKitレコードを物理削除せず、`isDeleted=true`と新しい`updatedAt`を保存する。非公開へ戻す場合は`isPublished=false`を使う
- CloudKit Dashboardでは`updatedAt`をQuery可能にし、利用者アプリはPublic Databaseを読取専用で利用する。レコード作成・更新権限は管理側だけに限定する
- NDJSONは中立的な管理用ペイロードであり、CloudKit Developmentへの投入とProductionへのスキーマ反映はApple側の管理作業として別途行う

## 寺社の選定根拠

`temple-shrine-selection-evidence.csv`は、寺社を共通候補へ採用するための第三者掲載根拠を、場所本体の一次情報とは分離して保存する監査原稿です。`placeID`で`place-catalog.csv`と1対1に対応し、`evidenceStatus`、掲載種別、掲載主体、掲載URL、確認日を保持します。

- `confirmed`: 公式観光案内、公式霊場会、または採用可能な観光サイトへの掲載を確認済み
- `needs_review`: 現在の場所出典が寺社公式・神社庁・宗派資料等のみで、第三者の観光・参拝掲載を追加確認する
- 場所の名称・住所・公式URL等の確認元は`place-catalog.csv`、採用条件を満たす掲載実績は本CSVを正本とし、用途を混同しない
- CloudKit公開DBへの投入対象は原則`confirmed`だけとし、`needs_review`は調査原稿には残すが公開候補から保留する

## 寺社の年間来訪者統計

`temple-shrine-visitor-statistics.csv`は、都道府県・自治体等が公表する年間の観光地点入込客数を保存する監査原稿です。人数はランキング表示や恒久的な施設属性ではなく、採用優先度と網羅確認の根拠として扱います。

- `surveyYear`: 調査対象年
- `annualVisitors`: 公表単位を人回相当へ換算した整数
- `unit`: 現在は`people_visits`。実人数ではなく延べ訪問地点数の場合がある
- `scope`: 初詣除外等を含む調査範囲
- `evidenceProvider` / `evidenceURL`: 統計公表主体と資料URL
- 同じ施設の複数年度は別行で保持できる設計とし、場所本体へ最新値を直接上書きしない
- 複数施設の合算値は個別Placeへ配分しない。伊勢神宮全体や日光二社一寺等は、構成施設別の値がない限り内宮・外宮や各社寺へ重複登録しない
- 年間2万人以上を優先採用対象とするが、統計非公表だからという理由だけで観光案内掲載済み寺社を除外しない

## 番号付き場所コレクション

`place-collection-memberships.csv`は、日本100名城等の番号付き公式コレクションとPlaceの所属関係を保存する原稿です。`collectionKey`、`collectionName`、整数の`itemNumber`、表示用の`itemNumberLabel`を場所本体から分離し、同じPlaceを複製せずに一覧・達成状況へ利用できる形にします。2026-07-21時点では、日本100名城の第1〜100番と続日本100名城の第101〜200番を、欠番・番号重複なしで収録しています。

## 巡礼・札所番号

`pilgrimage-memberships.csv`は、場所と巡礼の所属関係を保存する原稿です。札所番号は寺院名や検索用別名だけへ埋め込まず、次の専用項目で保持します。

- `membershipID`: 所属関係の安定ID
- `placeID`: `place-catalog.csv`の場所ID
- `pilgrimageKey` / `pilgrimageName`: 巡礼の識別子と表示名
- `siteNumber`: 並べ替え・欠番検証に使う整数の札所番号
- `siteNumberLabel`: `第1番`等の表示文字列。番外では任意表示を許可する
- `status`: 現役、変更、廃止等の管理状態
- `sourceURL` / `verifiedAt`: 公式出典と確認日

場所と所属を分離することで、同じ寺院が複数の霊場に属する場合もPlaceを複製せず、それぞれの札所番号を入力・編集できる。正規札所の欠番と番号重複は巡礼単位で検証し、番外札所は正規の件数へ混ぜない。

2026-07-21時点では、西国三十三所、坂東三十三観音、四国八十八箇所、秩父三十四観音、洛陽三十三所観音霊場、鎌倉三十三観音霊場、最上三十三観音、越後三十三観音霊場、北海道三十三観音霊場、伊勢西国三十三所、小豆島八十八ヶ所、九州西国霊場、知多四国八十八ヶ所の13霊場を収録する。正規札所595件に、公式一覧掲載の番外・奥の院・開山所・特別札所・旧札番23件を加えた計618所属である。正規外の所属は`siteNumber`を空欄にし、`siteNumberLabel`へ公式の表示を保持して正規番号と区別する。

## 宗派・御祭神

`religious-place-details.csv`は、寺院の宗派と神社の御祭神を場所本体へ紐付ける任意の1対1詳細原稿です。

- `placeID`: `place-catalog.csv`の場所ID。存在しないIDは投入対象にしない
- `placeKind`: `temple`または`shrine`。場所本体の`catalogID`と一致させる
- `templeSect`: 寺院の宗派。寺院以外は空欄にする
- `enshrinedDeities`: 神社の御祭神。複数の場合は`天照大神|須佐之男命`のように`|`区切りとし、神社以外は空欄にする。1つの文章へ連結せず、1柱ずつ独立した値として解釈する
- `sourceURL` / `verifiedAt`: 宗教法人、神社庁、公式霊場会、自治体等の確認出典と確認日
- `notes`: 出典上の表記境界や未詳項目を記録する。名称が分からない`外二柱`等を推測で御祭神名へ変換しない

全寺社へ一括で推測値を入れず、公式確認できた場所から追加する。CloudKit投入時は`|`を分割してString Listへ変換し、場所本体との1対1関係として結合できる形を維持する。候補画面では御祭神ごとの検索、詳細表示、管理画面での1柱ずつの追加・削除に利用する。表示順は公式情報の掲載順を保持し、同一神の重複だけを除外する。
