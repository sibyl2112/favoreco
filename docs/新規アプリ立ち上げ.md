# 新規アプリ立ち上げガイド

> **役割**: 「アイデアがある（or ふんわりある）」状態から「何を作るか確定して開発に入る」までの**前段**を定型化したもの。
> **位置づけ**: このガイドの出口（コンセプト確定・MVP範囲確定）が、`docs/開発プロセス.md` の入口（① 要件メモ）につながる。
> **使い方**: 新しいアプリを思いついたら、まず「§1 対話ブラッシュアップ台本」を上から回す。並行して各フェーズの「チェックリスト」を埋める。最後に「§3 コピペテンプレート」で成果物を残す。
> **最終更新**: 2026-06-12

---

## 0. 全体像

```
アイデアの種（ふんわり）
   ↓  フェーズ1: 言語化      ← まず1文にする
アイデアを疑う
   ↓  フェーズ2: 精査        ← 課題は本物か / 誰が / 代替 / なぜ今
作るものを絞る
   ↓  フェーズ3: コンセプト確定 ← 一言・コア体験・やらないこと
作れるか確かめる
   ↓  フェーズ4: 実現性・技術  ← データモデル素描・リスク・技術選定
最初に出す範囲を決める
   ↓  フェーズ5: MVP範囲確定   ← v1に入れる/入れないとロードマップ
開発プロセスへ引き継ぐ
   ↓  → docs/開発プロセス.md ① 要件メモ へ
```

**原則**:
- **作る前に「作らない理由」を一度探す**。精査（フェーズ2）で「やめる/方向転換する」判断が出てよい。むしろそれが目的。
- **コンセプトが1文で言えるまで、技術の話をしない**。フェーズ3が固まる前にフェーズ4へ飛ばない。
- **MVPは「削る」作業**。足し算ではなく、最初に出さないものを決める。
- このガイドは「何を作るか」まで。「どう作るか」は `docs/開発プロセス.md` に渡す。

# §1. 対話ブラッシュアップ台本（Claude ↔ ユーザー）

> アイデアを一緒に磨くときの**質問の順番**。Claude はこの順で問いを投げ、ユーザーの答えを下の各フェーズのテンプレに埋めていく。
> 一度に全部聞かない。**1フェーズずつ**進め、答えが曖昧なら次に進まず深掘りする。
> ユーザーが迷ったら、Claude は選択肢を2〜4個出して選んでもらう（`CLAUDE.md` 3.1 / 3.4）。

## フェーズ1: 言語化 — まず1文にする

Claude が聞くこと（順に）:
1. 「どんなアプリ？ まず思いついたまま1〜3行で教えて」
2. 「それは誰の、どんな困りごとを解決する？」
3. 「一言でいうと『◯◯するためのアプリ』になる？ 1文で言い切ってみて」

ゴール: **1文のキャッチ**（例: 「気になる → これから行く → 記憶になる、を1つのアプリで」）。
ここで詰まる＝アイデアがまだ固まっていない。フェーズ2に進む前に、Claude が候補を数パターン出して選んでもらう。

## フェーズ2: 精査 — アイデアを疑う

Claude が聞くこと（このフェーズが一番重要。遠慮なくツッコむ）:
1. **課題は本物か**: 「その困りごと、自分（or 想定ユーザー）は実際に困ってる？ どのくらいの頻度で？」
2. **既存の代替**: 「今はどうやってしのいでる？（紙・メモアプリ・既存サービス・我慢）」「その代替の何が不満？」
3. **なぜ今・なぜ自分**: 「なぜ今これを作る？ 自分が作る意味は？（情熱・知見・データを持ってる等）」
4. **使う場面**: 「いつ・どこで・どんなトリガーで開く？」
5. **続けて使うか**: 「1回使って終わりじゃない？ 何が再訪の理由になる？」

ゴール: 「やる/やめる/方向転換」の判断材料。
- 代替で十分なら **やめる勇気** を持つ（log に「なぜやめたか」を残す）。
- コア体験が弱いなら **方向転換**（誰・課題・使う場面のどれかを変える）。

> ⚠️ ここで Claude が「No」を言えること。「いいアイデアですね」だけ返すのは仕事をしていない。リスク・弱点を必ず1つは指摘する。

## フェーズ3: コンセプト確定 — 作るものを絞る

Claude が聞くこと:
1. **コア体験（1つだけ）**: 「このアプリの『これさえできれば成立』という体験を1つに絞ると？」
2. **差別化**: 「既存の代替と決定的に違う点は1つ何？」
3. **やらないこと**: 「逆に、似たアプリにありがちだけど今回やらないことは？」

ゴール: コンセプトシート（§3-A）が埋まる。コア体験が**2つ以上**出たら、まだ絞れていない。1つに削る。

## フェーズ4: 実現性・技術 — 作れるか確かめる

Claude が聞く / 一緒に詰めること:
1. **データの中心**: 「アプリが扱う一番大事なデータは何？（例: イベント / 訪問記録 / 写真）」→ 主要エンティティを3〜5個で素描
2. **保存と同期**: 「データはローカルだけ？ クラウド同期は要る？（最初は要らないことが多い）」
3. **技術的な山場**: 「一番難しそうな部分はどこ？（地図 / 通知 / 画像処理 / 課金 / 認証）」
4. **技術スタック**: 既存資産・スキルに寄せる（例: SwiftUI / SwiftData / MapKit）。新規に増やす技術は山場の数だけに絞る。
5. **致命リスク**: 「これが無理だと企画が崩れる、という前提は何？」を1つ特定し、最初に検証する

ゴール: 実現性メモ（§3-B）。主要エンティティ・致命リスク・技術選定の理由が言える。

## フェーズ5: MVP範囲確定 — 最初に出す範囲を決める

Claude が聞くこと:
1. 「フェーズ3のコア体験**だけ**で動くとしたら、最低限いる機能は？」（足し算しない）
2. 「それ以外は全部 v2 以降。v1 に入れない機能を列挙して」
3. 「v1 で『成功』とみなす条件は？（自分が毎日使う / 1機能が破綻なく回る 等）」

ゴール: MVP定義表（§3-C）。**「v1に入れない」リストが「入れる」リストと同じ密度**で書けていること。

## 引き継ぎ

フェーズ1〜5の成果物（§3-A/B/C）が埋まったら、`docs/開発プロセス.md` の **① 要件メモ** に渡す。
コンセプトシートの「一言キャッチ」がそのまま①の到達点になる。

# §2. フェーズ別チェックリスト

各フェーズ、これが埋まれば次へ進んでよい。

## フェーズ1: 言語化
- [ ] アイデアを1〜3行で書いた
- [ ] 「誰の・どんな困りごと」が言えた
- [ ] **一言キャッチ（1文）**ができた

## フェーズ2: 精査
- [ ] 課題の実在性を確認した（頻度・切実さ）
- [ ] 既存の代替手段を挙げ、その不満点を言語化した
- [ ] 「なぜ今・なぜ自分」に答えた
- [ ] 使う場面（いつ・どこで・トリガー）を書いた
- [ ] 再訪の理由を言えた
- [ ] **Claude がリスク/弱点を最低1つ指摘した**
- [ ] 「やる / やめる / 方向転換」を判断した

## フェーズ3: コンセプト確定
- [ ] コア体験を**1つ**に絞った
- [ ] 差別化ポイントを1つに言い切った
- [ ] 「やらないこと」を列挙した
- [ ] コンセプトシート（§3-A）が埋まった

## フェーズ4: 実現性・技術
- [ ] 主要エンティティを3〜5個で素描した
- [ ] ローカル/クラウドの方針を決めた
- [ ] 技術的な山場を特定した
- [ ] 技術スタックを既存資産に寄せて選んだ
- [ ] **致命リスクを1つ特定し、検証方法を決めた**

## フェーズ5: MVP範囲確定
- [ ] v1に入れる機能を列挙した（コア体験だけで動く最小）
- [ ] **v1に入れない機能を同じ密度で列挙した**
- [ ] v1の成功条件を書いた

## 引き継ぎ
- [ ] §3-A/B/C が埋まった
- [ ] `docs/開発プロセス.md` ① 要件メモ に渡せる状態になった
- [ ] `docs/project-log.md` に「立ち上げの意思決定（特にやめた/方向転換した判断）」を残した

# §3. テンプレートファイル一覧

各ファイルを `docs/APP_NAME-<種別>.md` としてコピーして使う。
例: `docs/myapp-concept.md`、`docs/myapp-mvp.md`

| テンプレートファイル | 使うフェーズ | コピー先の命名例 |
|------------------|-----------|--------------|
| `docs/_templates/concept.md` | フェーズ3 コンセプト確定 | `docs/APP_NAME-concept.md` |
| `docs/_templates/feasibility.md` | フェーズ4 実現性・技術 | `docs/APP_NAME-feasibility.md` |
| `docs/_templates/mvp.md` | フェーズ5 MVP範囲確定 | `docs/APP_NAME-mvp.md` |
| `docs/_templates/project-log-entry.md` | 立ち上げ判断の記録 | `docs/project-log.md` に追記 |

> 機能開発フェーズ（①以降）で使うテンプレートは `docs/開発プロセス.md` → `docs/_templates/spec.md` / `design.md` / `brief-for-claude.md` を参照。
