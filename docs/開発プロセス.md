# アプリ開発プロセス手順書

> **役割**: 新機能・新アプリを作るときの「進め方の正本」。実プロジェクトで回してきた流れを汎用化したもの。
> **使い方**: 新しい開発に着手するとき、この手順を上から順にたどる。各フェーズの成果物テンプレートは末尾にある。
> **最終更新**: 2026-06-12

---

> **このプロセスの手前**: 「そもそも何のアプリを作るか」がまだ固まっていないなら、先に `docs/新規アプリ立ち上げ.md`（アイデア精査 → コンセプト確定 → MVP範囲確定）を回す。その出口が下の① 要件メモにつながる。

---

## 0. このプロセスの全体像

```
要件を言葉にする
   ↓  ① 要件メモ
何を作るか確定する（仕様は2系統に分ける）
   ↓  ②-A ロジック・データ仕様書（*-spec.md）   ★ロジックの正本
   ↓  ②-B デザイン仕様書（*-design.md）          ★見た目の正本
   ↓  ②-C 事前の構造設計（後で重いリファクタを出さないための設計）
作業指示に落とす
   ↓  ③ 実装指示書（*-brief-for-claude.md）★Claudeに渡す
段階的に実装する
   ↓  ④ 実装（ビルドを通しながら小さく）
別視点で点検する
   ↓  ⑤ レビュー（review-YYYYMMDD.md）
記録を残す
   ↓  ⑥ project-log.md 追記
```

**原則**: フェーズを飛ばさない。特に「②仕様書なしで③指示書を書かない」「③指示書なしで④実装を始めない」。
曖昧なまま手を動かすと、後で必ず手戻りする。

**追加原則（過去の手戻りから）**:
- **構造は最初に決める（②-C）**。後から重いリファクタが発生しがちなのは、肥大化する前提の設計を先にしていないため。「育つ場所」は最初から分割しておく。
- **ロジックとデザインの仕様は分ける（②-A / ②-B）**。両者を混ぜると、デザイン変更のたびにロジック仕様まで読み直すことになり見通しが悪い。正本を2つに分け、片方だけ更新できるようにする。
- **ユニットテストは最初から入れる（④）**。集計・状態判定などのロジックは実装と同時にテストを書く。Claude（Linux）はビルド不可・実機確認はMacというAI分業の弱点を、テストで客観的に埋める。

## AIの役割分担（成果物で受け渡す）

会話ベースで往復すると重くなるため、AI間の受け渡しは**ドキュメント**で行う。

| 役割 | 担当 | 入力 | 出力 |
|------|------|------|------|
| **精査（事前レビュー）** | Codex | ②-A/②-B 仕様書・③指示書 | `docs/レビュー観点.md` に沿ったツッコミ |
| **実装** | Claude | 確定した③指示書 | 実装 ＋ ⑤レビュー ＋ ⑥log |

- **レビューは実装前（仕様・指示書）に寄せる**。実装後の精査は手戻りが大きい。
- AI間で意見が割れたら、採否と理由を `docs/project-log.md` に決定ログとして残す（同じ論点で再度往復しない）。

## ① 要件を言葉にする（要件メモ）

**目的**: 「何を解決したいか」「誰がどう使うか」を、実装の前に言語化する。

### やること
- 解決したい課題を1〜3行で書く（機能の説明ではなく、ユーザーの困りごと）
- 「この体験ができたら成功」という到達点を1文で書く
- スコープ外（今回やらないこと）を最初に決める

### 完成の目安
- 「一言でいうと何を作るのか」が1文で言える
  - 例: 「気になる → これから行く → 記憶になる、の流れを1画面で見せる」

> ⚠️ ここが曖昧なまま進むと、仕様書が発散する。迷ったらユーザーに選択肢を出して決めてもらう。

---

## ② 何を作るか確定する（仕様を2系統 + 構造設計に分ける）★正本

仕様書を「ロジック・データ（②-A）」と「デザイン（②-B）」に分けて持つ。
さらに、実装前に「後で重いリファクタを出さないための構造（②-C）」を決める。

> **なぜ分けるか**: ロジックとデザインを1枚に混ぜると、色や文言を直すたびに集計仕様まで読み直すことになり、どちらも更新が滞る。正本を分ければ「デザインだけ変えたい」「集計だけ直したい」が独立してできる。
> 小さな機能なら1ファイル内をセクションで分けるだけでもよいが、**章として明確に区切る**こと。

---

### ②-A ロジック・データ仕様書（`docs/<機能名>-spec.md`）★ロジックの正本

**目的**: 状態・データ・集計など「壊れると怖い部分」の定義を明示する。デザインが変わっても、ここは原則変わらない。

#### 必ず明文化するもの
- **状態の定義**: 取りうる状態と意味（例: `未着手 / 進行中 / 完了`）
- **データの正本**: 何を真とするか（例: 状態は永続化したフィールドを正本にし、他データから再推測しない）
- **集計の母数**: 何を数えるか・何を除外するか（例: 完了分のみを集計し、未着手/進行中を混ぜない）
- **画面構造とセクション順**: 並び順を固定する（※レイアウトの「見た目」ではなく論理的な順序）
- **表示する項目 / 表示しない項目**: セクションごとに列挙（※「どう見せるか」は②-Bへ）
- **空データ・0件・1件の扱い**: 非表示ルールを表にする
- **受け入れ条件**: 「これを満たしたら成立」という検証可能な条件
- **今回のスコープ外**: 「後でやる候補」ではなく「今回触らない」項目として明示

#### 完成の目安
- この仕様書だけ読めば、実装者が集計・状態遷移で迷わない
- 数値・集計に「どちらとも取れる」箇所が残っていない

> 📎 雛形: `docs/_templates/spec.md`

---

### ②-B デザイン仕様書（`docs/<機能名>-design.md`）★見た目の正本

**目的**: 「どう見せるか」を、ロジックから切り離して定義する。ここだけ更新すれば見た目を変えられる状態にする。

#### 必ず明文化するもの
- **レイアウト**: 各要素の配置・大きさ・余白（ワイヤー or ASCII図でよい）
- **タイポグラフィ**: フォント・サイズ・用途（例: 数値=セリフ体イタリック、ラベル=サンセリフ）
- **カラー**: 使用色とその意味。**既存のデザインシステム/カラーテーマ定義を参照する**（新規に色を増やさない）
- **状態ごとの見え方**: 強調 / 抑制する要素（②-A の状態定義に対応させる）
- **アニメーション・ジェスチャー**: 挙動と、他ジェスチャー（エッジスワイプ等）との干渉注意点
- **空状態の見せ方**: 0件メッセージの文言・配置（非表示ルールそのものは②-A）

#### 既存のデザイン資産を必ず参照する
- プロジェクトの `DESIGN.md` / カラーテーマ定義 / デザインシステム資料
- 新しい色・フォント・余白規則を勝手に増やさない。増やすなら先にデザインシステム側を更新する

#### 完成の目安
- この仕様書だけ見れば、ロジックを知らなくても見た目を再現できる
- 使う色・フォントがすべて既存定義のどれに当たるか紐づいている

> 📎 雛形: `docs/_templates/design.md`

---

### ②-C 事前の構造設計（後で重いリファクタを出さない）

**目的**: 実装前に「どこが育つか」を予測し、肥大化・密結合する前に分割の単位を決めておく。
**教訓**: 1つの View ファイルが数千行に膨らんでから分割すると、リファクタが重くなる。これを最初から避ける。

#### 着手前に決めること
- **育つ場所を予測する**: 今後セクション・状態・分岐が増えそうな View / 型を挙げる
- **分割単位を先に切る**: 大きくなる View は、最初からセクションごとに子 View へ分ける
  - 子 View は **dumb view**（データと closure を受け取って描くだけ）にする
  - **business logic は親に集約**し、複数 View にコピペしない
- **共有部品を先に切り出す**: section header / more button / カード等、2回以上出るUIは最初から shared 化
- **ロジックとViewを分離する**: 集計・状態判定は View の中に直書きせず、computed property / helper / store に寄せる
  - これにより②-Bのデザイン変更が②-Aのロジックに波及しなくなる
- **データの正本を1つに**: 状態を複数箇所から推測しない（永続化したフィールドを正本にし、都度再計算しない）

#### 肥大化を防ぐ目安（ソフトな閾値）
- 1つの View ファイルが **おおむね 500〜600 行** を超えそうなら、その時点で分割を検討する
- 同じ集計ロジックが **2箇所** に現れたら、共通化する
- 1つの型が「表示 + 集計 + 状態管理 + 保存」を全部持ち始めたら、責務を分ける合図

#### 「リファクタしなくて済む」と「リファクタしやすい」の両取り
- 最初から完璧な抽象化はしない（過剰設計も手戻りを生む）
- ただし **境界（View / logic / data の切れ目）は最初に引いておく**
- 後から分割するときも、境界が引いてあれば軽いリファクタで済む

#### 完成の目安
- 「このファイルは将来どこまで大きくなるか」「そのとき何を切り出すか」が事前に言えている
- View・ロジック・データの境界が指示書（③）に落ちている

> 後から重い分割リファクタが必要になった場合は、その経緯を `docs/project-log.md` に残し、次回は②-Cで前倒しする。

---

## ③ 作業指示に落とす（実装指示書 = `*-brief-for-claude.md`）★Claudeに渡す

**目的**: 仕様書を、Claude がそのまま着手できる作業指示に変換する。
仕様書が「何を作るか」なら、指示書は「どう手をつけるか・どこを触るか・何をやらないか」。

### 必ず入れる8ブロック（このテンプレを固定で使う）
1. **目的** — なぜやるか。改修かリファクタかを明言（挙動を変えるのか変えないのか）
2. **現状の前提** — 対象ファイル・行数・現在の構造・触ると危険な箇所
3. **必須方針** — 守るべきルール（例: 表示を変えない / ロジックを重複させない）
4. **対象としてよいファイル** — 変更してよいファイルを列挙。必要以上に広げない
5. **今回やらないこと** — schema変更・他画面・保存ルール等、触らない範囲を明示
6. **受け入れ条件** — 完了の判定基準（番号付き・検証可能に）
7. **確認してほしいポイント** — 壊れやすい箇所を先回りで指定
8. **期待する進め方** — 棚卸し → 小さく切り出し → ビルド → log更新、の順序

### 完成の目安
- 「やること」と同じ密度で「やらないこと」が書けている
- 受け入れ条件が、後でなぞって検証できる形になっている

> 📎 雛形: `docs/_templates/brief-for-claude.md`

---

## ④ 段階的に実装する

**目的**: 大きな変更を一度にやらず、機能・画面単位で区切ってビルドを通しながら進める。

### ルール（CLAUDE.md 3.3 準拠）
- 1変更が **3ファイル以上** に及ぶなら、先に影響範囲を一覧で出す
- **5ファイル以上** なら作業を分割し、ステップごとに確認を挟む
- デザイン系は「選択肢を出す → 選んでもらう → 実装」の順
- 描画を先に切り出し、business logic は親に残す（最初から作り直さない）
- **各段階でビルドを通す**。通らない状態を積み上げない

### ユニットテストは最初から書く（後追いにしない）
- **集計・状態判定などの純粋ロジックは、実装と同じタイミングでテストを書く**
- ②-Cで View からロジックを分離してあるので、ロジックは UI なしでテスト可能
- パターン: 純粋ロジックを View から切り出し、インメモリのテストデータで検証する
- テスト対象の優先度: 母数の定義（何を数え何を除外するか）／割合／並び順／空・0件・1件の分岐
- 効果: 精査役・実装役が「テストが緑か」で客観判定でき、口頭の往復が減る。リファクタ（②-C）しても「テスト緑＝挙動不変」を保証できる

### 完成の目安
- 受け入れ条件をすべて満たしている
- ②-Aのロジックにユニットテストがある（緑）
- ビルドが通る（実機/ビルドはMac側で確認）

---

## ⑤ 別視点で点検する（レビュー = `review-YYYYMMDD.md`）

**目的**: 自分が書いたコードを自分で「OK」にしない（CLAUDE.md 3.2）。

### やること
- **ロジックはテストで担保する**（目視レビューに頼らない）。④で書いたユニットテストが緑であることを確認
- 変更箇所を読み直し、意図通りか確認
- エッジケースを通す: 空データ / nil / 0件 / 1件 / 大量データ（できるものはテスト化）
- UI: 型の不一致・ビュー階層への影響・ジェスチャー干渉を確認（ここは実機確認が必要な領域）
- 受け入れ条件を1つずつなぞる
- 必要に応じて完成度・機能マップを `review-YYYYMMDD.md` に残す
- 事前レビュー（精査役）の観点は `docs/レビュー観点.md` を使う

### 変更報告に必ず含める（CLAUDE.md 3.2）
- 変更したファイル一覧
- 各ファイルで何を変えたか（1行要約）
- 影響する画面・機能の一覧
- 既知のリスク・副作用の可能性

---

## ⑥ 記録を残す（project-log.md 追記）

**目的**: 「なぜそうしたか / どう遷移したか」を残す。仕様書が「今どうあるか」、log が「なぜそうなったか」。

### 必ず残す項目（CLAUDE.md 2 準拠）
- 日付
- 変更概要
- 変更意図
- 主な変更ファイル
- 確認結果（実機 / ビルド）
- 残課題

### 同時に更新するもの
- 仕様が変わったなら、プロジェクトの `CLAUDE.md`（正本）も更新する

---

## 完了チェックリスト（CLAUDE.md 4 準拠）

- [ ] ロジック仕様（②-A）とデザイン仕様（②-B）が分かれて書かれているか
- [ ] 事前の構造設計（②-C）で「育つ場所」と「分割単位」を決めたか
- [ ] View・ロジック・データの境界が引けているか（集計を View に直書きしていないか）
- [ ] 同じロジック・同じUIを2箇所以上にコピペしていないか
- [ ] 集計・状態判定のロジックにユニットテストがあるか（緑か）
- [ ] 変更意図と実際の変更が一致しているか
- [ ] 変更したファイル・関数を読み直し、論理的な誤りがないか
- [ ] 影響を受ける他画面・機能に副作用がないか（影響範囲を列挙したか）
- [ ] エッジケースを考慮したか（空 / nil / 0件 / 型不一致）
- [ ] プロジェクト `CLAUDE.md` の記述と矛盾していないか
- [ ] `docs/project-log.md` に記録したか
- [ ] ユーザーが検証できる形で結果を提示したか（変更ファイル一覧＋確認手順）

# 付録: テンプレートファイル一覧

各ファイルを `docs/<機能名>-<種別>.md` としてコピーして使う。

| テンプレートファイル | 対応フェーズ | コピー先の命名例 |
|------------------|-----------|--------------|
| `docs/_templates/spec.md` | ②-A ロジック・データ仕様書 | `docs/<機能名>-spec.md` |
| `docs/_templates/design.md` | ②-B デザイン仕様書 | `docs/<機能名>-design.md` |

> 立ち上げフェーズ（コンセプト確定・MVP範囲確定）で使うテンプレートは `docs/新規アプリ立ち上げ.md` → `docs/_templates/concept.md` / `feasibility.md` / `mvp.md` を参照。

> **②-C 構造設計メモ**は独立ファイルが必要なほど大きくなければ、③ 実装指示書の「現状の前提」セクションに取り込む。必要なら `docs/_templates/brief-for-claude.md` の前に構造設計セクションを追記する。
