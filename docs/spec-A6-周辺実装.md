## 1. 同一対象判定（名寄せ / entity resolution）

CloudKit互換で `@Attribute(.unique)` を使わない（A1 §4）ため、一意性は**正規化＋fetch-first upsert**で担保する。

### 1-1. 正規化キー
- 文字正規化：前後空白除去・**全角/半角統一（NFKC）**・連続空白圧縮・大文字小文字統一。株式会社等の法人格サフィックスは**別途保持しつつ照合時は無視**（"旭酒造"≒"旭酒造株式会社"）。
- 正規化名を照合キーにし、**表示名は別に保持**（A2のkey/displayName分離と同思想・リネームで壊れない）。

### 1-2. マスターのupsert（Brewery/PlaceMaster/Person/Organizer/Companion/Tag）
- **参照DB/POIから選んだ場合**：そのエンティティのid（or 安定キー）で一致＝同一。手入力の重複を防ぐ第一手段。
- **手入力の場合**：正規化キーで fetch → 有れば再利用／無ければ作成。
- **近い別名**（表記ゆれ・0票判定できない揺れ）は**候補提示してユーザー確定**（自動マージしすぎない）。

### 1-3. Event同一性＝リピート判定（08 §5 持ち越しへの回答）
- **原則＝Event単位**。新規登録時、**タイトル正規化（＋種別/シリーズ等の同一性属性）**で既存Eventに一致候補があれば、
  「これは『◯◯』ですか？ **既存に追加（＝リピート＝新しいVisit）** / 別物として新規」を提示 → **ユーザー確定**（完全自動化しない）。
- ツアーは同一Event＋複数Visit（A1 §2）／再飲・再読・月替わり再訪も同型。

### 1-4. 手動マージ
- すり抜けた重複（Event同士・マスター同士）を**後からユーザーがマージ**できる。マージ＝一方のVisit/参照を他方へ付け替え＋スナップショット保持。破壊しない（記録は消えない）。

## 2. スナップショット（記録の耐久性）

「端末内データは絶対に消えない」（ルートCLAUDE.md／09）の実装的な要。

- **何を焼き付けるか**：マスター参照時、その時点の**名称・住所・座標・所在県など表示に要る値**を Event/Visit 側のプロパティ（文字列/数値）へコピーして保存。
- **いつ**：記録作成・編集でマスターを結び付けた瞬間（および参照DBから取得した瞬間）。
- **二重持ち＝生きた参照＋スナップショット**：
  - **生きた参照（relationship・optional）**＝ナビゲーション・横断リンク・集計に使う（§3）。
  - **スナップショット（値のコピー）**＝表示の担保。参照が消えても表示は残る。
- **削除・消滅時の挙動**：
  - マスター削除 → 参照は **nullify**、スナップショットで表示継続（A1 §6）。
  - **参照DB（DL）からスポットが消える/閉業** → ローカル記録は無傷（参照DBは正本でない・08 §4-10）。閉業は「もう行けない場所の記録」としてむしろ価値。
  - カテゴリ削除 → 未分類へ退避（A1 §6）。

## 3. 共有マスター横断リンク（酒蔵ハブ・一般則）

- **一般則**：同じマスターを参照する記録は、そのマスターを**ハブに横断でつながる**（feasibility §4-1f）。
- **第一例＝酒蔵(Brewery)**：御酒印の対象(Event)と酒(銘柄Event)が**同じBreweryを生きた参照**で指す（§1のupsertで同一エンティティが保証される）。
  - **「この酒蔵で飲んだお酒一覧」**＝`Fetch Event where category=酒 かつ brewery==thisBrewery`。
  - **御酒印から「お酒を記録」**＝brewery をプリセットした新規 酒Event を開く（A7）。
  - 双方向（酒→この酒蔵の御酒印）も同様。
- **成立条件**：横断は**生きた参照**を使う。Breweryが削除されればリンクは切れる（スナップショットは表示のみ担保）。
- **将来の同型**：飲んだ店(Venue/PlaceMaster)↔飲食店テンプレ、人物(Person)↔各作品。

## 4. PhotoBlob 拡張（role / caption・09へ追記）★2026-07-06

A3の写真/コレクション区別を実装で成立させる。**09（写真ストレージ仕様）に以下を追記**：

- `role: String = "photo"`：**photo / collection / label / eyecatch** の役割。
- `caption: String = ""`：任意（主にcollection。空でも可・例：観た絵画の作品名）。
- **役割の意味**：
  - photo＝土台の写真（とりあえず入れる・キャプション不要）
  - collection＝U11コレクション（＋caption）
  - label＝U16ラベル表裏
  - eyecatch＝代表アイキャッチ（※Event.eyecatchPath/Visit.eyecatchPath と役割整合。実装は「どのBlobを代表に使うか」の指定でも可）
- **写真↔コレクションの移動＝roleの変更のみ**（実体の再保存・再アップロードなし・caption付与/除去可）。
- CloudKit互換：既定値あり・optional不要（String既定）。externalStorageのdata本体には触れない（roleはメタ列）。

## 5. CloudKit整合（まとめ）

- 全マスター/Event/Visit/PhotoBlob：全プロパティ既定値・`.unique`なし・リレーション全optional（A1 §4）。
- 一意性＝fetch-first upsert（§1）。削除＝nullify＋スナップショット or cascade（明示操作のみ・A1 §6）。
- 同期公開はv2（07）だが本規約でv1スキーマを互換に保つ（後付け不可）。
