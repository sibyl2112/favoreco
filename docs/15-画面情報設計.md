# Favoreco 画面情報設計

> 2026-07-13確定。画面デザインを作る前の情報階層・画面責務の正本。
> Mystoriumの見た目は流用せず、閲覧/編集分離、対象/体験回分離、Draft保存方式だけを参照する。

## 1. 基本原則

1. 閲覧画面に編集コントロールを混在させない。閲覧画面には編集への入口だけを置く。
2. `ExperienceEvent`（作品・公演・本・施設などの対象）と `Visit`（その日に体験した記録）を画面上でも分離する。
3. `Plan`（予定）、`TicketAttempt`（申込1件）、`Visit`（体験済み記録）を同じ入力フォームで同格に扱わない。
4. 一覧/サマリーでは厳選情報、詳細では保存情報を原則すべて表示する。
5. 登録/編集はDraftへ入力し、保存時だけモデルへ反映する。キャンセルでは正本を変更しない。
6. 長い入力は必須情報を常設し、補助情報をアコーディオンへ分ける。

### 対応OS方針

- 最低対応OSはiOS 18とし、保存・登録・閲覧・バックアップなど主要機能を利用可能にする。
- 主対象とUI品質基準はiOS 26以上とする。設計をiOS 18の見た目や操作へ固定せず、iOS 26で利用できる新しいSwiftUIのナビゲーション、スクロール、視覚効果、アクセシビリティ、パフォーマンス改善を優先する。
- iOS 26専用APIは`#available`で分岐し、iOS 18では情報や主要操作を失わない簡潔なフォールバックを提供する。
- 実機の最終デザイン確認はiOS 26を優先し、iOS 18は互換性、レイアウト崩れ、起動、保存データの確認を中心にする。

## 2. 情報階層

```text
Home（全ジャンル横断）
├─ ジャンルトップ
│  ├─ 対象詳細（ExperienceEvent）
│  │  ├─ 記録詳細（Visit）
│  │  └─ 予定詳細（Plan）
│  │     └─ 申込詳細（TicketAttempt）
│  └─ ジャンル内記録
└─ 全記録一覧
   └─ 記録詳細（Visit）
```

## 3. 画面責務

| 画面 | 主役 | 役割 |
|---|---|---|
| Home | 全体の今・思い出 | 全ジャンルの予定、期限、最近の写真、ジャンル入口を横断表示 |
| ジャンルトップ | 対象ライブラリ | そのジャンルで登録済みの作品・施設・銘柄などを探す |
| 記録一覧 | Visitの時系列 | 全ジャンルまたは選択ジャンルの体験履歴を探す |
| 対象詳細 | ExperienceEvent | 作品・公演・本・施設など、何についての記録かを表示 |
| 記録詳細 | Visit | その日に体験した内容をすべて閲覧 |
| 予定詳細 | Plan | これから参加する日時・会場と申込状況を確認 |
| 登録 | 新しいVisit | 必須情報を短く入力し、必要な補足だけ展開 |
| 編集 | 対象/記録/予定のいずれか | 1画面で1種類の正本だけを変更 |

## 4. Home

表示順:

1. 共通ヘッダー: Favoreco、通知、プロフィール
2. 次の予定: 直近の予定をファーストビューの主役とし、複数件を左右スワイプで切り替え
3. お知らせ・アテンション: 締切・入金・発券・会員期限と、対応事項がない状態を同じ位置に表示
4. ジャンル入口: コンパクトな`1段横スクロール`と`4列グリッド`を実機比較して最終決定する
5. 最近の思い出: 写真中心の横スクロール
6. 横断統計: Homeへ置く場合もファーストビューより下で簡潔に表示

次の予定は開催日時が近い順に並べ、左右スワイプで1件ずつ切り替えるページ式カルーセルとする。自由な慣性スクロールではなくカード単位でスナップし、自動切替は行わない。複数件がある時は`1 / N`で現在位置を示してスワイプ可能だと分かるようにする。1件だけの場合は不要な見切れやページ表示を出さず、0件時は`次の予定はありません`と`予定を立てる`を表示する。Home通常表示では大きなブランドメッセージを次の予定より上へ置かない。

セクションの`予定一覧`は独立した重複画面を作らず、下部のカレンダータブを`予定一覧`モードで直接開く。カレンダー画面は`カレンダー / 予定一覧`を切り替えられ、予定一覧では今後のPlanを日付順・月単位でグループ表示する。行から予定詳細を開き、要対応がある予定には期限アイコンを付ける。ジャンル絞り込みを許可し、完了済みVisitは通常の予定一覧へ残さない。中止・見送り・アーカイブはフィルターから確認できるようにする。

お知らせ・アテンションのエリアは常設する。画面幅を使う横長の一覧行とし、横スクロールや自動切替を使わない。対応事項がある時は期限や当落を優先順に最大2件まで縦に表示し、3件以上は`ほかN件`から専用一覧シートへ遷移する。0件時は大きな空白を作らず、チェックアイコンと`対応が必要なお知らせはありません`を1行で表示する。通知ベルのバッジは対応事項がある時だけ表示する。

チケット追加機能をHomeの独立メニューとして常設せず、期限や当落など利用者の対応が必要な時だけお知らせ・アテンションへ表示する。

ジャンル入口は大きなタイルにせず、アイコンと短い名称だけのコンパクトなボタンとする。実機比較では、縦スクロールと干渉しないかを見る`1段横スクロール`と、基本8ジャンルへ直接タップできる`4列×2段`を切り替える。Debugビルドの`設定 > 開発 > Homeジャンル表示`から即時切替でき、比較後に片方へ固定する。カスタムジャンルを含め8件を超える場合、4列案は必要な段数を追加する。

Homeへ長い記録一覧、対象一覧、詳細統計、全ての`気になる`対象一覧を置かない。最近の思い出と最近の記録リストは重複させず、Homeでは写真ギャラリーを優先する。最近の思い出のアイキャッチは保存済みの縦横比を維持し、タイルの横幅へ合わせて表示する。タイルの最終サイズ、比率混在時の整列、画像以外の表示内容はデザイン時に確定する。

Homeに記録追加ショートカット帯は置かず、追加導線はまず下部の`追加`タブへ集約する。実機操作で主要導線が遠いと判断した場合のみ、役割が重複しないショートカットを後から追加する。

## 5. ジャンルトップ

主役はジャンル別の対象ライブラリとする。

表示順:

1. ジャンル切替ヘッダー
2. ジャンル内ミニ統計
3. 直近の予定/要対応（存在時のみ）
4. 対象ライブラリ
5. 最近の記録
6. 検索、並び替え、絞り込み

対象単位の例:

| ジャンル | 対象 |
|---|---|
| 映画 | 作品 |
| 観劇 | 公演 |
| ライブ | ライブ・イベント |
| 本 | 書籍 |
| 美術展 | 展覧会 |
| お酒 | 銘柄 |
| 御朱印 | 寺社 |
| おでかけ施設 | 施設 |

ジャンルトップはHomeの縮小版にしない。横断情報や他ジャンルのギャラリーは置かない。

## 6. 対象詳細

`ExperienceEvent`の閲覧専用画面。

置く情報:

- 対象名、ジャンル、シリーズ、サブ種別
- 代表写真
- 基本となる人物・団体
- 公式URL、対象メモ
- 関連する予定
- 体験履歴
- 記録回数、最新日、平均評価

置かない情報:

- 個別の体験日、座席、当日の金額、当日の写真
- 当日の天気、感想、同行者

これらは記録詳細へ置く。対象編集は別画面へ遷移する。

対象編集では対象名などの基本情報に加えて、対象アイキャッチを追加・変更・解除できる。解除するのは`ExperienceEvent.eyecatchData`だけとし、各記録に保存した写真は削除しない。対象詳細とジャンルトップで使う代表画像の優先順位は、利用者が明示指定した記録写真、対象アイキャッチ、記録写真からの自動候補の順とする。

## 7. 記録詳細

`Visit`の閲覧専用画面。保存情報を原則すべて表示する。

表示順:

1. カバー写真
2. 対象名、ジャンル、体験日
3. 評価または状態
4. 場所、住所、Map
5. 当日の人物・団体
6. チケット、座席
7. 写真ギャラリー
8. OCR・取込内容
9. 金額
10. 天気
11. 公式・参考リンク
12. メモ・感想
13. 詳細オプション

編集コントロールは置かず、右上の編集ボタンから記録編集へ遷移する。

## 8. 一覧・サマリー

一覧カードに出す情報は最大で以下に限定する。

- カバー写真
- タイトル
- ジャンル
- 日付
- 場所
- 評価または現在状態
- メモ1行

人物、金額、OCR、URL、天気は通常一覧に常設しない。期限や未整理など、次の操作に必要な状態だけ補助表示する。

## 9. 思い出の登録

下部タブバーは`Home / 記録 / 追加 / カレンダー / 統計`の5項目を同じ高さ・同じアイコン寸法で並べる。`追加`だけを円形に浮かせたり、タブバーから持ち上げたりしない。`追加`は独立画面へ切り替えず、現在のタブを維持したまま次の4入口を固定順で表示する。

1. `予定を立てる`: これから体験するPlanを登録
2. `体験済みを記録`: 予定を経由しない正式なVisitを直接登録
3. `クイック登録`: 通常と同じExperienceEventを、ジャンル/タイトル必須、アイキャッチ/URL/読み取りメモ/短いメモ任意の簡易フォームで`気になる`として保存
4. `チケットスケジュールを追加`: 対象イベントを検索または最小作成し、そのPlanへ抽選/発売スケジュールを紐付ける

初期状態では`予定を立てる`を既定の追加方法とする。利用者が設定で既定値を変更できるようにしても、入口の並び順自体は動かさない。観劇、ライブ、美術展、おでかけなどは予定起点を基本とし、御朱印、お酒、過去記録の移行などは`体験済みを記録`から直接登録できる。

`予定を立てる`と`体験済みを記録`は対象情報と基本入力部品を共用するが、初期状態を分ける。

| 入口 | 初期表示する情報 | 初期状態で閉じる情報 | 主な表示先 |
|---|---|---|---|
| 予定を立てる | 対象、日時、会場、通知、チケット | 写真、感想、評価 | 次の予定、カレンダー、要対応 |
| 体験済みを記録 | 対象、体験日、写真、感想、評価 | 通知、抽選、発売 | 最近の思い出、記録一覧、統計 |

`予定を立てる`は最初に予定の対象を確定する。主操作は`新しく登録`とし、同じフォーム内でExperienceEventとPlanをまとめて保存する。補助操作の`気になるから選ぶ`では、`気になる`状態の既存ExperienceEventを検索して選び、その対象へ新しいPlanを追加する。検索対象はタイトル、シリーズ、ジャンルとし、文字幅・大文字小文字・空白の差を無視する。選択後は対象名、ジャンル、公式URLを予定フォームへ引き継ぐ。対象詳細から開いた場合は選択を省略し、表示中のExperienceEventへ直接追加する。

Planは必ずExperienceEventへ紐付け、Planだけを単独保存しない。予定日時を持たない`気になる`登録はExperienceEventだけで成立するが、利用者自身の候補日または確定日を入力した時点でPlanを作る。開催日・開催期間など対象そのものの日付と、利用者が行く予定日時は区別する。Plan作成後は対象の`気になる`状態を通常状態へ移す。

予定日を迎えたPlanから`記録を残す`を実行した場合は、対象、日時、会場、チケット参照を引き継いでVisit作成を開始する。元のPlanとTicketAttemptは削除せず完了状態として保持し、同じExperienceEventへVisitを紐付ける。

`チケットスケジュールを追加`は`新規イベント / 登録済みイベント`の2入口とし、チケットだけを独立保存しない。新規イベントではExperienceEvent、Plan、TicketAttemptを同じ保存処理内で作成する。登録済みイベントではタイトル、シリーズ、ジャンルからExperienceEventを検索し、そのイベントに未完了Planがあれば紐付け先を選んでTicketAttemptだけを追加する。文字幅・大文字小文字・空白の差は検索時に無視する。

登録済みイベントにPlanがない場合、または同じイベントに別の予定を作る場合は、選択したExperienceEventへPlanとTicketAttemptをまとめて追加する。既存Planを選んだ後は予定名、日時、会場を参照表示とし、チケット保存によってPlanの日時・会場・公式URLを上書きしない。通常は予定詳細から追加し、下部の`追加`タブはイベント詳細まで移動せず登録するための短縮導線として扱う。

`クイック登録`はPlanやVisitを作らず、通常と同じExperienceEventを`気になる`状態で作る入力項目限定版とする。別のInboxItemへ保存して後から変換する工程は設けない。URLとOCRは独立した登録方式ではなく、同じ画面内でタイトルと画像を埋める入力補助とする。URLから取得したタイトルや、カメラ/写真のOCR先頭行は候補として表示し、利用者が`タイトルに使う`を選んだ時だけ反映する。OCRに使った画像は対象のアイキャッチ候補にし、読み取り全文は全ジャンル共通の`読み取りメモ`へ保存する。後から同じ対象へPlanまたはVisitを追加し、その時点で状態を通常へ移す。

クイック登録後も対象詳細の`対象情報・画像を編集`からアイキャッチを差し替え・解除できる。画像は読み込み時に長辺1600px以内、JPEG品質85%へ圧縮し、元画像のメタデータを保持しない。

登録フロー:

1. `体験済みを記録`を選択
2. コンパクトなジャンルメニューでジャンルを選択。全ジャンル一覧を常時展開しない
3. 主操作の「新しい作品・対象を登録」、または補助操作の「登録済み作品・対象に記録を追加」を選択
4. 既存対象の場合は検索結果から対象を選び、対象側情報を参照して今回固有のVisitだけを入力
5. 新規対象の場合は対象名だけで作成可能とし、必要に応じてシリーズを検索・連結
6. 記録の必須情報を入力
7. 必要な補助ユニットだけ展開
8. 保存後に記録詳細を開く

下部の`追加`タブの`体験済みを記録`では、ジャンルをコンパクトなメニューで選んだあと、主操作の`新しい作品・対象を登録`と、補助操作の`登録済み作品・対象に記録を追加`を同じ選択画面に表示する。既存対象は選択ジャンル内をタイトルまたはシリーズ名で検索でき、未入力時は最近更新した対象を少数だけ表示する。

予定詳細の`参加記録を入力`はVisitを即時作成せず、同じ記録入力フォームを開く。対象、予定日時、終了日時、会場、座席、金額、メモ、公式URL/購入URLを初期値として引き継ぎ、利用者が確認・修正して保存した時だけVisitを作成する。保存成功と同じ処理内でPlanを参加済みにし、代表のTicketAttemptも参加済みに更新する。入力をキャンセルした場合はPlanとTicketAttemptを変更しない。

入口の情報優先順位:

```text
ジャンル [観劇 ▼]

[ ＋ 新しい作品・対象を登録 ]

または

登録済み作品・対象に記録を追加
[ タイトルを検索 ]
```

「既存対象への記録追加」と「シリーズ連結」は別機能とする。既存対象への追加は再鑑賞、再訪、再読、同じ銘柄の再記録に使う。シリーズ連結は新しい対象を別作品として作成したうえで任意の親シリーズへ紐付ける。

既存対象を選んだ場合に参照する対象側情報:

- タイトル、ジャンル、シリーズ、代表写真
- 基本の人物・団体、公式URL、対象種別

Visitへ入力する今回固有の情報:

- 体験日、会場、当日の人物・団体
- 写真、評価、座席、金額、感想、天気

対象情報をVisitへ無条件コピーせず、対象はリレーション参照、当日固有情報はVisitのスナップショットとして保存する。

常時表示:

- ジャンル
- 対象名
- 日付
- 場所
- 評価または状態
- メモ

アコーディオン:

- 人物・団体
- 写真
- チケット・座席
- OCR・取込
- 金額
- 公式情報
- 詳細オプション

新規対象を作る場合でも、対象側の詳細情報をすべて同時入力させない。まず名称だけで対象を作成でき、対象詳細から後で補完できる。

## 10. 編集画面

編集画面は次の3種類へ分離する。

### 対象を編集

- 対象名
- シリーズ、サブ種別
- 代表写真
- 基本となる人物・団体
- 公式情報
- 対象メモ

### 記録を編集

- 日付、場所
- 評価、状態
- 当日の人物・団体
- 写真、OCR
- 座席、金額、天気
- 感想、詳細オプション

対象名や対象の公式情報を変更しない。

### 予定・チケットを編集

- 開催日時、開場、会場
- 抽選または発売・予約のスケジュール
- 取得後のチケットメモ、リンク、任意のQR・画像
- 発券・表示確認と任意の確認リマインダー
- 予定メモ

体験済みの感想や写真を変更しない。参加後はPlanからVisit作成へ遷移する。

## 11. 御朱印専用登録フロー

御朱印は一般的な対象選択より先に、登録先の御朱印帳を選ぶ。

```text
御朱印を登録
↓
使用中の御朱印帳を選択
↓
新しい寺社 / 登録済み寺社を選択
↓
御朱印を入力
↓
保存して完了 / 保存して同じ寺社で続けて登録
```

御朱印帳選択画面:

- 使用中の御朱印帳を表紙画像、名称、登録済み御朱印件数で表示
- 同じ利用者が寺社や用途ごとに複数冊を並行利用できる
- `新しい御朱印帳を登録`を補助操作として同じ画面へ置く
- 閉じた御朱印帳は新規追加候補に出さないが、過去帳一覧から閲覧・編集できる

御朱印帳の管理:

- ページ数、総ページ容量、使用ページ数を管理しない
- 裏面利用、見開き、書置きなどの差を自動判定しない
- 満了の自動判定、自動クローズを行わない
- 利用者の明示操作だけで御朱印帳を閉じる
- 閉じた御朱印帳は新規御朱印を追加できないが、既存御朱印と帳面情報は編集できる
- 御朱印帳には登録済み御朱印の件数を表示する

寺社選択:

```text
[ ＋ 新しい寺社の御朱印を登録 ]

または

登録済み寺社から選ぶ
[ 寺社名を検索 ]
```

御朱印登録項目:

- 御朱印画像/アイキャッチ: 任意
- 寺社名: 必須
- よみ: PlaceMasterの読みを寺社名上部へ薄いルビとして表示
- 住所、ミニMap、公式URL、御祭神: 任意。登録済み寺社から引継可能
- 参拝・拝受日時、写真、同行者、メモ: 任意
- 御朱印名/種類、通常/限定、直書き/書置き: 任意
- 初穂料・納経料: 任意。`300円 / 500円 / その他`をクイック選択し、その他は任意金額を入力

保存操作:

- `保存して完了`
- `保存して同じ寺社で続けて登録`

続けて登録では御朱印帳、寺社、日時、同行者を保持する。御朱印画像、御朱印名/種類、金額、写真、メモは次の記録へ引き継がない。

御朱印帳の基本情報:

- 表紙画像
- 御朱印帳名、または購入した寺社
- `神社用 / お寺用 / 共用`
- 使い始めた日
- メモ

御朱印帳は長押しから編集、閉じる/再開、削除へ進める。削除は帳面内の御朱印と写真を含むため、対象件数表示、バックアップ案内、`削除する`の確認入力を含む3段階確認にする。削除しても寺社マスターと別の御朱印帳は削除しない。

## 12. 共通入力ルール

### OCR・URL取込

OCRとURL取込は保存操作ではなく入力補助とする。

1. 画像、URL、貼り付けテキストを読み取る
2. タイトル、日付、会場、人物、金額、ISBNなどの反映先を候補表示する
3. 利用者が候補ごとに反映先変更、編集、除外を行う
4. `反映する`を押した項目だけフォームへ入れる

- 信頼度が低い候補は要確認表示にする
- 既存入力を自動上書きしない
- OCR全文を確認画面へ残し、選択、コピー、貼り直しを可能にする
- 公演系は登録画面上部に`公式サイトから情報を取得`と`チラシ・チケットをOCRで読み取る`を常設する
- 本はISBNバーコードを優先し、奥付OCRを予備経路にする。ISBNは任意の内部識別情報として保持し、通常入力で手入力を要求しない

### 人物・団体の重複候補

人物名の検索用文字列は、Unicode正規化後に半角/全角スペース、連続空白、改行、タブを除去し、英字の大文字/小文字を無視する。

```text
山田 太郎
山田　太郎
山田太郎
```

上記は同一候補として提示する。表示名の原文は保持し、自動統合せず、既存人物を使うか別人物として作るかを利用者が選ぶ。この規則は著者、俳優、アーティスト、監督、作家などPersonMasterを使う全ジャンルへ適用する。

### SNSタグ

全ジャンルの対象と記録に、複数登録可能な任意のSNSタグを持たせる。

- `#`なしの入力は表示時に補完する
- 全角/半角スペースを無視して重複候補を出す
- URL/OCRから公式タグ候補を提案する
- タグ群をまとめてコピーできる
- 通常は`公式情報`アコーディオン内へ置く

### 日付

期間は終了日を必須にしない。開始日だけで保存でき、終了日がない対象を無期限の`開催中`とは自動判定しない。

### 写真と写真に紐づく情報

写真を用途別に二重保存しない。共通写真へ任意の作品情報または体験情報を付け、詳細画面で意味のあるまとまりとして再構成する。

- 美術展: 作品名、作家名、メモ
- おでかけ: 名前、種類、メモ、任意の金額
- 写真がない場合はテキストだけの作品/体験メモを追加できる
- 情報付き写真も通常の写真ギャラリーへ含める

### 評価

映画と本は、記録回ごとに`1.0...5.0`を0.5刻みで評価できる。ジャンルごとに機能ON/OFF可能、初期値はOFF、未評価保存を許可する。OFFにしても保存済み評価は削除しない。一覧への評価表示は別設定とする。

## 13. ジャンル別登録フロー

### 映画

```text
映画を記録
├ 新しい映画を登録
└ 登録済み映画に鑑賞記録を追加 [検索]
```

映画情報:

- タイトル必須
- 読み、原題、ポスター、公開年、監督、出演者、制作会社、公式URLは任意
- 1作品は複数のコレクションへ所属できる
- コレクションは代表画像、作品数、作品一覧、公開順/時系列/任意順を持つ

鑑賞記録:

- 鑑賞予定/鑑賞済み、鑑賞日、映画館、上映形式、同行者、写真/半券/OCR、金額、感想/メモ、任意評価
- 再鑑賞は映画情報を参照し、新しい鑑賞記録だけを追加する

映画トップはコレクションカードを横スクロール表示し、`一覧を表示`からコレクション一覧へ進む。主要都市のシネコンとミニシアターをPlaceMasterの初期候補へ収録する。

### 観劇

通常利用は`公演 -> 観劇回`の2層とし、作品/演目の理解を登録時に要求しない。再演をまとめる機能は内部拡張余地を残し、必要な利用者だけ後から使えるようにする。

公演情報:

- 公演名必須
- メイン画像/チラシ、公演種別、劇団/主催、公式URLは任意
- 人物・団体は閉じたアコーディオンに置き、原作/原案、作/脚本、演出、出演者、音楽、振付などを役割付きで複数登録できる
- 公演名と主催が近い既存公演がある場合は、情報を引き継ぐか新規作成するか確認する。自動統合しない
- チラシ/ポスターは公演、当日の写真/半券は観劇回へ保存する

観劇回:

- 観劇日時、会場、座席自由入力、当日の出演者差分、写真/半券、金額、同行者、感想/メモ
- 開場時刻は予定/チケット側、上演時間は公演側の任意項目、休憩は標準項目にしない
- 感情タグ初期候補は`また観たい / 泣いた / 笑った / 驚いた / 圧倒された / 考えさせられた / 音楽が好き / 演出が好き / 衣装・美術が好き`

公演種別初期候補:

- 演劇、ミュージカル、歌舞伎、落語・寄席、ダンス・舞踊、バレエ、オペラ、コンサート、アイスショー、マジック・サーカス、その他
- カスタム種別は名前、アイコン、並び順、非表示を管理できる
- 使用済み種別は削除せず、別種別への統合または非表示にする

一覧サマリーはチラシ、公演名、観劇日、劇場、予定/観劇済みに限定する。

### ライブ

通常利用は`ライブ・イベント -> 参加回`の2層とし、入口はアーティストではなくライブ名から始める。

```text
ライブを記録
├ 新しいライブを登録
└ 登録済みライブに参加記録を追加 [検索]
```

ライブ情報:

- ライブ名必須
- 種別は`ワンマン / 対バン / フェスティバル / ライブイベント / 配信ライブ / その他`
- アーティストは最初から表示して複数登録可能。役割は`メイン / 出演 / ゲスト / オープニングアクト`を任意設定する
- ツアーは種別にせず、必要な場合だけ関連ライブとして後からまとめる

参加回:

- 参加日時、会場、写真/アイキャッチ、感想/メモを基本とする
- 座席/整理番号、当日のゲスト、セットリスト、同行者、金額、公式情報は閉じた任意項目
- セットリストは曲名と順番を持ち、まとめて貼り付け可能。アンコールは区切りで表現する
- フェスは出演者全員と実際に観たアーティストを分け、詳細ON時だけアーティスト別セットリスト、ステージ、時間帯を記録する
- 配信ライブは会場の代わりに配信サービスと視聴URLを使う
- 感情タグ初期候補は`また行きたい / 泣いた / 笑った / 驚いた / 圧倒された / 盛り上がった / 音が良かった / 演出が好き / セットリストが好き`

ライブトップは今後のライブ、アーティスト、参加したライブ、データがある場合だけツアー/関連ライブを表示する。

### 本

通常利用は`本 -> 読書記録`の2層とする。

```text
本を記録
├ 新しい本を登録
└ 登録済みの本に読書記録を追加 [タイトル・著者検索]
```

本の登録はISBNバーコード、表紙/奥付OCR、タイトル入力から開始できる。本の情報は表紙、タイトル必須、著者、巻数、出版社、発売日、ISBN、公式/参考URLを持つ。

- 1冊は複数のシリーズ/コレクションへ所属できる
- ISBNが異なる単行本、文庫、電子版を同じシリーズへ入れられる
- 状態は`気になる / 積読 / 読書中 / 読了 / 中断`
- クイック登録は気になる、開始日あり/読了日なしは読書中、読了日ありは読了を自動候補にする
- 積読と中断は利用者が選択し、自動判定後も手動修正できる
- 読書記録は開始日、読了日、`紙 / 電子 / オーディオ`、感想/メモ、写真、任意評価を持つ
- 再読は同じ本へ新しい読書記録を追加する
- 保管場所、貸出状況、読書タイマー、読書速度、毎日のページ進捗は扱わない

本の種別初期候補は`小説 / エッセイ / ノンフィクション / ビジネス・経済 / 実用・趣味 / 人文・思想 / 科学・技術 / 漫画 / 絵本 / 詩歌 / 雑誌 / その他`とし、カスタム管理可能にする。印象タグ初期候補は`また読みたい / 心に残った / 泣いた / 笑った / 驚いた / 考えさせられた / 学びがあった / 読みやすかった`とする。引用は閉じた任意項目で、本文、ページ、メモを複数登録できる。

本トップは読書中、気になる/積読、最近読んだ本、シリーズ/コレクションを表示する。

### 美術展・展示会

通常利用は`展覧会 -> 鑑賞記録`の2層とする。同じ巡回展は展覧会を共通化し、会場と鑑賞日は鑑賞記録へ持たせる。会場ごとに内容やタイトルが大きく違う場合だけ別展覧会として関連付ける。

展覧会情報:

- 展覧会名必須
- メイン画像/チラシ、開催開始日、任意の終了日、作家/アーティスト、主催/企画、公式URL、SNSタグ
- 排他的な単一種別を使わず、`現代美術 / 絵画 / 彫刻 / 写真 / 映像 / インスタレーション / メディアアート / 建築 / デザイン / 工芸 / ファッション / 漫画・アニメ / 歴史・文化 / 科学・自然`などを複数選択できるジャンル・テーマとして扱う

鑑賞記録:

- 鑑賞日、会場、写真、同行者、金額、感想/メモ、印象タグ
- 写真へ作品名、作家名、メモを付けると`印象に残った作品`として自動表示する
- 写真がない作品はテキストだけで記録できる
- 印象タグ初期候補は`また行きたい / 心に残った / 驚いた / 圧倒された / 考えさせられた / 展示空間が好き / 作品構成が好き`

展覧会トップは開催中/これから行きたい展覧会、チケットスケジュール最大2行、最近観た展覧会、アーティスト、美術館/ギャラリー、写真ギャラリーを表示する。

### おでかけ施設

```text
おでかけを記録
↓
テーマパーク / 動物園 / 水族館 / 植物園 / その他
↓
お気に入り、最近使った施設、全施設から選択 / 新しい施設を登録
↓
訪問記録を入力
```

施設情報は施設名必須、種別、メイン画像、読み、住所/Map、公式URL、SNSタグ、お気に入りを持つ。訪問記録は訪問日、写真/アイキャッチ、同行者、チケット/予約、金額、天気、体験、感想/メモ、SNSタグを持つ。

体験情報:

- 共通写真へ体験名、種類、メモ、任意金額を付ける。写真なしでも登録可能
- テーマパークの初期候補は`アトラクション / ショー・パレード / グリーティング / イベント / フード・ドリンク / グッズ・購入品 / その他`
- 動物園、水族館、植物園では表示候補を動物、生き物、植物、季節展示、餌やり/体験などへ変える
- 体験種類はカスタム追加、並び替え、非表示、統合可能

おでかけトップはこれからのおでかけ、チケットスケジュール最大2行、お気に入り施設、最近行った施設、写真ギャラリーを表示する。

### お酒

```text
お酒を記録
├ 新しいお酒を登録
└ 登録済みのお酒に飲酒記録を追加 [検索]
```

- 初期種別は日本酒、ワイン、ビール、ウイスキー、焼酎、その他
- 既存対象は種別、銘柄、メーカー/蔵、産地、ラベル、仕様、公式URLを参照する
- 飲酒記録は日付、場所、写真、飲み方、金額、同行者、感想/メモを持つ
- 軽い印象は`また飲みたい / 好み / ふつう / 合わなかった`の単一選択、未選択可
- 香り、甘辛、軽重、酸味、余韻、飲みやすさ等の詳細評価は機能ON時だけ表示し、酒種別でラベルを変える

## 14. チケット共通フロー

チケット登録の入口は次の3択とする。

```text
チケットを登録
├ 抽選に申し込む
├ 発売・予約を待つ
└ 取得済みチケットを登録
```

抽選:

- 申込開始、申込締切、当落発表、入金期限、発券開始、発券/受取期限を、分かる項目だけ登録する
- 申込URLからプレイガイドを自動候補にし、URLがない場合だけプレイガイドの一般入口を利用できる

発売・予約:

- 発売/予約開始を基本とし、販売終了、発券開始、発券/受取期限は任意
- 先行販売、一般発売、日時指定予約、当日券は販売区分として扱う

取得済み:

- 購入メールや案内を崩さず貼れるチケットメモを中心にする
- メモ内URLをタップ可能にし、対応アプリがあればアプリ、なければブラウザで開く
- プレイガイド、座席、番号、枚数等の個別フォームを標準表示しない
- 画像/PDF、発券・表示確認、確認リマインダーは任意
- `チケットQR・画像`は通常写真と分け、年間まとめ、写真ギャラリー、共有画像へ出さない
- QR全画面表示中は明るさと自動ロックを入場確認向けに調整する
- 動的QRやスクリーンショット無効の場合があるため、公式サイト/アプリ表示を優先する注意を出す

開催日時と会場は公演/展覧会/予定側を参照し、チケットへ重複保存しない。ジャンルトップのチケットスケジュールは共通チケットデータから直近1〜2件を表示し、登録がなければセクションごと隠す。

## 15. 場所・種別・タグのマスターUI

会場入力は利用者へ施設/ホールの階層選択を要求せず、1つの検索/自由入力欄にする。

```text
会場 [東京芸術劇場 コンサートホール]
```

- 施設名だけでも、ホール名込みでも保存可能
- 住所がある場合は名称検索ではなく住所を優先してMap表示する
- 主要都市圏の映画館、劇場/ホール、美術館/博物館/ギャラリー、おでかけ施設を初期候補として収録する
- 内部で親施設を共有しても、通常登録では細かな施設階層を見せない

カスタム種別と印象タグは、名前、アイコン、色、並び順、表示/非表示を管理できる。未使用のカスタム項目は削除可能、使用済み項目は削除せず別項目へ統合または非表示にする。初期項目は削除せず非表示のみ可能とする。

## 16. 実装移行方針

1. 既存モデルは維持し、まずViewの責務を分離する。
2. `EditExperienceView`が同時更新しているExperienceEvent/Visitを、対象編集と記録編集へ分割する。
3. `AddExperienceView`の先頭に既存対象検索を追加し、新規対象作成と既存対象へのVisit追加を同じ入口から選べるようにする。
4. `CategoryTopView`は対象ライブラリを主役に再構成する。
5. `ExperienceDetailView`と`EventDetailView`を閲覧専用として整理する。
6. View分割後にデザインラフを各画面へ適用する。

## 17. 完成条件

- どの情報がExperienceEvent、Visit、Plan、TicketAttemptのどれに属するか画面上で判断できる
- 一覧から編集コントロールが排除されている
- 対象編集で過去Visitの固有情報が変わらない
- 記録編集で対象名や対象公式情報が意図せず変わらない
- Plan編集とVisit編集が混在しない
- 登録は必須情報だけで保存でき、詳細入力は任意展開できる
