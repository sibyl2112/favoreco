# Favoreco 画面情報設計

> 2026-07-13確定。画面デザインを作る前の情報階層・画面責務の正本。
> Mystoriumの見た目は流用せず、閲覧/編集分離、対象/体験回分離、Draft保存方式だけを参照する。
> 2026-07-21更新: 下部タブの`記録`は`FAVO`へ変更し、推し専用ではなく人物・場所・作品・体験を含む「自分の好き」から横断して振り返るページとする。本文中の旧`記録`タブ表記より本更新を優先する。

## 1. 基本原則

1. 閲覧画面に編集コントロールを混在させない。閲覧画面には編集への入口だけを置く。
2. `ExperienceEvent`（作品・公演・本・施設などの対象）と `Visit`（その日に体験した記録）を画面上でも分離する。
3. `Plan`（予定）、`TicketAttempt`（申込1件）、`Visit`（体験済み記録）を同じ入力フォームで同格に扱わない。
4. 一覧/サマリーでは厳選情報、詳細では保存情報を原則すべて表示する。
5. 登録/編集はDraftへ入力し、保存時だけモデルへ反映する。キャンセルでは正本を変更しない。
6. 長い入力は必須情報を常設し、補助情報をアコーディオンへ分ける。

### 対応OS方針

- 最低対応OSはiOS 18とし、保存・登録・閲覧・バックアップなど主要機能を利用可能にする。
- 主対象とUI品質基準はiOS 26以上とする。設計をiOS 18の見た目や操作へ固定せず、iOS 26で利用できる新しいSwiftUIのナビゲーション、スクロール、視覚効果、アクセシビリティ、パフォーマンス改善を優先する。
- iOS 26専用APIは`#available`で分岐し、iOS 18では情報や主要操作を失わない簡潔なフォールバックを提供する。
- 実機の最終デザイン確認はiOS 26を優先し、iOS 18は互換性、レイアウト崩れ、起動、保存データの確認を中心にする。

## 2. 情報階層

```text
Home（全ジャンル横断）
├─ ジャンルトップ
│  ├─ 対象詳細（ExperienceEvent）
│  │  ├─ 記録詳細（Visit）
│  │  └─ 予定詳細（Plan）
│  │     └─ 申込詳細（TicketAttempt）
│  └─ ジャンル内記録
└─ 全記録一覧
   └─ 記録詳細（Visit）
```

## 3. 画面責務

| 画面 | 主役 | 役割 |
|---|---|---|
| Home | 全体の今・思い出 | 全ジャンルの予定、期限、最近の写真、ジャンル入口を横断表示 |
| ジャンルトップ | 対象ライブラリ | そのジャンルで登録済みの作品・施設・銘柄などを探す |
| 記録一覧 | Visitの時系列 | 全ジャンルまたは選択ジャンルの体験履歴を探す |
| 対象詳細 | ExperienceEvent | 作品・公演・本・施設など、何についての記録かを表示 |
| 記録詳細 | Visit | その日に体験した内容をすべて閲覧 |
| 予定詳細 | Plan | これから参加する日時・会場と申込状況を確認 |
| 登録 | 新しいVisit | 必須情報を短く入力し、必要な補足だけ展開 |
| 編集 | 対象/記録/予定のいずれか | 1画面で1種類の正本だけを変更 |

記録一覧の検索欄直下には、`すべて`と記録が存在するジャンルを並べた横スクロール式チェックボックスを置く。初期状態は`すべて`を選択し、ジャンルを押すと`すべて`を解除してそのジャンルだけへ絞り込む。その後は複数ジャンルを同時選択でき、最後の選択を外した場合は`すべて`へ戻る。その下中央に`詳細検索`ボタンを置き、右上のフィルターアイコンは廃止する。詳細検索はNavigationStack内の独立ページとして開き、`日付・期間 / タグ / 同行者 / 写真有無 / 並び順`を表示する。タグと同行者は記録に保存済みの値だけを2列チェックボックスで提示し、複数選択できる。同一項目内はOR、期間・タグ・同行者・写真・ジャンル間はANDで絞り込む。期間は`すべて / 今月 / 今年 / 期間指定`から選び、期間指定時だけ開始日と終了日を表示する。開始日・終了日は両日を含み、開始日を終了日より後へ変更した場合は終了日も同日に揃える。条件クリアではジャンルを含むすべての条件を初期状態へ戻す。

上部の自由検索はタイトル・シリーズ・会場・メモだけでなく、キャスト、監督、主催、作者などの人物/団体名と役割、タグ、同行者、OCR文字、自由追加項目、座席、場所名・住所を横断する。空白区切りの複数語はAND条件とし、各語が異なる保存項目に存在しても一致扱いにする。

記録一覧の件数右へ、ジャンルトップと同じ3列ギャラリー、2列ミニ詳細、横長バナーの表示切替を置く。3列は保存したアイキャッチ比率の画像と`日付｜評価`の1行情報帯、2列は画像・2行タイトル・日付・5段階評価、横長は画像・ジャンル別の体験済み状態・2行タイトル・日付・会場を表示する。画像は各対象のアイキャッチ比率を維持した中央aspect fillとし、映画・観劇のポスター、書影、写真系ジャンルを同じ基準で扱う。選択した表示は次回も維持し、旧3列/4列/横長の保存値は対応する新表示へ引き継ぐ。予定・チケット入口は記録画面から外し、カレンダー上部のチケットアイコンへ集約する。

## 4. Home

Home / 記録 / カレンダー / 統計のルート画面は、ステータスバー直下に共通の固定ヘッダーを置く。左の画面名は1行で全体を見せ、Homeのブランド名は先頭を大文字にした `Favoreco` とし、34ptのCormorant Garamond Boldで表示する。ブランド色はライト時にアプリアイコン中央のFに合わせた青墨色 `#192735`、ダーク時は可読性を保つ明るい青灰色とする。ほかの画面名は30ptを基準とする。右は22ptの通知ベルと38ptのプロフィールの順とする。2操作はそれぞれ44ptの押下領域を持つが、ピル、カード、外側の共通背景で囲わない。ヘッダーは安全領域内の基準位置から6pt上へ寄せ、下端にテーマ色を薄くした全画面幅1ptの区切り線を置く。iOS 26のToolbar自動装飾でダークモードに黒いコンテナが出るのを避けるため、ルート画面のヘッダーは通常のHStackで描画する。詳細画面の戻るナビゲーションは従来どおりシステムのNavigationBarを使う。

表示順:

1. 共通ヘッダー: Favoreco、通知、プロフィール
2. ジャンル入口: 共通ヘッダー直下のテキストピルによるコンパクトな`1段横スクロール`を基本とする
3. `PICK UP`: 未来予定と最近記録を混在したHeroを、複数件は全幅ページングで切り替え
4. `次にやること`: 日時と状態から必要なチケット対応を上位3件＋全件シートで表示
5. `Ticket Schedule`: 全ジャンルの進行中TicketAttemptを共通の工程カードで表示
6. `Interesting`: 気になる対象と未整理Inboxを、ポスター／タイル／横長の固定レイアウトで表示
7. `Coming Up`: 全ジャンルの未来Plan/Visitを直近1件＋下方向展開で表示
8. 最近の思い出・最近の記録
9. 横断統計: Homeへ置く場合もファーストビューより下で簡潔に表示

次の予定は開催日時が近い順に並べ、左右スワイプで1件ずつ切り替えるページ式カルーセルとする。自由な慣性スクロールではなくカード単位でスナップし、自動切替は行わない。複数件がある時は、先頭では右側に次カード、末尾では左側に前カード、中間では左右に前後カードを少し見切れさせ、横スワイプできることを視覚的に示す。現在位置は`1 / N`ではなく件数分の丸で示し、選択中の丸だけをテーマ色にする。カード枠と丸の間、丸と次セクションの間は必要最小限に詰める。1件だけの場合は不要な見切れやページ表示を出さない。0件時も実データの予定カードと同等の高さを確保した枠を表示し、その枠内に`次の予定はありません`と`＋ 予定を立てる`を置く。Home通常表示では大きなブランドメッセージを次の予定より上へ置かない。

次の予定カードは、内側の利用可能幅の約46%（最大148pt）をポスター/チラシ/書影に使い、残りを公演情報に使う。アイキャッチはジャンル既定または記録に保存した正式な縦横比を変形せず、画像幅から成り行きで決まる画像高を使う。総合HomeのHeroは、観劇・映画・美術展で共通するB判`1:1.414`の縦長枠とカード内余白を欠けずに収めつつ、操作列下の余白を抑えた248ptを共通カード高とし、複数件の横ページング枠も同じ高さにする。横長・正方形など他ジャンルもこの高さへ揃え、画像の下を無理に伸ばさず右側情報欄の余白で情報量の差を吸収する。画像と情報欄の間隔は10pt、情報欄の行間は4ptを基準とする。右側にはジャンル、タイトル、公演名、日時、会場を収め、画像と情報をカード外へはみ出させない。タイトルは常に2行分の高さを予約し、3行目以降は末尾省略する。操作列の前へ可変余白を置き、Planカード下部の`予定詳細`と`編集`、未来日Visitカードの`記録詳細`と`編集`をカード下端へ揃える。画像・情報欄・主ボタンは同じ詳細、編集だけは各編集シートを開く。

`次の予定`の表示対象は、未終了の`Plan`に加えて、参加日を今日以降へ変更した`Visit`も含める。日付変更だけで暗黙の`Plan`は新規作成せず、未来日の記録を表示上の予定として扱う。同じ`Visit`に表示対象の`Plan`が紐付いている場合は`Plan`を優先し、二重表示しない。両者を日時順にまとめ、`Plan`からは予定詳細、未来日の`Visit`からは記録詳細へ遷移する。

セクションの`予定一覧`は独立した重複画面を作らず、下部のカレンダータブを`予定一覧`モードで直接開く。カレンダー画面は`カレンダー / 予定一覧`を切り替えられ、予定一覧では今後のPlanを日付順・月単位でグループ表示する。行から予定詳細を開き、要対応がある予定には期限アイコンを付ける。ジャンル絞り込みを許可し、完了済みVisitは通常の予定一覧へ残さない。中止・見送り・アーカイブはフィルターから確認できるようにする。

`次にやること`は画面幅を使う高さ40ptのカプセル型1行表示とし、`アイコン / 日付・やること・タイトル`の順で並べる。タイトルが長い場合は末尾を省略し、日付とやることを優先して残す。横スクロールや自動切替は使わない。各TicketAttemptは現在状態に対応する申込・締切・当落・入金・発券の1項目だけを出し、期限超過を先頭、その後を日時順に並べる。Homeは上位3件まで縦に表示し、4件以上は`ほかN件を見る`から全件シートへ遷移する。行は予定詳細を開き、状態更新後は該当項目を外して次項目を繰り上げる。0件時も同じ高さのカプセルで、チェックアイコンと`今すぐ対応することはありません`を1行表示する。発売・当落・入金・発券の日時到達後または会員期限が近い場合だけ、通知ベルへ赤いマークを表示する。ベルを開いただけでは消さず、チケット状態の更新または期限対象外への変化で消す。

Homeの`次にやること`の日付は`7/18(土)`の半角括弧・括弧前余白なしとし、日付・対応内容・タイトル間は左右間隔を詰めた`|`で区切る。

チケット追加機能をHomeの独立メニューとして常設せず、期限や当落など利用者の対応が必要な時だけ`次にやること`へ表示する。

ジャンル入口は大きなタイルやアイコンを使わず、`総合 / 観劇 / 美術展 / ライブ...`の短い名称を明朝体の1段横スクロールで表示する。先頭の`総合`は全件一覧ではなく、予定・チケット・最近の記録をジャンル横断でまとめるHomeを意味する。Homeとジャンルトップで同じ`GenreNavigationStrip`を使い、ジャンル間は細い縦線、選択中はジャンル色の文字＋2pt下線、メニュー下は全幅1pt線とする。ジャンルトップ中央のジャンル名は見出しに限定し、直下メニューと重複する▼、タップ選択、一覧ダイアログは置かない。総合HomeもHeroを除いた本文へジャンルトップと同じ方向限定スワイプを適用する。左スワイプでは画面上の右隣へ進む`総合 → 先頭ジャンル → … → 最終ジャンル → 総合`、右スワイプではその逆順に循環する。左右24ptの画面端、Hero、内側の横スクロールチップは認識対象外とする。実機で横スワイプと縦スクロールの干渉、縦線と下線のコントラスト、選択ジャンルの自動中央寄せを確認する。

Homeの英字セクション見出し（`PICK UP / Ticket Schedule / Interesting / Coming Up`）はCormorant Garamond 22pt Semibold、その他の主要見出しはNoto Serif JP 17pt Boldとし、左上 `Favoreco` と同じ適応ブランド色で表示する。`Interesting`は観劇ライブラリと同じ3列ポスター／2列ミニ詳細／横長バナーの切替を持つ。横方向へ一覧を送らず、初期9/8/6件と`さらにn件`で下方向へ展開する。見出し右のシェブロンで開閉し、閉じた時は見出しと件数だけを残す。`Coming Up`も横カルーセルにせず、最初の1件と残件の下方向展開にする。

Homeは全ジャンルの予定、期限、思い出をまとめる横断ダッシュボードとし、公式サイトに合わせたクリーム色を基調に、桃色・青緑・ベージュがごく薄く移る背景と微細な紙粒子を使う。青緑のニュアンスは残すが、桃色・薄赤ベージュがわずかに優勢な配色とする。紙粒子は操作を妨げず、カードや文字の可読性を落とさない固定背景とする。ジャンル固有の対象一覧や長い記録一覧は置かない。

Homeへ長い記録一覧、対象一覧、詳細統計、全ての`気になる`対象一覧を置かない。最近の思い出と最近の記録リストは重複させず、Homeでは写真ギャラリーを優先する。最近の思い出は横スクロールを使わず、最大8件を2列の縦グリッドで表示する。アイキャッチは保存済みの縦横比を維持し、各列の幅へ合わせて表示する。

Homeに記録追加ショートカット帯は置かず、追加導線はまず下部の`追加`タブへ集約する。実機操作で主要導線が遠いと判断した場合のみ、役割が重複しないショートカットを後から追加する。

## 4.1 更新案内

アプリのMarketing Versionが前回確認済みのバージョンと異なる場合、オンボーディング完了後の最初の起動で更新概要をシート表示する。同じバージョンでは繰り返し表示しない。新規インストールではオンボーディング前に現在バージョンを確認済みとして保存し、初回利用を更新案内で遮らない。

更新概要はネット接続なしでも読める短いアプリ内データとし、版番号、見出し、要約、主な変更を表示する。`詳細な更新履歴を見る`は`https://ranoviqo.com/favoreco/`を開く。Web側の掲載先をNotion等へ変える場合はこの固定URLから転送し、アプリ更新を不要にする。設定の`サポート > 更新履歴`では、確認済みを含む全更新履歴をいつでも閲覧できる。

## 5. ジャンルトップ

主役はジャンル別の対象ライブラリとする。

表示順:

1. ジャンル切替ヘッダー
2. ジャンル内ミニ統計
3. 直近の予定/要対応（存在時のみ）
4. 対象ライブラリ
5. 最近の記録
6. 検索、並び替え、絞り込み

対象単位の例:

| ジャンル | 対象 |
|---|---|
| 映画 | 作品 |
| 観劇 | 公演 |
| ライブ | ライブ・イベント |
| 本 | 書籍 |
| ミュージアム | 展示・イベント |
| お酒 | 銘柄 |
| 御朱印 | 寺社 |
| おでかけ施設 | 施設 |

ジャンルトップはHomeの縮小版にしない。横断情報や他ジャンルのギャラリーは置かない。

おでかけ施設はパークブルー`#2F7FB8`を標準ジャンル色とする。背景は白を主体に上部へ薄い水色のグラデーションを重ね、選択状態、カード枠、ボタン、ミニ統計、訪問場所MAPのマーカーへ同じ青を展開する。濃紺は文字色と写真のコントラストを支える補助色とし、画面全体を暗色にはしない。

従来の`美術展`は利用者向け名称を`ミュージアム`へ変更し、美術館・博物館・科学館・資料館の展示／イベントを扱う。上部統計は`訪れた館 / 鑑賞イベント / 気になる`とし、前者はPlaceMasterまたは入力済み館名の重複なし件数、後者は鑑賞記録がある展示・イベントの重複なし件数とする。館名未入力の記録は訪れた館へ推測加算しない。

`Coming Up / Interests`Heroはタイトルが1行でも2行分の高さを確保し、その下にサブタイトル、日付、会場を積む。サブタイトルは通常のセカンダリ色より少し濃くする。予定カード下部は`予定詳細 / 編集`とし、主ボタンを12pt、編集を10.5pt、アイコンと文言の間を3pt、高さを34ptへ揃える。画像・情報欄・`予定詳細`は同じ予定詳細へ進み、編集だけは保存済みPlanの編集シートを直接開く。対象一覧タイルは作品・公演単位の対象詳細であり、予定Heroの遷移先とは役割を分ける。

ジャンルトップもHomeと同じ固定ヘッダー部品を使う。左はHomeへ戻る27ptの控えめな`Favoreco`、中央は現在のジャンル名をNoto Serif JP 20pt Bold・文字間隔5ptで表示し、右はHomeと同じ通知ベルとプロフィールを置く。システムNavigationBarの戻る・追加ボタンは重ねず、ヘッダー下には現在ジャンルのメインカラーを反映した全画面幅1ptの区切り線を置く。

観劇ジャンルトップは舞台の章として専用外観を使う。背景は黒へ沈むワインレッド、カード本体は8pt角丸と細いゴールド枠、文字はアイボリーとゴールドを基準にする。左上の`Favoreco`は金〜銅のグラデーションにする。ポスター/アイキャッチだけは角丸を付けず、一般的な縦長ポスターに相当するB判比率`1:1.414`の固定枠へ中央基準のaspect fillで表示する。カード本体やボタンの角丸は残す。ライブジャンルトップは見出しとジャンル列を`LIVE`表記にし、紺から黒紺へ沈む背景、ティールのアクセント、青白い本文色を使う。観劇と同じ構造のミニ統計、`Ticket Progress`、`Coming Up`を使い、参加・記録済みの公演一覧は`Live History`とする。LIVEの`Interests` / `Live History`は横長バナーだけを使い、表示切替は置かない。

ジャンル切替はHomeと共通の1段横スクロールを画面冒頭へ置き、現在ジャンルを色つきアイコンと下線で明示する。従来のHero内プルダウンは使わない。御朱印以外のHeroは`Coming Up / Interests`として、未来Planを開始日時の近い順、その後へ興味あり対象を更新日の新しい順に並べ、合計最大10件を横スワイプ表示する。同じ対象に未来Planがある場合はPlan側だけを表示する。ミニ統計は全ジャンルで観劇と同じ構成を使い、1枚の3列横長パネル、アイコンなし、隣接項目の細い縦線、各列を上段`12pt名称`、中央`33pt数値＋単位`、下段`9pt補足`で組む。名称と数値の間は1pt、数値と補足の間は2pt、上下余白は10ptを基準とする。文言はジャンル別とし、観劇は`作品・公演 / 観劇済み / 気になる`、ライブは`ライブ / 参加済み / 気になる`、映画は`映画 / 鑑賞済み / 観たい`、本は`本 / 読了 / 読みたい`などとする。観劇は上側をわずかに明るく下側を黒に近づけたワイン系グラデーション、淡い金光、0.7pt金枠、アイボリー文字を維持する。ライブは紺〜黒紺の弱い縦グラデーション、ティールの光・枠・区切り、青白い文字を使う。その他は白系の弱い縦グラデーション、ジャンル色の光・枠・区切り・単位色を使う。内部実装用の有効ユニット数は見せない。観劇のセクション見出しは`Ticket Progress / Productions`、ライブは`Ticket Progress / Coming Up / Live History`を22pt基準の英字ディスプレイ明朝で表示する。その他カード外周は0.75ptを基準とする。背景は白ベースを保ちながら、現在ジャンル色を上部へ薄く反映する。

総合と観劇の`Coming Up`行は共通部品を使い、左から`24ptの月/日と12ptの日本語曜日を縦2段`、`64pt正方形アイキャッチ`、`ジャンル・タイトル・会場を2pt間隔で縦3段`、右端シェブロンの順にする。画像なしはジャンル色のアイコンで代替する。タイトルは2行、会場は1行までとし、総合は白系カード、観劇はワイン系カードと金枠を使う。

ジャンルトップの左右スワイプはHeroを除き、統計から章末までの本文で前後ジャンルへ切り替える。Heroは自身のカルーセル操作を優先し、ジャンル切替を開始しない。公演選択チップ、チケット情報チップなど内側に横スクロールを持つ場所でも内側操作を優先する。映画を含む全ジャンルで同じ共通ジェスチャー部品を使い、認識器はジャンル内容の`.id`と遷移Viewより外側に1個だけ維持する。これにより切替アニメーション中も新旧ジャンルの認識器を重複させず、前後方向を同じ条件で判定する。判定用マーカーは本文と同じ実サイズへ広げる。左右24ptの画面端はシステムの戻る操作を守る。横スクロールチップは自身の表示枠でクリップし、カードや画面の外へ描画しない。

美術展・LIVE・おでかけ施設は対象一覧と段階展開ボタンの後、章末カードの前へ`行った美術館・会場MAP / 行ったライブ会場MAP / 行った施設MAP`を表示する。同じ場所の記録は1地点へまとめ、訪問回数が多いほどジャンル色のマーカーを濃く大きくする。座標はVisitを優先し、未設定ならPlaceMasterを使う。上部は`全国 / 北海道 / 東北 / 関東 / 中部 / 近畿 / 中国 / 四国 / 九州・沖縄`のチェック式単一選択を横スクロールし、選択地域へカメラを移動する。`詳細検索`は施設・会場・住所・記録内容を対象とする自由ワードと都道府県を持ち、適用すると該当地点へ再フォーカスする。位置未設定はMAP対象外として件数を下部へ示す。MAP内のパン・ズーム、エリア列の横操作はジャンル切替ジェスチャーを開始しない。

対象一覧の見出し右へ、3列ギャラリー／2列ミニ詳細／横長バナーの切替アイコンを置く。ただしLIVEは横長バナー固定としてアイコンを置かない。選択はジャンル単位で保存する。ギャラリーは画像と情報帯を同じ背景・枠線のタイルへ一体化し、全ジャンルで`7/7 (火)`形式の日付を左、ジャンル色の短い縦線、星＋評点を右へ1行表示する。未記録は`—`、土曜は青、日曜と平日の未来予定日は赤字。2列ミニ詳細は列間を12ptとし、左右の枠線を離す。タイトルはゴシック体へ変更し、従来より2pt小さく、字間`-0.35`・行間`-1.5`で2行分を常時確保する。通常カードは106pt、映画カードは108pt固定高とし、題名が1行でも2行目を空白として高さを揃える。2列の日付は先頭へカレンダーアイコンを置き、`7/16 (木)`の半角括弧にし、従来より1pt小さくする。評価済みは0.5刻みに丸めた約7.8ptの5つ星と9.35ptの正確な数値を1行に並べ、未評価は`☆ —`とする。映画ではB判ポスターをカードの縦いっぱいへ表示する。バナー画像は保存済み比率枠へ中央aspect fillし、状態・2行タイトル・日付・会場を右へ表示する。映画・書籍・おでかけ施設の共通ファーストビュー型Heroは、アイキャッチを内側幅45%・最大152.5pt（従来比1.25倍）とする。作品タイトルは18.5ptを基準に固定の空行を予約せず2行まで、右側情報の行間は4ptを基準とし、周辺情報を圧迫しない。

表示切替のStateと保存値はテンプレートキー単位に分離し、美術展を含む御朱印/LIVE以外のジャンルで同じ操作にする。LIVEは保存済みの旧選択値があっても横長バナーを優先する。2列ミニ詳細は日付と評価の間に補足行を置き、観劇は会場、映画は映画館名、おでかけ施設は予定／記録の場所住所から抽出した都道府県を表示する。未設定は`—`で行を保ち、従来の固定高は変えない。

御朱印の専用Heroは最近いただいた御朱印を新しい順に最大5件表示する。現在の和紙カードサイズを変えず、前後カードを見切れさせない全幅ページ式とし、2件以上の時だけ下へ丸インジケータを表示する。統計は`参拝先 / ご記帳済み / 気になる`の3列とする。参拝先は御朱印Visitが紐づくEvent IDの重複除外数で、同じ寺社への複数参拝と未参拝の「気になる」寺社を含めない`総参拝先数`、ご記帳済みはVisit単位で御朱印をいただくたび増える`総御朱印数`とし、単位はそれぞれ`寺社 / 印 / 寺社`とする。

一覧の初期表示は3列ギャラリー9件、2列ミニ詳細8件、横長バナー6件とする。残件があれば区切り線付き・14pt基準・44pt押下高の`さらにn件`を表示し、タップごとに同数を追加する。全件表示後は`閉じる`へ切り替え、初期件数へ戻す。各一覧自身が表示件数を保持し、展開状態はジャンル・セクション・表示方式ごとに分ける。観劇・美術展・LIVE・映画では`stateKey == interested`を`Interests`として通常一覧より前へ独立表示し、その直後へ`Coming Up`を置く。未来Planが紐づく興味あり対象は`Coming Up`だけへ出し、通常一覧を含めて重複させない。`Interests`は0件でも見出しと空状態を表示する。本は通常一覧前へ`気になる！ → 積読`を置き、興味ありを前者、興味ありではなくVisitがない本を後者、Visitがある本を通常一覧へ分類する。両方とも0件時に見出しと空状態を表示する。横長バナーは正方形トリミングをやめ、82pt幅で登録済みアイキャッチ比率を維持する。情報欄は9pt・18pt高の状態チップ、2行分を常時確保した15ptタイトル、日付、会場の縦順とする。状態は有効なチケット工程を最優先し、`チケット発売待ち / 当落待ち / 入金待ち / 発券待ち / 発券済み`などを表示する。工程がなければジャンルに応じた予定・体験済み・気になるを表示する。LIVEは工程がない場合を`参加予定 / 参加済み`とし、シリーズ/ツアー名、出演者、主催も入力済みの範囲で表示してテキスト情報を優先する。

書籍は登録・編集の写真欄と対象アイキャッチ編集で本の種類を選択する。コミック`112:174`、新書`103:182`、ハードカバー`127:188`、文庫`105:148`、雑誌`182:257`、写真集・画集`210:297`の実寸比率をそのままアイキャッチ枠へ反映し、書影は中央aspect fillする。種類は作品と記録へ保存し、Home、ジャンルトップ、一覧、対象/記録詳細、入力サムネイルで共通利用する。新規既定はハードカバー、既存未選択データは従来の`1:1.45`を維持する。

進行中のチケット申込があるジャンルトップでは、対象一覧の前にチケット進捗を置く。複数公演・複数申込は上部の可変幅チップだけを横スクロールし、選択中の詳細カード1枚へ切り替える。日時、会場、申込経路、プレイガイド、任意タグも可変幅チップで表示する。カード内は公演名16pt、情報チップ21pt高、工程丸34pt、工程名・日付9pt、内側余白9ptを基準にコンパクトに組む。工程は抽選の`申込・当落・入金・発券・観劇`と先着/購入の`購入・入金・発券・観劇`を基準に、不要・未登録工程を除いた2〜5個へ可変とする。工程名は丸内、日付は丸下、接続線は丸の外周間だけに置き、完了をジャンル色の実線、未来をグレーの点線、現在をジャンル色の二重丸で表す。チップ外のカード本体ではジャンル横スワイプを維持する。

御朱印以外のジャンルは共通ファーストビューを使う。これから参加予定のPlanと興味あり対象を横スナップカルーセルで並べ、予定を先、興味ありを後として合計最大10件にする。カードと複数件カルーセルは総合Homeと同じ248pt高とし、観劇・映画・美術展のB判アイキャッチ、他ジャンルの情報・操作を共通枠内へ収める。その下の集計も3列連結ミニ統計へ統一し、アイコン型4列パネルは使わない。続く対象ライブラリでは、観劇・美術展・LIVE・映画に`Interests → Coming Up`の独立一覧を残す。御朱印Heroは専用寸法を維持する。

横長バナーの日付行は`日付 時刻`とする。上段右は観劇で`主催: 名称`、映画で`監督: 氏名`を入力済みの場合だけ表示する。観劇の主催は予定の主催名、作品の主催名、人物・団体の主催役割の順で補完する。映画の監督は人物・団体の監督役割から表示する。未入力なら上段右の領域は表示しない。

## 6. 対象詳細

`ExperienceEvent`の閲覧専用画面。

置く情報:

- 対象名、ジャンル、シリーズ、サブ種別
- 代表写真
- 基本となる人物・団体
- 公式URL、対象メモ
- 関連する予定
- 体験履歴
- 記録回数、最新日、平均評価

置かない情報:

- 個別の体験日、座席、当日の金額、当日の写真
- 当日の天気、感想、同行者

これらは記録詳細へ置く。対象編集は別画面へ遷移する。

観劇だけは作品・公演詳細を複数回の観劇を束ねるハブとして扱う。Step 1〜2の表示順は`Hero / 公演サマリー / 次の予定 / Ticket Progress / あらすじ・公演情報 / キャスト・スタッフ / 参加履歴 / 思い出ギャラリー`とする。公演サマリーは参加回数、平均評価、初回、最新を表示する。次の予定は未来Planを日時昇順で並べ、初期表示は直近1件、超過分は同じ画面内で展開し、行から該当Plan詳細へ進む。予定0件では空状態から同じ対象へ予定を追加できる。Ticket Progressは対象に紐づく非表示でない全Plan・全TicketAttemptを対象とし、進行中だけでなく落選・参加済み・見送りも残す。未来公演を昇順、過去公演を降順にし、公演日・会場・状態のチップで選択して既存の工程カードを表示し、カードから該当Plan詳細へ進む。参加履歴は新しい順に並べ、各行は日付・会場・座席・感想などを持つ該当Visitの記録詳細へ進む。思い出ギャラリーは各Visitの`思い出`分類写真だけを横断表示し、写真タップも元のVisit詳細へ進む。履歴は最初の5件、写真は最初の6枚を表示し、超過分は同じ画面内で展開する。0件のセクションは空状態を表示し、写真0件のギャラリーは置かない。金額総計と内訳は二重加算防止ルール確定後のStep 3とする。

対象編集では対象名などの基本情報に加えて、対象アイキャッチを追加・変更・解除できる。解除するのは`ExperienceEvent.eyecatchData`だけとし、各記録に保存した写真は削除しない。対象詳細とジャンルトップで使う代表画像の優先順位は、利用者が明示指定した記録写真、対象アイキャッチ、記録写真からの自動候補の順とする。

## 7. 記録詳細

`Visit`の閲覧専用画面。保存情報を原則すべて表示する。

表示順:

1. 背景写真、対象名、アイキャッチ、日付、上演時間、観劇スタイル、会場、座席、評価
2. メモ・感想
3. 背景とアイキャッチに使っていない写真ギャラリー
4. チケット／グッズ写真と各金額・OCR
5. 観劇のキャスト
6. 費用合計とチケット申込／遠征ToDo導線
7. 場所、住所、Map
8. 当日の人物・団体、スタッフ・関係者
9. OCR・取込内容
10. 天気
11. 公式・参考リンク
12. 詳細オプション

上部は全ジャンル共通の画像主役型Heroとする。標準ナビゲーションバーは完全に隠し、写真上へ独自の戻るボタンと右上メニューを重ねる。右上メニューはジャンル正規色を面、観劇ではゴールドをアイコンと枠の差し色に使い、独立したヘッダー帯を作らない。写真は上部の高さを制限した帯へ中央基準のaspect fillで配置する。縦長写真は天地を均等に切り、画像比率でHero高を押し広げない。白へのフェードは使わず、元画像の色を残したまま下端だけを該当ジャンルの正規色へ溶かす。ページ全体はジャンル色から深色へ続く暗色基調とし、観劇はワインレッド`#8B2F45`を面、ゴールドを操作と装飾の差し色に使う。明るい写真でもステータス、上部操作、Hero文字が沈まないよう、局所コントラストスクリーンと文字影を使う。対象名は帯の下端と重なる位置に置き、その下へ左アイキャッチ、右へ日付、上演時間、観劇スタイル、会場、座席、評価を置く。日付と時間は別行とし、終演が入力済みなら`12:00–15:00`形式で表示する。対象名は対象詳細へのリンクにする。背景またはアイキャッチとして使った写真は直下の写真一覧で重複させない。写真がない場合はジャンル色グラデーションとジャンルアイコンをフォールバックに使う。

観劇スタイルは`Visit.unitFieldsRaw`内の`styleNames`へ複数値で保存する。旧記録は空配列として互換読込し、SwiftDataスキーマは増やさない。追加／編集フォームでは主要候補を複数選択でき、候補にない表記もカンマ区切りで入力できる。対象そのものの分類に使う`ExperienceEvent.subTypeKey`や、全ジャンル共通のタグとは分け、今回の観劇回の表示情報として扱う。

編集とこの記録だけの削除は、全ジャンル共通で右上の`…`へまとめる。削除は確認ダイアログを経由する。

観劇のキャストはメモ・感想と写真の後へ横スクロール表示する。出演、主演、アーティスト、演奏、ゲスト、および表示役割に出演/主演/キャストを含む人物が対象。人物写真があれば円形に切り抜き、なければ活動タグ由来アイコンをジャンル色の円内へ表示する。演出、脚本、原作、音楽など出演系以外は`スタッフ・関係者`として通常の役割つき一覧へ残す。キャスト0件時はキャスト欄自体を表示しない。

写真は`思い出 / チケット / グッズ`の3分類とする。空値・未知値を含む既存写真は思い出として扱い、利用者が編集して明示分類できる。思い出写真だけをHero背景・アイキャッチ・通常ギャラリーの候補とし、チケット／グッズ写真はそれぞれ独立セクションへ画像、画像別金額、OCR本文を表示する。

登録・編集では各写真サムネイルから分類と付随情報を編集する。チケット／グッズでは同じ写真へVision OCRを実行し、OCR本文を画像に紐づけたまま保存する。本文から見つけた金額は候補として提示し、利用者が選んだ場合だけ画像別金額へ反映する。入力中の新規写真は`PendingPhoto.metadata`、既存写真は`PhotoMetadataDraft`に保持し、記録の保存確定前にSwiftDataモデルを書き換えない。思い出へ戻した写真はOCR本文・金額を保存時に空へ戻す。

費用合計は、確保済み以降の`TicketAttempt`にある券面＋手数料、グッズ写真の確認済み金額、遠征ToDoの確認済み金額を費目別に加算する。チケット写真の金額はTicketAttempt金額が0円の時だけ代替値として使い、両方は足さない。既存の`Visit.amount`は構造化明細がすべて0円の時だけ旧合計として使い、明細がある場合は参考値として保持して加算しない。観劇／ライブの記録詳細には`申込を追加`と`遠征ToDo`を置き、元Planがない過去記録では利用者がこの導線を押した時だけVisitへ紐づくPlanを作る。

遠征ToDoは既存の公演準備リストを拡張し、通常の準備に加えてホテル、新幹線、飛行機、現地交通、その他の遠征を選べる。遠征項目は実際の開始／終了日時、任意期限、費用、OCR原文、完了状態を持つ。画像OCRは原文だけを先に保持し、タイトル、日付、金額、種類は候補を利用者が押した場合だけ反映し、保存前には確定しない。通常準備の金額は費用合計へ入れず、遠征種別だけを遠征費として加算する。

## 8. 一覧・サマリー

一覧カードに出す情報は最大で以下に限定する。

- カバー写真
- タイトル
- ジャンル
- 日付
- 場所
- 評価または現在状態
- メモ1行

人物、金額、OCR、URL、天気は通常一覧に常設しない。期限や未整理など、次の操作に必要な状態だけ補助表示する。

## 9. 思い出の登録

下部タブバーはSwiftUI標準`TabView`を使い、`Home / 記録 / 追加 / カレンダー / 統計`の5項目をシステム標準の寸法で並べる。独自のカプセル、背景色、影、下端余白は追加せず、`.toolbarBackground(.ultraThinMaterial, for: .tabBar)`で透過ブラーを明示する。iOS 26では本文上へ浮く透過ブラーの標準タブバー、iOS 18では従来型の半透明タブバーとして表示する。各タブのルートは下端container Safe Areaを越えて描画し、本文とポスターをバーの背後まで続ける。スクロール末尾にはバー高相当の操作余白を持たせ、最後の要素をバーに隠さない。`追加`は独立画面へ切り替えず、標準タブ項目の選択をBindingで横取りして現在のタブを維持したまま次の4入口を固定順で表示する。

1. `予定を立てる`: これから体験するPlanを登録
2. `体験済みを記録`: 予定を経由しない正式なVisitを直接登録
3. `クイック登録`: 通常と同じExperienceEventを、ジャンル/タイトル必須、アイキャッチ/URL/読み取りメモ/短いメモ任意の簡易フォームで`気になる`として保存
4. `チケットスケジュールを追加`: 対象イベントを検索または最小作成し、そのPlanへ抽選/発売スケジュールを紐付ける

初期状態では`予定を立てる`を既定の追加方法とする。利用者が設定で既定値を変更できるようにしても、入口の並び順自体は動かさない。観劇、ライブ、美術展、おでかけなどは予定起点を基本とし、御朱印、お酒、過去記録の移行などは`体験済みを記録`から直接登録できる。

`予定を立てる`と`体験済みを記録`は対象情報と基本入力部品を共用するが、初期状態を分ける。

| 入口 | 初期表示する情報 | 初期状態で閉じる情報 | 主な表示先 |
|---|---|---|---|
| 予定を立てる | 対象、日時、会場、通知、チケット | 写真、感想、評価 | 次の予定、カレンダー、要対応 |
| 体験済みを記録 | 対象、体験日、写真、感想、評価 | 通知、抽選、発売 | 最近の思い出、記録一覧、統計 |

`予定を立てる`は最初に予定の対象を確定する。主操作は`新しく登録`とし、同じフォーム内でExperienceEventとPlanをまとめて保存する。補助操作の`気になるから選ぶ`では、`気になる`状態の既存ExperienceEventを検索して選び、その対象へ新しいPlanを追加する。検索対象はタイトル、シリーズ、ジャンルとし、文字幅・大文字小文字・空白の差を無視する。選択後は対象名、ジャンル、公式URLを予定フォームへ引き継ぐ。対象詳細から開いた場合は選択を省略し、表示中のExperienceEventへ直接追加する。

予定日時はアプリ全体で日本語の年月日表示とし、時刻入力は5分刻みに統一する。観劇とライブだけは`開場 → 開始 → 終了`の順に表示し、開場を変更した時は開始を30分後、終了を開始の2時間後へ一旦連動する。開始だけを変更した時も終了を2時間後へ一旦連動し、その後は開始・終了を個別修正できる。映画、美術展、おでかけ、書籍、酒、御朱印などには開場を表示・保存しない。観劇/ライブのPlanを外部カレンダーへ追加または片方向更新する時は開場を開始時刻とし、終了はPlanの終了時刻を使う。

Planは必ずExperienceEventへ紐付け、Planだけを単独保存しない。予定日時を持たない`気になる`登録はExperienceEventだけで成立するが、利用者自身の候補日または確定日を入力した時点でPlanを作る。開催日・開催期間など対象そのものの日付と、利用者が行く予定日時は区別する。Plan作成後は対象の`気になる`状態を通常状態へ移す。

予定日を迎えたPlanから`記録を残す`を実行した場合は、対象、日時、会場、チケット参照を引き継いでVisit作成を開始する。元のPlanとTicketAttemptは削除せず完了状態として保持し、同じExperienceEventへVisitを紐付ける。

`チケットスケジュールを追加`は`新規イベント / 登録済みイベント`の2入口とし、チケットだけを独立保存しない。新規イベントではExperienceEvent、Plan、TicketAttemptを同じ保存処理内で作成する。登録済みイベントではタイトル、シリーズ、ジャンルからExperienceEventを検索し、そのイベントに未完了Planがあれば紐付け先を選んでTicketAttemptだけを追加する。文字幅・大文字小文字・空白の差は検索時に無視する。

登録済みイベントにPlanがない場合、または同じイベントに別の予定を作る場合は、選択したExperienceEventへPlanとTicketAttemptをまとめて追加する。既存Planを選んだ後は予定名、日時、会場を参照表示とし、チケット保存によってPlanの日時・会場・公式URLを上書きしない。通常は予定詳細から追加し、下部の`追加`タブはイベント詳細まで移動せず登録するための短縮導線として扱う。

`クイック登録`はPlanやVisitを作らず、通常と同じExperienceEventを`気になる`状態で作る入力項目限定版とする。別のInboxItemへ保存して後から変換する工程は設けない。URLとOCRは独立した登録方式ではなく、同じ画面内でタイトルと画像を埋める入力補助とする。URLから取得したタイトルは候補として表示する。カメラ/写真のOCRは先頭行を使わず、認識行の文字領域の高さ・面積・認識確度を基に、大きく目立つ文字をタイトル候補として順位付けする。1位が十分に目立ち、同じ領域の文字認識にも曖昧さがない場合だけ候補表示し、利用者が`タイトルに使う`を選んだ時だけ反映する。確定できない場合は自動候補を出さず、目立つ領域のVision上位候補とOCR全行を提示して利用者に選ばせる。OCR全文は選択・コピー可能な確認欄へ残し、対象の`読み取りメモ`へ保存する。OCRに使った画像は対象のアイキャッチ候補にする。後から同じ対象へPlanまたはVisitを追加し、その時点で状態を通常へ移す。

クイック登録後も対象詳細の`対象情報・画像を編集`からアイキャッチを差し替え・解除できる。画像は読み込み時に長辺1600px以内、JPEG品質85%へ圧縮し、元画像のメタデータを保持しない。

登録フロー:

1. `体験済みを記録`を選択
2. コンパクトなジャンルメニューでジャンルを選択。全ジャンル一覧を常時展開しない
3. 主操作の「新しい作品・対象を登録」、または補助操作の「登録済み作品・対象に記録を追加」を選択
4. 既存対象の場合は検索結果から対象を選び、対象側情報を参照して今回固有のVisitだけを入力
5. 新規対象の場合は対象名だけで作成可能とし、必要に応じてシリーズを検索・連結
6. 記録の必須情報を入力
7. 必要な補助ユニットだけ展開
8. 保存後に記録詳細を開く

下部の`追加`タブの`体験済みを記録`では、ジャンルをコンパクトなメニューで選んだあと、主操作の`新しい作品・対象を登録`と、補助操作の`登録済み作品・対象に記録を追加`を同じ選択画面に表示する。既存対象は選択ジャンル内をタイトルまたはシリーズ名で検索でき、未入力時は最近更新した対象を少数だけ表示する。

登録済み対象の候補部分は画面全体とは別の縦スクロール領域にし、候補が増えても下部説明や閉じる操作へ到達できる高さに制限する。右端にはスクロールインジケータを表示する。

予定詳細の`参加記録を入力`はVisitを即時作成せず、同じ記録入力フォームを開く。対象、予定日時、終了日時、会場、座席、金額、メモ、公式URL/購入URLを初期値として引き継ぎ、利用者が確認・修正して保存した時だけVisitを作成する。保存成功と同じ処理内でPlanを参加済みにし、代表のTicketAttemptも参加済みに更新する。入力をキャンセルした場合はPlanとTicketAttemptを変更しない。

入口の情報優先順位:

```text
ジャンル [観劇 ▼]

[ ＋ 新しい作品・対象を登録 ]

または

登録済み作品・対象に記録を追加
[ タイトルを検索 ]
```

「既存対象への記録追加」と「シリーズ連結」は別機能とする。既存対象への追加は再鑑賞、再訪、再読、同じ銘柄の再記録に使う。シリーズ連結は新しい対象を別作品として作成したうえで任意の親シリーズへ紐付ける。

既存対象を選んだ場合に参照する対象側情報:

- タイトル、ジャンル、シリーズ、代表写真
- 基本の人物・団体、公式URL、対象種別

Visitへ入力する今回固有の情報:

- 体験日、会場、当日の人物・団体
- 写真、評価、座席、金額、感想、天気

対象情報をVisitへ無条件コピーせず、対象はリレーション参照、当日固有情報はVisitのスナップショットとして保存する。

常時表示:

- ジャンル
- 対象名
- 日付
- 場所
- 評価または状態
- メモ

アコーディオン:

- 人物・団体
- 写真
- チケット・座席
- OCR・取込
- 金額
- 公式情報
- 詳細オプション

新規対象を作る場合でも、対象側の詳細情報をすべて同時入力させない。まず名称だけで対象を作成でき、対象詳細から後で補完できる。

## 11. 御朱印専用登録フロー

御朱印は一般的な対象選択より先に、登録先の御朱印帳を選ぶ。

```text
御朱印を登録
↓
使用中の御朱印帳を選択
↓
新しい寺社 / 登録済み寺社を選択
↓
御朱印を入力
↓
保存して完了 / 保存して同じ寺社で続けて登録
```

御朱印帳選択画面:

- 使用中の御朱印帳を表紙画像、名称、登録済み御朱印件数で表示
- 同じ利用者が寺社や用途ごとに複数冊を並行利用できる
- `新しい御朱印帳を登録`を補助操作として同じ画面へ置く
- 閉じた御朱印帳は新規追加候補に出さないが、過去帳一覧から閲覧・編集できる

御朱印帳の管理:

- ページ数、総ページ容量、使用ページ数を管理しない
- 裏面利用、見開き、書置きなどの差を自動判定しない
- 満了の自動判定、自動クローズを行わない
- 利用者の明示操作だけで御朱印帳を閉じる
- 閉じた御朱印帳は新規御朱印を追加できないが、既存御朱印と帳面情報は編集できる
- 御朱印帳には登録済み御朱印の件数を表示する

寺社選択:

```text
[ ＋ 新しい寺社の御朱印を登録 ]

または

登録済み寺社から選ぶ
[ 寺社名を検索 ]
```

御朱印登録項目:

- 御朱印画像/アイキャッチ: 任意
- 寺社名: 必須
- よみ: PlaceMasterの読みを寺社名上部へ薄いルビとして表示
- 住所、ミニMap、公式URL、御祭神: 任意。登録済み寺社から引継可能
- 参拝・拝受日時、写真、同行者、メモ: 任意
- 御朱印名/種類、通常/限定、直書き/書置き: 任意
- 初穂料・納経料: 任意。`300円 / 500円 / その他`をクイック選択し、その他は任意金額を入力

保存操作:

- `保存して完了`
- `保存して同じ寺社で続けて登録`

続けて登録では御朱印帳、寺社、日時、同行者を保持する。御朱印画像、御朱印名/種類、金額、写真、メモは次の記録へ引き継がない。

御朱印帳の基本情報:

- 表紙画像
- 御朱印帳名、または購入した寺社
- `神社用 / お寺用 / 共用`
- 使い始めた日
- メモ

御朱印帳は長押しから編集、閉じる/再開、削除へ進める。削除は帳面内の御朱印と写真を含むため、対象件数表示、バックアップ案内、`削除する`の確認入力を含む3段階確認にする。削除しても寺社マスターと別の御朱印帳は削除しない。

## 12. 共通入力ルール

### OCR・URL取込

OCRとURL取込は保存操作ではなく入力補助とする。

1. 画像、URL、貼り付けテキストを読み取る
2. タイトル、日付、会場、人物、金額、ISBNなどの反映先を候補表示する
3. 利用者が候補ごとに反映先変更、編集、除外を行う
4. `反映する`を押した項目だけフォームへ入れる

- 信頼度が低い候補は要確認表示にする
- 既存入力を自動上書きしない
- OCR全文を確認画面へ残し、選択、コピー、貼り直しを可能にする
- 公演系は登録画面上部に`公式サイトから情報を取得`と`チラシ・チケットをOCRで読み取る`を常設する
- 本はISBNバーコードを優先し、奥付OCRを予備経路にする。ISBNは任意の内部識別情報として保持し、通常入力で手入力を要求しない

### 人物・団体の重複候補

人物名の検索用文字列は、Unicode正規化後に半角/全角スペース、連続空白、改行、タブを除去し、英字の大文字/小文字を無視する。

```text
山田 太郎
山田　太郎
山田太郎
```

上記は同一候補として提示する。表示名の原文は保持し、自動統合せず、既存人物を使うか別人物として作るかを利用者が選ぶ。この規則は著者、俳優、アーティスト、監督、作家などPersonMasterを使う全ジャンルへ適用する。

### SNSタグ

全ジャンルの対象と記録に、複数登録可能な任意のSNSタグを持たせる。

- `#`なしの入力は表示時に補完する
- 全角/半角スペースを無視して重複候補を出す
- URL/OCRから公式タグ候補を提案する
- タグ群をまとめてコピーできる
- 通常は`公式情報`アコーディオン内へ置く

### 日付

期間は終了日を必須にしない。開始日だけで保存でき、終了日がない対象を無期限の`開催中`とは自動判定しない。

### 写真と写真に紐づく情報

写真を用途別に二重保存しない。共通写真へ任意の作品情報または体験情報を付け、詳細画面で意味のあるまとまりとして再構成する。

- 美術展: 作品名、作家名、メモ
- おでかけ: 名前、種類、メモ、任意の金額
- 写真がない場合はテキストだけの作品/体験メモを追加できる
- 情報付き写真も通常の写真ギャラリーへ含める

### 評価

映画と本は、記録回ごとに`1.0...5.0`を0.5刻みで評価できる。ジャンルごとに機能ON/OFF可能、初期値はOFF、未評価保存を許可する。OFFにしても保存済み評価は削除しない。一覧への評価表示は別設定とする。

## 13. ジャンル別登録フロー

### 映画

```text
映画を記録
├ 新しい映画を登録
└ 登録済み映画に鑑賞記録を追加 [検索]
```

映画情報:

- タイトル必須
- 読み、原題、ポスター、公開年、監督、出演者、制作会社、公式URLは任意
- 1作品は複数のコレクションへ所属できる
- コレクションは代表画像、作品数、作品一覧、公開順/時系列/任意順を持つ

鑑賞記録:

- 鑑賞予定/鑑賞済み、鑑賞日、映画館、上映形式、同行者、写真/半券/OCR、金額、感想/メモ、任意評価
- 再鑑賞は映画情報を参照し、新しい鑑賞記録だけを追加する

映画トップはコレクションカードを横スクロール表示し、`一覧を表示`からコレクション一覧へ進む。主要都市のシネコンとミニシアターをPlaceMasterの初期候補へ収録する。

### 観劇

通常利用は`公演 -> 観劇回`の2層とし、作品/演目の理解を登録時に要求しない。再演をまとめる機能は内部拡張余地を残し、必要な利用者だけ後から使えるようにする。

公演情報:

- 公演名必須
- メイン画像/チラシ、公演種別、劇団/主催、公式URLは任意
- 人物・団体は閉じたアコーディオンに置き、原作/原案、作/脚本、演出、出演者、音楽、振付などを役割付きで複数登録できる
- 公演名と主催が近い既存公演がある場合は、情報を引き継ぐか新規作成するか確認する。自動統合しない
- チラシ/ポスターは公演、当日の写真/半券は観劇回へ保存する

観劇回:

- 観劇日時、会場、座席自由入力、当日の出演者差分、写真/半券、金額、同行者、感想/メモ
- 開場時刻は予定/チケット側、上演時間は公演側の任意項目、休憩は標準項目にしない
- 感情タグ初期候補は`また観たい / 泣いた / 笑った / 驚いた / 圧倒された / 考えさせられた / 音楽が好き / 演出が好き / 衣装・美術が好き`

公演種別初期候補:

- 演劇、ミュージカル、歌舞伎、落語・寄席、ダンス・舞踊、バレエ、オペラ、コンサート、アイスショー、マジック・サーカス、その他
- カスタム種別は名前、アイコン、並び順、非表示を管理できる
- 使用済み種別は削除せず、別種別への統合または非表示にする

一覧サマリーはチラシ、公演名、観劇日、劇場、予定/観劇済みに限定する。

### ライブ

通常利用は`ライブ・イベント -> 参加回`の2層とし、入口はアーティストではなくライブ名から始める。

```text
ライブを記録
├ 新しいライブを登録
└ 登録済みライブに参加記録を追加 [検索]
```

ライブ情報:

- ライブ名必須
- 種別は`ワンマン / 対バン / フェスティバル / ライブイベント / 配信ライブ / その他`
- アーティストは最初から表示して複数登録可能。役割は`メイン / 出演 / ゲスト / オープニングアクト`を任意設定する
- ツアーは種別にせず、必要な場合だけ関連ライブとして後からまとめる

参加回:

- 参加日時、会場、写真/アイキャッチ、感想/メモを基本とする
- 座席/整理番号、当日のゲスト、セットリスト、同行者、金額、公式情報は閉じた任意項目
- セットリストは曲名と順番を持ち、まとめて貼り付け可能。アンコールは区切りで表現する
- フェスは出演者全員と実際に観たアーティストを分け、詳細ON時だけアーティスト別セットリスト、ステージ、時間帯を記録する
- 配信ライブは会場の代わりに配信サービスと視聴URLを使う
- 感情タグ初期候補は`また行きたい / 泣いた / 笑った / 驚いた / 圧倒された / 盛り上がった / 音が良かった / 演出が好き / セットリストが好き`

ライブトップは今後のライブ、アーティスト、参加したライブ、データがある場合だけツアー/関連ライブを表示する。

### 本

通常利用は`本 -> 読書記録`の2層とする。

```text
本を記録
├ 新しい本を登録
└ 登録済みの本に読書記録を追加 [タイトル・著者検索]
```

本の登録はISBNバーコード、表紙/奥付OCR、タイトル入力から開始できる。本の情報は表紙、タイトル必須、著者、巻数、出版社、発売日、ISBN、公式/参考URLを持つ。

- 1冊は複数のシリーズ/コレクションへ所属できる
- ISBNが異なる単行本、文庫、電子版を同じシリーズへ入れられる
- 状態は`気になる / 積読 / 読書中 / 読了 / 中断`
- クイック登録は気になる、開始日あり/読了日なしは読書中、読了日ありは読了を自動候補にする
- 積読と中断は利用者が選択し、自動判定後も手動修正できる
- 読書記録は開始日、読了日、`紙 / 電子 / オーディオ`、感想/メモ、写真、任意評価を持つ
- 再読は同じ本へ新しい読書記録を追加する
- 保管場所、貸出状況、読書タイマー、読書速度、毎日のページ進捗は扱わない

本の種別初期候補は`小説 / エッセイ / ノンフィクション / ビジネス・経済 / 実用・趣味 / 人文・思想 / 科学・技術 / 漫画 / 絵本 / 詩歌 / 雑誌 / その他`とし、カスタム管理可能にする。印象タグ初期候補は`また読みたい / 心に残った / 泣いた / 笑った / 驚いた / 考えさせられた / 学びがあった / 読みやすかった`とする。引用は閉じた任意項目で、本文、ページ、メモを複数登録できる。

本トップは読書中、気になる/積読、最近読んだ本、シリーズ/コレクションを表示する。

### 美術展・展示会

通常利用は`展覧会 -> 鑑賞記録`の2層とする。同じ巡回展は展覧会を共通化し、会場と鑑賞日は鑑賞記録へ持たせる。会場ごとに内容やタイトルが大きく違う場合だけ別展覧会として関連付ける。

展覧会情報:

- 展覧会名必須
- メイン画像/チラシ、開催開始日、任意の終了日、作家/アーティスト、主催/企画、公式URL、SNSタグ
- 排他的な単一種別を使わず、`現代美術 / 絵画 / 彫刻 / 写真 / 映像 / インスタレーション / メディアアート / 建築 / デザイン / 工芸 / ファッション / 漫画・アニメ / 歴史・文化 / 科学・自然`などを複数選択できるジャンル・テーマとして扱う

鑑賞記録:

- 鑑賞日、会場、写真、同行者、金額、感想/メモ、印象タグ
- 写真へ作品名、作家名、メモを付けると`印象に残った作品`として自動表示する
- 写真がない作品はテキストだけで記録できる
- 印象タグ初期候補は`また行きたい / 心に残った / 驚いた / 圧倒された / 考えさせられた / 展示空間が好き / 作品構成が好き`

展覧会トップは開催中/これから行きたい展覧会、チケットスケジュール最大2行、最近観た展覧会、アーティスト、美術館/ギャラリー、写真ギャラリーを表示する。

### おでかけ施設

```text
おでかけを記録
↓
テーマパーク / 動物園 / 水族館 / 植物園 / その他
↓
お気に入り、最近使った施設、全施設から選択 / 新しい施設を登録
↓
訪問記録を入力
```

施設情報は施設名必須、種別、メイン画像、読み、住所/Map、公式URL、SNSタグ、お気に入りを持つ。訪問記録は訪問日、写真/アイキャッチ、同行者、チケット/予約、金額、天気、体験、感想/メモ、SNSタグを持つ。

体験情報:

- 共通写真へ体験名、種類、メモ、任意金額を付ける。写真なしでも登録可能
- テーマパークの初期候補は`アトラクション / ショー・パレード / グリーティング / イベント / フード・ドリンク / グッズ・購入品 / その他`
- 動物園、水族館、植物園では表示候補を動物、生き物、植物、季節展示、餌やり/体験などへ変える
- 体験種類はカスタム追加、並び替え、非表示、統合可能

おでかけトップはこれからのおでかけ、チケットスケジュール最大2行、お気に入り施設、最近行った施設、写真ギャラリーを表示する。

### お酒

```text
お酒を記録
├ 新しいお酒を登録
└ 登録済みのお酒に飲酒記録を追加 [検索]
```

- 初期種別は日本酒、ワイン、ビール、ウイスキー、焼酎、その他

- 飲酒記録は日付、場所、写真、飲み方、金額、同行者、感想/メモを持つ

## 14. チケット共通フロー

チケット登録の入口は次の3択とする。

```text
チケットを登録
├ 抽選に申し込む
├ 発売・予約を待つ
└ 取得済みチケットを登録
```

抽選:

- 申込開始、申込締切、当落発表、入金期限、発券開始、発券/受取期限を、分かる項目だけ登録する
- 申込URLからプレイガイドを自動候補にし、URLがない場合だけプレイガイドの一般入口を利用できる

発売・予約:

- 発売/予約開始を基本とし、販売終了、発券開始、発券/受取期限は任意
- 先行販売、一般発売、日時指定予約、当日券は販売区分として扱う

取得済み:

- 購入メールや案内を崩さず貼れるチケットメモを中心にする
- メモ内URLをタップ可能にし、対応アプリがあればアプリ、なければブラウザで開く
- プレイガイド、座席、番号、枚数等の個別フォームを標準表示しない
- 画像/PDF、発券・表示確認、確認リマインダーは任意
- `チケットQR・画像`は通常写真と分け、年間まとめ、写真ギャラリー、共有画像へ出さない
- QR全画面表示中は明るさと自動ロックを入場確認向けに調整する
- 動的QRやスクリーンショット無効の場合があるため、公式サイト/アプリ表示を優先する注意を出す

開催日時と会場は公演/展覧会/予定側を参照し、チケットへ重複保存しない。ジャンルトップのチケットスケジュールは共通チケットデータから直近1〜2件を表示し、登録がなければセクションごと隠す。

## 15. 場所・種別・タグのマスターUI

会場入力は利用者へ施設/ホールの階層選択を要求せず、1つの検索/自由入力欄にする。

```text
会場 [東京芸術劇場 コンサートホール]
```

- 施設名だけでも、ホール名込みでも保存可能
- 住所がある場合は名称検索ではなく住所を優先してMap表示する
- 主要都市圏の映画館、劇場/ホール、美術館/博物館/ギャラリー、おでかけ施設を初期候補として収録する
- 内部で親施設を共有しても、通常登録では細かな施設階層を見せない

カスタム種別と印象タグは、名前、アイコン、色、並び順、表示/非表示を管理できる。未使用のカスタム項目は削除可能、使用済み項目は削除せず別項目へ統合または非表示にする。初期項目は削除せず非表示のみ可能とする。

## 16. 実装移行方針

1. 既存モデルは維持し、まずViewの責務を分離する。
2. `EditExperienceView`が同時更新しているExperienceEvent/Visitを、対象編集と記録編集へ分割する。
3. `AddExperienceView`の先頭に既存対象検索を追加し、新規対象作成と既存対象へのVisit追加を同じ入口から選べるようにする。
4. `CategoryTopView`は対象ライブラリを主役に再構成する。
5. `ExperienceDetailView`と`EventDetailView`を閲覧専用として整理する。
6. View分割後にデザインラフを各画面へ適用する。

## 17. 完成条件

- どの情報がExperienceEvent、Visit、Plan、TicketAttemptのどれに属するか画面上で判断できる
- 一覧から編集コントロールが排除されている
- 対象編集で過去Visitの固有情報が変わらない
- 記録編集で対象名や対象公式情報が意図せず変わらない
- Plan編集とVisit編集が混在しない
- 登録は必須情報だけで保存でき、詳細入力は任意展開できる

## 18. 設定の情報設計

設定トップは機能を直接並べず、次の5分類から各設定へ移動するハブとする。各行には設定できる内容を短いマイクロテキストで表示する。

```text
設定
├ マイ・登録情報
│  ├ プロフィール
│  └ 登録情報・連携
├ アプリ設定
│  ├ 表示・外観
│  ├ ジャンル
│  ├ 記録・入力補助
│  └ 通知
├ データと同期
│  ├ データ管理
│  └ 同期・バックアップ
├ プラン
└ サポート
```

「アプリ設定」のマイクロテキストは次を正本とする。

- 表示・外観: Home表示、外観、文字、フォント、テーマ
- ジャンル: 表示順、表示・非表示、自作ジャンル、有効ユニット
- 記録・入力補助: 初期値、写真圧縮、URL・OCR・Map・天気の補助
- 通知: チケット期限、公演前日・当日、会員期限、レポート

初回ジャンル選択のやり直しは「アプリ設定 > ジャンル」の末尾に置き、既存記録を削除しないことを明記する。テスト権利、仮データ、通知診断、全データ削除テストはDEBUGビルドだけの「開発者メニュー」へ分離し、通常利用者へ表示しない。

「記録・入力補助」には現在動作する設定だけを置く。記録日は今日を初期値として入力画面で変更できるため、単一選択肢の設定は置かない。Apple Musicなど未実装の将来候補も通常設定へ予告表示しない。

「表示・外観」のHome表示には、現在内容を表示できるセクションだけを置く。写真付き記録の横スクロールは「最近の思い出」と呼ぶ。お気に入り/ベストは集計元が実装されるまでHomeにも設定にも予告枠を出さない。
