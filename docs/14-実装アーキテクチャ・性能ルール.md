# 14. 実装アーキテクチャ・性能ルール

> 作成日: 2026-07-08
> ステータス: 正本
> 位置づけ: Mystoriumで発生した重いリファクタ・性能問題をfavorecoで再発させないための実装ルール。`favoreco/CLAUDE.md` §5から参照する。

---

## 0. 最重要4原則

favorecoの実装では、以下を最初から守る。

1. **入力中にDBを書かない**
2. **一覧で原寸画像を使わない**
3. **SwiftUI `body` で全件処理しない**
4. **巨大Viewを作らない**

Mystoriumではこの4点を後から直したため、リファクタ・性能改善・副作用確認のコストが大きかった。favorecoでは設計ルールとして固定する。

---

## 1. View構造

### 1.1 画面は最初から分割する

大きくなりそうな画面は、初日から以下に分ける。

- `ContainerView`: fetch / state / navigation の境界
- `SectionView`: Homeやジャンルトップの各ブロック
- `CardView`: 一覧カード・棚・行程カード
- `RowView`: リスト行
- `SheetView`: 目的別入力・確認・クイック操作

例:

```text
HomeView
  HomeContainerView
    NextExperienceSection
    AttentionSection
    QuickInboxSection
    RecentMemorySection
```

```text
GenreTopView
  GenreTopContainer
    HeroRecordSection
    PrimaryShelfSection
    AttentionOrProgressSection
    StatsStripSection
```

### 1.2 便利な1画面を作りすぎない

詳細・編集・共有・写真・評価・地図・チケット管理を1画面に密集させない。長いFormの奥へ自動スクロールするより、目的別のSheetに分離する。

---

## 2. SwiftData更新

### 2.1 入力中はDraftStateへ逃がす

登録/編集画面では、SwiftDataモデルを入力ごとに直接更新しない。

必須フロー:

```text
Model -> DraftState -> Save -> Model
```

対象:

- `EventDraft`
- `PlanDraft` / `PerformanceDraft`
- `TicketAttemptDraft`
- `VisitDraft`
- `MemoryDraft`
- `InboxItemDraft`

保存ボタン、明示的な完了操作、または自動保存タイミングでだけSwiftDataへ反映する。

### 2.2 クイック登録は本体モデルを汚さない

クイック登録は `InboxItem` に隔離する。ジャンル・タイトル・画像・URL・メモの一部だけでも保存できるが、未整理データを直接 `Event` / `Visit` / `TicketAttempt` に混ぜない。

昇格フロー:

```text
InboxItem -> Event
InboxItem -> Event + Plan
InboxItem -> Event + Plan + TicketAttempt
InboxItem -> Event + Visit
```

---

## 3. モデル責務分離

### 3.1 ライフサイクル状態を混ぜない

| モデル | 責務 |
|---|---|
| `InboxItem` | クイック登録の一時受け皿。ジャンル未確定でもよい |
| `Event` | 対象。映画作品、公演、展示、酒銘柄、本、施設、寺社など |
| `Plan` / `Performance` | 未来の予定・公演回・訪問予定 |
| `TicketAttempt` | 申込1件。FC先行、一般、リセールなど |
| `Visit` | 実際に体験した1回 |
| `MemoryDraft` / `VisitDraft` | 体験後に予定・写真・天気から作る記録下書き |

### 3.2 TicketAttemptをVisitに直持ちしない

複数先行・落選履歴・名義別当選率・通知更新に耐えるため、チケット申込は `TicketAttempt` として独立させる。

例:

```text
Event: 緋色のカーテンコール
  Performance: 2026-07-12 18:30 赤羽アートシアター
    TicketAttempt: FC先行 申込済み -> 当落待ち
    TicketAttempt: 一般販売 未申込
  Visit: 実際に観た記録
```

`Visit` は体験後の記録であり、申込状態の正本ではない。

### 3.3 見送り・落選・非表示を混同しない

| 状態 | 使う場所 | 意味 |
|---|---|---|
| `dismissed` | Inbox / Interested | 気になったが不要になった |
| `skipped` | Plan / TicketAttempt | 申込しなかった、行かなかった |
| `lost` | TicketAttempt | 抽選に落選した |
| `archived` | 全モデルの表示制御 | 一覧から隠す。データは消さない |

記録フォームに常に「見送り」を出さない。見送りは予定・候補・申込側の状態。

## 4. 表示用DTO / Snapshot

### 4.1 Viewに永続化モデルを直接なめさせない

Home、ジャンルトップ、Calendar、StatsではSwiftDataモデルを直接走査して集計しない。

推奨フロー:

```text
SwiftData Model
  -> Repository / QueryService
  -> Lightweight Snapshot / DTO
  -> ViewModel / Metrics
  -> SwiftUI View
```

### 4.2 bodyで重い処理をしない

`body` 内に以下を書かない。

- 全件 `filter`
- 全件 `sort`
- 全件 `map`
- JSON decode
- DB fetch
- 画像 decode
- 検索index構築

1描画内で必要な集計は `let metrics = ...` のように一度だけ評価し、Sectionへ渡す。画面横断の集計はSnapshot層で作る。

## 5. 画像・写真

### 5.1 thumbnail / detail / original の3段階

画像は用途ごとに扱う。

| 用途 | 方針 |
|---|---|
| `thumbnail` | Home / Library / Calendar / Collection用。表示サイズに合わせて縮小 |
| `detail` | 詳細画面用。中解像度 |
| `original` | 編集・書き出し・明示表示時のみ |

一覧で原寸画像を直接デコードしない。

### 5.2 DBにはメタデータ、重いバイナリは別管理

写真/動画本体をSwiftDataに安易に入れない。`PhotoBlob` / externalStorage / file reference / Photos reference / cacheを分離する。画面側はrelativePathまたは参照IDを扱い、重いDataを直接持たない。

### 5.3 画像処理はMainActor禁止

写真I/O、サムネ生成、圧縮、OCR前処理、import/exportはMainActorで実行しない。Swift 6でstaticユーティリティが意図せずMainActor扱いにならないよう、必要に応じて `nonisolated` を明示する。

---

## 6. 検索・サジェスト・Smart Add

### 6.1 debounce / cacheを必須にする

検索欄・サジェスト欄では、キー入力ごとに全データを再計算しない。

対象:

- 作品検索
- 人物検索
- 場所検索
- 酒銘柄検索
- 書籍検索
- タグ/同行者サジェスト
- URL解析候補
- OCR候補統合

原則:

- 入力debounceを入れる
- 表示中だけのcacheを持つ
- 入力変更時に古いTaskをキャンセルする
- 大きい検索indexは画面表示時または遅延で構築する

### 6.2 Smart Addは中間モデルを通す

URL、画像、OCR、カレンダー、写真からの登録は、直接モデルへ保存せず解析結果の中間モデルを通す。

```text
RawInput -> ParsedCandidate -> UserConfirmation -> Draft -> Model
```

---

## 7. import / export / migration

### 7.1 background + batch save

import、export、migration、bulk updateでは1件ごとに `context.save()` しない。バッチ単位で保存し、進捗表示を出す。

### 7.2 初期表示へ重いwarmupを乗せない

Calendar、検索index、統計、写真cacheのwarmupは起動直後に集中させない。初期表示に不要な処理は遅延構築する。

---

## 8. Apple Kit依存の境界

iOSネイティブ実装は優先するが、ドメインモデルと状態遷移をApple APIへ直結させない。

抽象化する境界:

- `CalendarProvider`（EventKit）
- `PhotoProvider`（Photos / PhotosPicker）
- `OCRProvider`（Vision）
- `WeatherProvider`（WeatherKit）
- `NotificationScheduler`（UserNotifications）
- `PurchaseProvider`（StoreKit）
- `MapProvider`（MapKit）
- `SyncProvider`（CloudKit）

将来Android対応を完全に捨てないため、UI/OS連携以外は移植可能な形を保つ。

## 9. Performance Budget

実装時の目標値:

| 操作 | 目標 |
|---|---|
| Home初期表示 | 300ms以内 |
| ジャンル切替 | 100ms以内 |
| 検索入力反応 | 50ms以内 |
| 画像一覧スクロール | 60fps目標 |
| クイック登録保存 | 体感即時 |

早期に Instruments（Time Profiler / Allocations / SwiftUI Body Evaluation）で確認する。
