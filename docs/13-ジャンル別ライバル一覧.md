# 13. ジャンル別ライバルアプリ・サービス一覧

> **役割**: テンプレ第1弾（実装予定8ジャンル）ごとに、直接・間接のライバル（記録アプリ／情報サービス／代替手段）を書き出した参照リスト。戦略の総論は `docs/01-競合分析.md`、本書は**ジャンル別の実装参入時に「誰と並ぶか」を一目で見る台帳**。
> **調査**: 2026-07-21。観劇・推し活の重点5アプリをApp Storeで再確認。酒・御朱印・おでかけ施設は2026-07-07調査を継承。数値・料金・提供状況は変動するため実装判断時に再確認する。
> **凡例**: 🟥 強大な壁（DB・ソーシャル資産）／🟨 直接競合あり／🟩 手薄（参入余地大）

FAVO画面の改善判断に使う18アプリの横断比較、口コミパターン、現行実装監査、方向3案は`docs/18-FAVO競合深化リサーチ.md`を参照する。

## 0. 全ジャンル共通の「間接ライバル」

どのジャンルでも最初に比較されるのはこれら。専用アプリ以前の「みんなの初期状態」。

| 相手 | 性格 | favorecoが超える点 |
|---|---|---|
| **キログ**（個人開発・買い切り） | 汎用の記録カスタマイズ。統計・カレンダー。最も近い直接競合 | 機能表ではなく「写真主役・世界観・ジャンル専用の器」で差別化 |
| Notion / Google Keep / メモ | 自作テンプレで観劇・展覧会を記録する層が多い | 入力の速さ・美しさ・自動取得（OGP/TMDb/NDL）で置き換え |
| Googleマップ 保存リスト＋写真 | 「行きたい/行った」の無料代替 | 体験のライフサイクル（YOUR JOURNEY）と記録の器 |
| Instagram / X（半券・チケ写真の投稿先） | 実質「みんなのアーカイブ」になっている | 私的・非ソーシャル・流れて消えない保管 |

## 0-A. 重点比較：観劇記録3アプリ＋推し活2アプリ

ユーザー提示の5アプリを、単なる機能数ではなく「何を記録の主語にしているか」で比較する。

| アプリ | 記録の主語 | 強い体験 | Favorecoが参考にする点 | 正面から追わない点 |
|---|---|---|---|---|
| **Stagefolio** | 観た舞台・座った席 | 作品名・日付・劇場の3項目から始める軽い入力、座席メモ、劇場ノート、推し別/月間/年間レポートを上品な誌面風に見せる | 入力開始の軽さ、劇場ごとの記憶、年間ハイライトの編集的な見せ方 | 公式公演DB取得は行わないと明記されており、Favorecoもデータ網羅競争には入らない |
| **StageDiary** | 1回の観劇体験 | 劇場・座席・費用・写真・感想、対応劇場の座席マップ、チケット/キャストボードOCR、ウィジェット、共有カード、FC更新期限、公開記録 | 回ごとに違う座席・出演者を残す粒度、次の予定と期限の即時性、写真付き共有カード | 公開フィード、全劇場の座席マップ保有、FC管理アプリ化は中核にしない |
| **TheaterRecords** | 構造化された観劇データ | 複数チケット一括スキャン、座席ヒートマップ、購入元×席種、回別キャスト、費用・劇場・作品ランキング、年間ハイライト | チケット入力短縮、回別キャスト、会場/席/費用を後から発見へ変える集計 | 観劇専用の詳細分析を全ジャンルへ一度に持ち込まず、段階導入する |
| **Pics** | 日ごとの推し活日記 | 写真中心の日記、カレンダー、月ごとのカバー、前後の日記閲覧、アカウント登録なしのバックアップ | 記録を開いた瞬間に写真と感情が戻ること、日記として続けやすい軽さ | 日付だけを主語にして推し・作品・場所の関係が埋もれる構造、広告中心の体験 |
| **推して何日？ byGMO** | 推し本人 | 最大100件の推し別ページ、推し始めた日、記念日カウント、色・写真・好きなもの、ストーリー、支出、カスタマイズ | 推しプロフィール、最推し、開始日と数え方、記念日、推しカラー、次に会える日 | 動画広告、ジェム交換、演出を目的にしたゲーム化、推し別に記録を分断する構造 |

### 市場から見える二つの強さ

1. 観劇専用アプリは、`チケット → 劇場 → 座席 → キャスト → 費用 → 集計`の構造化が強い。
2. 推し活アプリは、`推し本人 → 写真 → 推し始めた日 → 記念日 → 感情`の愛着形成が強い。

両者はそれぞれ強いが、観劇、LIVE、映画、書籍、美術展、聖地巡礼等に分かれた体験を、同じ人物・団体へ戻して一続きに見せる部分は弱い。Favorecoはここを主戦場にする。

### Favorecoへ反映する優先順位

#### P0：推しを横断軸として成立させる

- `PersonMaster`へ利用者固有の`FavoriteProfile`を重ねる
- 最推し、ピン留め、写真、推しカラー、呼び名、推し始めた日、日数の数え方、複数記念日を保存する
- 観劇・LIVE等の`Event / Plan / Visit`を推しへリンクし、記録自体はジャンル側に一度だけ保存する
- 記録タブを`推し / 人物・団体 / 場所 / タグ・同行者 / すべての記録を検索`へ再編する
- 推し詳細の先頭へ`推して○日 / 次に会える予定 / チケットの次の期限 / 最近の思い出`を出す

#### P1：記録を発見へ変える

- 推し別の横断タイムライン、ジャンル別回数、年別回数、初回・最新、よく行く会場、支出を集計する
- 公演回ごとの出演者差分を保持し、同じ作品を複数回観ても上書きしない
- 座席は当面、階・列・番号・席種・見え方メモ・写真を保存し、全施設の座席図や座標データは持たない
- OCRはチケット・チラシから入力候補を作る補助として使い、確定は利用者が行う

#### P2：愛着と共有

- 年間ハイライト、推し別レポート、思い出カード、ウィジェット
- 会場別の座席傾向や購入元別分析は、十分な記録件数がある利用者向けの高度統計として後段へ置く

### Favorecoの差別化文

> 推しの記録を、観劇だけ、ライブだけ、日記だけに分けない。会いに行った日、作品を観た日、読んだ本、訪れた場所、写真、チケット、使った金額を、ジャンルを越えて同じ推しの物語へ戻せる。

## 1. 観劇 🟩 手薄

| アプリ／サービス | モデル | 性格・強み | 弱点 |
|---|---|---|---|
| **シアティ Theaty**（カンフェティ系） | 無料（チケット送客・チェックインP） | 観劇記録＋推し活。最有力の専用アプリ | 「チェックイン不具合」「写真登録できない」等の品質不満レビュー |
| CoRich舞台芸術 | 無料（口コミ＆予約ポータル） | 小劇場の口コミ・予約 | UIが古い・私的記録用途は薄い |
| TheaterRecords / 観劇記録系（個人開発） | 無料＋アプリ内購入 | OCR、座席ヒートマップ、回別キャスト、費用・年間分析まで観劇特化 | 観劇外の体験を同じ推しへ横断統合する用途には向かない |
| Stagечиな自作Notion勢 | 無料 | 「使いやすい観劇アプリない→Notion」が知恵袋の定番回答 | 手間・美しさ不足＝潜在需要の証拠 |

**favorecoの立ち位置**: 座席番号・マチソワ・キャスト変更・複数回観劇という特有ニーズ＋チケット抽選/支払い期限のライフサイクル（YOUR JOURNEY）を美しく。開発者の既存ユーザー層（Mystorium／観劇クラスタ）との重なりが最大。**最有力の主戦場**。

## 6. おでかけ施設（テーマパーク・水族館・動物園）🟩 記録用途は空白

| アプリ／サービス | 性格 | 記録アプリか |
|---|---|---|
| 東京ディズニーリゾート・アプリ／USJ公式アプリ | チケット・待ち時間・モバイルオーダー | ✕（運営アプリ・訪問記録ではない） |
| インパ記録の自作（Notion/手帳/Instagram） | 「何回インパした」「アトラクション制覇」を自作 | ✕（専用アプリ不在＝空白） |
| **Biome（バイオーム）** | いきものAI図鑑（約1万種・無料無広告・写真で記録・図鑑埋め） | △（生物の記録に隣接。動物園・水族館で機能が重なる） |

**favorecoの立ち位置**: 06リサーチで発掘した空白。テーマパークの**インパ回数・アトラクション・年パス**、水族館/動物園の**見た生き物・企画展**を1テンプレ＋施設種別サブレイヤーで。Biomeは「野生の生物同定」が主で「施設での体験の記録」ではない＝正面衝突しない。※AI生き物ライフリスト構想は体験ファースト外としてfavorecoには入れない（姉妹アプリ案・project-log）。

## 9. チケット管理（横断機能・spec-A8が正本）

> **正本は `docs/spec-A8-チケット管理・通知.md`**（状態pipeline・通知・FCアカウント・OCR）。ここは競合の俯瞰のみ。

### 9-A. チケット管理アプリ＝直接競合（spec-A8のリサーチ準拠）
状態管理・抽選・当選率・名義でfavorecoと正面から競合する層。

| アプリ | 強み | 弱点／favorecoの超えどころ |
|---|---|---|
| **LiveSoul** | 実質の王者。抽選管理＝当選率追跡・締切リマインダー・座席（一部3D）・遠征マップ（★4.5・1273件） | 単一ジャンル専門。favorecoは横断＋登録の楽さで抜く |
| **TheaterRecords** | チケットOCR自動認識・分析 | 観劇専門。OCRはfavorecoもA7/A8で実装（横断で持つ） |
| **Cherie Log** | 複数名義・掛け持ち管理 | 専門。favorecoはFCアカウントを登録情報ハブに一元化（A8 §3.4） |
| Live Rock / relight / Tickemo | セトリOCR・シェアカード・カウントダウン（ビジュアル派） | 「全手動入力が面倒」の不満。A7のOCR/URL/参照DBで潰す |
| ~~チケコレ~~ | （旧設計の基準だった） | **更新停止・レビュー無し＝基準にしない**（2026-07-07） |

### 9-B. プレイガイド＝販売元・状態語彙の参考（競合ではない）
発券・販売の土俵。favorecoは**発券・購入はしない**が、状態遷移の語彙・電子券の多様性を参考に。

- **チケットぴあ / イープラス(e+) / ローチケ / 楽天チケット / カンフェティ**＝発売日・抽選・当落・入金期限という状態語彙の源
- **MOALA Ticket / tixeebox / スマチケ（松竹）等の電子チケット**＝年パス・回数券・分配など券の多様性。**発券アプリが公演ごとに分散する現実＝「締切・座席・当落を横断で1か所に集める母艦」の空白**がfavorecoの余地

**favorecoの立ち位置**: 専門アプリは全て単一ジャンル。①**横断**（観劇/ライブ/映画を1アプリ・統一統計）②**登録の楽さ**（競合最大の共通不満をA7のOCR/URL/参照DBで解消）で抜く。通知・状態pipeline・FCアカウントはv1で徹底作り込み（spec-A8）。

## 10. スケジュール管理（横断機能の参考サービス）

favorecoカレンダー（A: 体験特化カレンダー ＋ B: EventKitで外部予定を薄く重ね表示）の参考元。**汎用予定作成では戦わない**（concept「やらないこと」）が、体験の予定・会期・発売期間の見せ方で参考にする。

| サービス／アプリ | 性格 | 参考にする点 | 関係 |
|---|---|---|---|
| **Googleカレンダー / Appleカレンダー** | 標準カレンダー | 週/日の時間軸グリッド（feasibility §4-3でGoogle式を仕様の参照に） | **連携先**（EventKitで読み取り重ね表示）。競合ではなく土台 |
| **TimeTree** | 共有・コミュニケーション型カレンダー（月300円プレミアム） | 予定に写真・メモを添付する文化、**推し活の"公開カレンダー"ウィジェット**（イベント予定を通知） | 共有・招待は狙わない。favorecoは私的・体験特化。締切通知の思想は参考 |
| Jorte（ジョルテ） | 手帳風・高カスタマイズ・日記/タスク内蔵 | 手帳的な世界観・記録と予定の同居 | デザインの参考（favorecoは写真主役でより静か） |
| Yahoo!カレンダー / Lifebear | 多機能カレンダー | イベント連携・リマインド | 直接の参考は薄い |
| Oshibana（再掲・7章） | 推し活の予定・お金・グッズ管理 | 「推しの予定を落とさない」ニーズ | ジャンル横断で一部重なる |

**favorecoの立ち位置**: 標準カレンダーは**倒す相手ではなく重ねる土台**（B機能）。体験の予定・会期バー・発売期間の描き分け（feasibility §4-3で確定済みの6ルール）が独自価値。汎用予定作成はGoogle/Appleの土俵として明確に「やらないこと」。

## まとめ：ジャンル別の攻めやすさ

| ジャンル | 状況 | favorecoの勝ち筋 |
|---|---|---|
| 観劇 | 🟩 手薄 | **最有力**。特有ニーズ×既存ユーザー層×ライフサイクル |
| 美術展 | 🟩 手薄 | 決定版不在。会期バー・愛でる体験 |
| 御朱印 | 🟨 DB王者はデザイン手薄 | 美しさ＋参照DB課金の柱 |
| おでかけ施設 | 🟩 記録用途は空白 | インパ・生き物の体験記録 |
| ライブ参戦 | 🟨 激戦 | 横断統合＋遠征アーカイブ |
| 酒 | 🟨 種別分立 | 全酒種×種別サブレイヤー×酒蔵横断 |
| 映画 | 🟥 強大な壁 | 統計全部入り×劇場体験。横断の1ジャンル |
| 書籍 | 🟥 参入余地薄 | 受け皿に徹する。私的記録のみ |

**共通の攻めどころ（01より）**: ①統計の課金壁 → 買い切り全部入り ②ソーシャル疲れ → 私的記録 ③サブスク疲れ → 買い切り ④データ人質 → エクスポート自由 ⑤ニッチ専用アプリの品質不足 → デザインで上回る。
