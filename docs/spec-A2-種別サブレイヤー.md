# ②-A / spec-A2 種別サブレイヤー（2階層マスター）

> ステータス: 清書（2026-07-06 再作成）
> 位置づけ: ②-A の第2分冊。カテゴリの下に「種別（subType）」をもう1段挟み、種別ごとに評価軸・スペック・OCR辞書のプリセットを出し分ける2階層マスターを定める正本。
> 関連: A1=データモデル基盤 ／ 08 §4-6★★ 酒種別サブレイヤー ／ favoreco-feasibility §6 PoC必須項目 ／ 04 §3・§6 Mystorium enum変更クラッシュの教訓
>
> **ユーザー強調（2026-07-05）**: 「これめちゃくちゃ大事なんだけど、ちゃんと下地作らないと後で後悔するよ」。本レイヤーは**v1必須の下地**（中身は日本酒のみでも、仕組みは最初から通す）。

---

## 0. これが解く問題

酒テンプレは1カテゴリだが、**種別（日本酒/ビール/ウィスキー/ワイン/焼酎/その他）ごとに味覚軸・スペック項目・ラベルOCRの辞書が違う**。Mystorium のマスターは〈カテゴリ→軸/成果/形態〉の**1階層**しかない。これを後から2階層に拡張すると、

- 既存の日本酒記録に種別を埋め直す**データ移行**が発生
- Mystorium で痛い目を見た「**enum の rawValue 変更 → 既存データのデコード不能クラッシュ**」「後から重いリファクタ」（04 §3・§6）の二の舞

になる。だから**種別フィールドと、種別キーでプリセットを引く仕組みを最初から通す**。

---

## 1. 2階層マスターの構造

```
Mystorium（1階層）:   RecordCategory ──▶ { AxisDefinition, OutcomeOption, FormatMaster }
favoreco（2階層）:    RecordCategory ──▶ SubType ──▶ { AxisDefinition, SpecPreset, OcrDictionary }
```

- **RecordCategory**（A1）: 「酒」「観劇」などのカテゴリ。
- **SubType**（新規・本書）: カテゴリの下の種別。「日本酒」「ビール」…。
- プリセット群（軸/スペック/OCR辞書）は **(category, subType) の組**で引く。

**重要な一般化**: 1階層のカテゴリは「**subType = nil の特例**」として同じ仕組みに乗せる。観劇のように種別を持たないカテゴリは、subType を使わず nil のまま運用する。**別の分岐コードを書かない**——1階層も2階層も同じ解決関数を通す。

---

## 2. SubType モデル

| プロパティ | 型 | 既定値 | 備考 |
|---|---|---|---|
| id | UUID | 生成 | upsertキー（unique属性は使わない・A1 §4） |
| category | RecordCategory? | nil | 親カテゴリ（optional・CloudKit互換） |
| key | String | "" | 種別キー（"sake_junmai" 等の不変ID。**表示名と分離**して rawValue 変更クラッシュを避ける） |
| displayName | String | "" | 表示名（"日本酒"）。変えても key は不変 |
| sortOrder | Int | 0 | 並び順 |
| isBuiltIn | Bool | false | プリセット由来フラグ |

- **key と displayName を分離**するのが Mystorium の教訓への直接の回答。enum の rawValue を直接キーにしていたから、表示名や順序を変えるとデコードが壊れた。favoreco は **不変の key（英数字ID）** をキーにし、表示名は別プロパティで自由に変える。
- Event 側は `Event.subTypeKey: String = ""`（A1 §2 で Event の同一性属性）。参照ではなく**キー文字列**を持つ（マスターを消しても記録が壊れない・スナップショット同様の耐性）。

---

## 3. プリセットの引き方（解決ルール）★PoC必須

種別に紐づくプリセット（軸/スペック/OCR）を引くとき、**カテゴリ一致 AND（種別一致 OR 種別なし）**で解決する:

```
該当プリセット = { p | p.categoryKey == C かつ (p.subTypeKey == S または p.subTypeKey == "") }
```

- `p.subTypeKey == ""`（種別なし）= **そのカテゴリ共通のプリセット**（全種別に効く土台）。
- `p.subTypeKey == S` = **その種別専用**の上乗せ/差し替え。
- 観劇など1階層カテゴリは S 自体が空なので、共通プリセットだけがヒットする（＝1階層特例が自動で成立）。

### 3-1. #Predicate を使わず Swift 側でフィルタする ★実装制約
SwiftData の `#Predicate` は **optional + OR の組み合わせでクラッシュ/誤動作の実績**がある（04 §6 の罠系）。よって:

1. まず **categoryKey だけ**で `FetchDescriptor`（`#Predicate` は等値1本のみ・安全）で候補を取る。
2. `subTypeKey == S || subTypeKey == ""` の **OR 判定は取得後の Swift 配列フィルタ**で行う。

> プリセットは件数が小さい（カテゴリあたり数十）ので、Swift フィルタのコストは無視できる。安全側に倒す。

## 4. 種別で切り替わる3つのプリセット

種別キーで出し分けるのは以下3つ。**スキーマは一切変えない**——切り替わるのは全て「どのプリセットを出すか」だけ（08 §4-6 要件4）。

| # | プリセット | 対応ユニット | 日本酒の中身（v1実装） |
|---|---|---|---|
| ① | 味覚軸定義 | U17（2軸）/ U7（6軸） | U17=甘辛度×濃淡度（辛口⇔甘口／淡麗⇔芳醇）。U7=さけのわ6軸（オプション） |
| ② | スペック（数値列＋自由ペア） | 専用列 / U9 | **専用数値列**: 精米歩合／日本酒度／酸度／アルコール分（検索・グラフ用・A1 §8-1）。**U9自由ペア**: 蔵元・銘柄名・使用米・酵母 等（表示専用・A1 §8-2） |
| ③ | OCR拾い辞書 | U16 | 裏ラベル定型（精米歩合／日本酒度／酸度／アルコール分）の語彙＋振り分け先指定 |

- **①味覚軸**: U17 の2軸ラベルは種別ごとに差し替え可能（ビールなら苦味×コク等・v1.x）。軸ラベルは `AxisDefinition` を subTypeKey 付きで持つ。
- **②スペック（2026-07-06 二本立てに確定）**: 種別プリセットは**「どの項目を専用数値列に入れ、どれを自由ペアに入れるか」の振り分け表**を持つ。日本酒は定番4数値（精米歩合/日本酒度/酸度/アルコール分）を**Event側の専用数値列**（クエリ可能）へ、それ以外（使用米・酵母・蔵元コメント等）を **U9自由ペア（JSON）**へ。→ 「精米歩合でソート」「日本酒度の分布グラフ」が成立する。他種別（ビール=IBU/ABV等）も同じ枠組みで、種別ごとに「昇格させる数値列」を定義する。
- **③OCR辞書**: U16 が Vision OCR の認識結果から拾う語彙リスト（半自動・feasibility §4-1b）。種別で「何を探すか」に加え、**読み取り値を専用数値列と自由ペアのどちらへ流すか**もこの辞書が指定する。

## 5. v1 の範囲

- **下地（仕組み）はフル実装**: SubType モデル＋ `Event.subTypeKey` ＋ 2階層マスター＋種別キーでの解決関数（§3）。
- **中身は日本酒プリセットのみ**（味覚2軸・スペック・OCR辞書）。
- **ビール/ウィスキー/ワイン/焼酎は「スキーマ変更なしに」後から追加**（v1.x/v2）＝プリセットのデータを足すだけ。
- **種別未選択でも壊れない**: `subTypeKey == ""` の共通プリセットにフォールバック（§3）。既定＝未分類 or 日本酒（酒カテゴリ新規時の初期値・A4で確定）。

## 6. PoC 検証項目（favoreco-feasibility §6 に対応）

実装の最初の PoC で、下地が「Mystorium 1階層マスターの素直な一般化」で組めるか検証する:

1. `SubType` ＋ `Event.subTypeKey` ＋ プリセット群が **CloudKit互換3条件**（A1 §4）を保ったまま成立するか。
2. **§3 の解決関数**（categoryKey で fetch → Swift で OR フィルタ）が、`#Predicate` の optional+OR 罠を踏まずに動くか。
3. 種別を後から足したとき（日本酒→ビール追加）、**既存の日本酒記録が無傷**か（スキーマ・デコード・移行なし）。
4. `key`/`displayName` 分離で、種別の表示名変更が既存記録を壊さないか。

> ここが崩れると酒の拡張（ビール/ウィスキー）が後で重いリファクタになる。**ユーザー強調の「下地」なので PoC 必須項目**（08 §4-6★★・feasibility §6）。

## 7. 汎用化の余地（将来）

- 2軸味覚マップ（U17）の軸ラベルは種別で差し替え可能なので、**酒に限らず「2軸で評価したいカスタムカテゴリ」全般に転用**できる（コーヒー=酸味×コク 等）。
- 「産地/所在県を持つカテゴリ」の地図集計（酒蔵所在県マップ・feasibility §4-1c）も、subType 単位でなくカテゴリ単位の汎用マップに育てる余地（A5 集計仕様で扱う）。
