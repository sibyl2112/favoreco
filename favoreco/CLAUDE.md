## 1. コンセプト

**観た・行った・体験したを、美しく一生残す。**（詳細: `docs/favoreco-concept.md`＝確定）

- コア体験: 記録が、美しい思い出になる
- 人間観: 人は雑食、ただし個人ごとに「柱」が2〜3本ある
- プロダクトの重点: **ジャンルを横断して「推し」に関する予定・体験・作品・場所・座席・写真・チケット・費用を一生分まとめて振り返れることへ強くコミットする**。推し活をライブだけの専用ジャンルには閉じず、人物・団体を核に観劇、映画、美術展、イベント等の既存記録を横断する。
- 標準アプリアイコン: Ticket Album水彩滲み版。Fモノグラム、チケットアルバム、ワインレッド、淡い青緑/薄赤の水彩滲みで、好きな体験の記録とチケット管理を表す。AppIconでは外側余白を落とし、角丸四角のアイコン本体を1024pxへ大きく切り出した左右詰め版だけを参照する。過去候補は`favoreco/assets/app-icon`へ保管し、AppIconアセットへ同居させない。Dark/Tinted外観は画像を個別指定せずiOSの自動生成を使う。

## 3. データモデル（正本）

実装開始済み。正本仕様は `docs/spec-A1-データモデル基盤.md`。初期実装は [CoreModels.swift](../favorecoAPP/favorecoAPP/Models/CoreModels.swift)。

- `RecordCategory`: ユーザー定義カテゴリ。テンプレも保存済みカテゴリとして扱う。初回起動時に `CategoryPresetSeeder` が標準カテゴリをfetch-first方式で注入/更新する。初回ジャンル選択後は `isArchived` で表示対象を制御し、少なくとも1件は有効になるよう補正する。ジャンルの表示順は `sortOrder`、テーマカラーは `colorHex`、有効ユニットは `enabledUnitsRaw` で管理する。自作ジャンル用に `templateTypeKey`、`targetNameLabel`、`recordUnitName`、`dateLabel` を持つ。
- `ExperienceEvent`: 対象（作品・公演・銘柄など）。Swift標準/API名との衝突を避けるため、実装名は `Event` ではなく `ExperienceEvent` とする。
- `CollectibleItem` / `CollectibleTransaction`: カプセルトイ、ランダムアクキー、缶バッジ等を扱う汎用収集モデル。`ExperienceEvent`をシリーズ、`CollectibleItem`を種類別ラインナップ、`CollectibleTransaction`を入手・手放し履歴として使う。履歴の数量は常に正数で保存し、種別により入手を加算、交換放出・譲渡・売却・紛失を減算する。現在所持は0未満へ表示せず、同一種類の`現在所持 - 1`をダブり数とする。コンプリート分母は非表示でなく`isCompletionTarget`の種類だけ、関連支出は入手側履歴の金額だけを合計する。両モデルは個人CloudKit、JSON schema 8／完全／自動バックアップ、イベント連鎖削除、全データ削除の対象とする。
- `Visit`: 体験した各回。会場・評価・座席・写真・金額など回ごとの値を持つ。
- `InboxItem`: 気になるもの・あとで記録するものの一時保存。
- `PhotoBlob`: 写真バイナリ。`@Attribute(.externalStorage)`でSQLite本体の外へ保存し、CloudKit同期対象を維持する。存在確認・件数・容量集計・代表写真候補の判定は`byteCount`だけを使い、`data`をフォールトさせない。写真本体はサムネイル/詳細表示/バックアップ書き出しなど必要時だけ読む。
- `PhotoBlob`の記録写真分類は`purpose`の`memory / ticket / goods`を正本とし、空値・未知値は`memory`として読む。チケット／グッズ写真は画像単位の`ocrText`と`amount`を保持し、思い出写真では両値を表示・集計しない。入力中の新規写真は`PendingPhoto.metadata`、既存写真はUUIDキーの`PhotoMetadataDraft`へ保持し、記録保存時だけモデルへ反映する。JSON schema 6と完全バックアップで新規メタデータを往復し、schema 5以前は空OCR・0円で復元する。
- `SocialAccount`: プロフィール用SNSアカウント。Instagram / X / Threads / Facebookを複数登録でき、任意で `RecordCategory` に紐付ける。未紐付けは全体プロフィール扱い。
- `PersonMaster`: アーティスト、俳優、作家、画家、声優、演者、劇団、出版社などを横断で扱う人物・団体マスター。ジャンル別DBに分けず、人物自体の活動分野は複数選択の`roleTagsRaw`、作品・回ごとの役割は`EventPersonLink`へ分離する。活動タグの初期候補は、俳優、舞台俳優・ミュージカル俳優、声優、アイドル、モデル、お笑い芸人、ナレーター、アーティスト、歌手、ミュージシャン、バンド・音楽グループ、DJ、作曲家、作詞家、ダンサー、振付師、作家、脚本家・劇作家、翻訳者、編集者、写真家、撮影監督、監督、演出家、クリエイター、イラストレーター、漫画家、美術家・画家、彫刻家、デザイナー、建築家、キュレーター、プロデューサー、劇団、制作会社・制作団体、芸能事務所、出版社、主催者・主催団体、醸造家・杜氏、酒蔵・醸造所の40種とし、任意の自作タグも保持する。2026年7月の人物スターターとして、アーティスト／最近の俳優8名に加え、舞台俳優の市村正親、大竹しのぶ、井上芳雄、山崎育三郎、生田絵梨花、上白石萌音、演出家・監督の蜷川幸雄、宮本亞門、野田秀樹、三谷幸喜、是枝裕和、李相日、作家の村上春樹、宮部みゆき、東野圭吾、恩田陸、朝井リョウを含む計25名を起動時に一度だけ追加する。投入は候補ごとの導入バージョンを持ち、旧8名を投入済みの端末では新規17名だけを追加するため、利用者が削除した旧候補をバージョン更新で復活させない。全員へひらがなの読みを持たせ、一般に検索で使われる別名・愛称がある人物だけ、ハチ、風くん、バウンディ／バウ、お亮、ほっくん、流星くん、愛ちゃん、ばんちゃん、ミュージカル界のプリンス、いっくん、いくちゃん、萌音ちゃん、ニナガワ、宮本亜門等を登録する。正規化名が既存人物と一致する場合は追加も上書きもせず、Vaundy／バウンディと宮本亞門／宮本亜門の表記違いは同一判定するが、一般的な愛称は誤判定防止のため自動重複判定には使わない。スターターは推し設定や記録リンクを自動作成せず、削除後にも再追加しない。管理一覧上部は推しと使用中の活動タグをチェック式チップでOR絞り込みし、名前検索とはANDで組み合わせる。表示は登録写真を最優先、写真がなければ先頭の既知活動タグに対応するSF Symbol、該当なしは共通人物アイコンを使う。人物写真は`imageData`をexternal storageで保存し、旧`imagePath`も読み取り互換として残す。保存必須は表示名だけ。管理画面では表示名、よみ、別名・愛称、活動タグ、写真を基本情報として編集でき、メモ、公式URL、SNS/参考リンクは任意の詳細オプションで扱う。保存成功時は画面を閉じて触覚フィードバックを返す。一覧は名称・よみ・別名・愛称・活動タグで検索できる。MusicBrainz ID / Wikidata QID / Apple Music IDなどの外部IDは、候補取得・名寄せ用のoptional内部IDとし、ユーザー入力必須にしない。名称・よみ・別名が近い候補を表示し、ユーザー確認で全EventPersonLinkを統合先へ付け替える。同じ作品/回/役割の二重リンクだけをアーカイブし、異なる役割は保持する。編集/統合で過去の`nameSnapshot`は変更せず、元マスターは即削除せずアーカイブする。

- `PlaceMaster`: 会場/場所を横断で扱うマスター。劇場、ライブ会場、映画館、美術館、寺社、酒蔵、飲食店、テーマパーク/施設などを単一カテゴリで分断せず、タグ/種別で表現する。名称と都道府県を必須保存し、都道府県は住所からの都度推定ではなく独立項目として持つ。営業状態は`operationalStatusRaw`へ`未設定 / open / closed`を保存し、既存場所は未設定として互換維持する。管理一覧・編集・FAVO・過去記録の場所候補では閉館を明示し、未来予定の通常候補からは閉館施設を除外する。状態は個人CloudKit同期、場所統合、JSON schema 7／完全／自動バックアップへ接続する。管理画面では名称、よみ、都道府県、住所、別名、タグを基本情報として編集でき、寺社仏閣、ランドマーク、城郭、ダム、ライブ会場、公共ホール・文化会館、スタジアム・アリーナ、テーマパーク、動物園・水族館、美術館・博物館、自然・公園、酒蔵・産業観光の12分類を複数選択できる。公式URLとメモは任意アコーディオン、保存座標は読み取り表示にする。巡礼・札所は`pilgrimageMembershipsRaw`の構造化JSONへ複数所属を保存し、巡礼・霊場名、正規の整数番号、番外等の任意表示を管理画面から編集する。一覧は名称・よみ・住所・別名・分類名で検索でき、全国／8地方／47都道府県／未設定と12分類／未分類を組み合わせて絞り込める。旧データは住所から都道府県を補完し、未設定レコードは専用フィルターから修正でき、都道府県未選択では編集保存できない。同名、同住所、または150m以内の座標を「同じ場所の可能性」として提示し、ユーザー確認でVisit/Planの参照を統合先へ付け替える。元マスターは即削除せずアーカイブする。編集/統合でも記録時の名称/住所/座標スナップショットは変更せず、PlaceMasterや参照DBが変わっても過去記録が壊れないようにする。巡礼所属は場所統合時に重複を除いて併合し、個人CloudKit同期とJSON／完全／自動バックアップの対象にする。初期実装では `Visit.placeMaster` で参照し、複数地点が必要になったら `VisitPlaceLink` へ拡張する。

- 共通場所カタログ（調査・設計中）: 会場だけでなく寺社仏閣、文化施設、レジャー施設、酒蔵、史跡・ランドマーク等を分類タグ付きの同一カタログとしてCloudKit Public Databaseから差分取得し、端末へキャッシュする。各分類で利用・来訪機会が多い場所から件数上限を設けず、公式一次情報を出典付きで順次登録する。調査原稿は2,227場所で、小劇場50件、ライブハウス83件、寺院666件、神社59件を含む。寺社は札所収録数だけで充足とせず、各都道府県・公式観光連盟・自治体の観光案内へ定番スポットとして掲載される場所を重複確認しながら地域別に補完する。寺院の宗派と神社の御祭神は場所本体へ推測で混ぜず、`religious-place-details.csv`の任意1対1詳細として公式確認できた場所から追加する。複数の御祭神は`|`区切りとし、未詳名は注記だけへ残す。全国公立文化施設協会から選定した公共ホール470施設に加え、別バッチで公共劇場タグを持つ施設もある。劇場・ライブハウスは収容人数だけで足切りせず、大・中規模会場に加えて、長期運営、劇場群、登竜門、地域・インディー文化の拠点等の小規模な文化的重要会場を別枠で収録する。ただし地域別件数は均等化せず、公式の公演・イベント日程、貸館実績、または主催者等の公式発表で原則18か月以内の実利用か今後の開催予定を確認できる実在施設だけを現役候補にする。現役候補は公式サイトの存在、会場一覧掲載、知名度だけでは採用せず、長期休止・実会場を持たない団体・直近利用を確認できない候補を除外する。一方、閉館済みでも全国的知名度、文化的・地域的重要性、世代横断の記録需要がある実在施設は`status=closed`の歴史施設枠へ収録し、過去記録では検索可能、未来予定の通常候補からは除外する。利用者向け情報は名称、よみ・別名、種別、都道府県・住所・座標、公式URL、公表済み収容人数程度に限定し、都道府県は全レコード必須とする。巡礼の所属と札所番号はPlaceの別名へだけ埋め込まず、同じ寺院の複数霊場所属を扱える専用関係として保持し、場所管理から巡礼・整数番号・表示番号を入力編集できる設計とする。営業時間、休館日、料金、最寄駅、アクセス経路、開催情報等の変動情報は保存せず公式へ任せる。選択時は既存の`PlaceMaster`へ取り込み、Plan/Visitには当時値をスナップショットする。アプリ本体への全国データ固定同梱はしない。
- 共通定期イベントカタログ（調査・設計中）: トリエンナーレ、ビエンナーレ、地域芸術祭、写真祭、工芸祭、舞台芸術祭等を、場所ではなく`ExperienceEvent`候補としてCloudKit Public Databaseから配信する。シリーズの安定情報と各開催回の会期を分離し、同じ芸術祭へ複数年参加しても対象を増殖させず、`Plan`と`Visit`を年ごとに追加できる形とする。複数会場型イベントは単一の`PlaceMaster`へ変換せず、会場が共通場所カタログに存在する場合だけ関係データで接続する。都道府県は1件以上必須、複数県は配列相当で保持し、会期未発表時は推測せず未発表状態を保存する。初回調査原稿は、公式一次情報で直近開催または次回開催を確認した20シリーズ・20開催回。現時点ではCSV調査原稿であり、CloudKit投入・候補選択UI・差分取得は未実装。

CloudKit互換のため、全モデルで「デフォルト値あり」「uniqueなし」「リレーションoptional」を守る。

## 4. 画面構成

- 組み込み`museum`ジャンルの利用者向け名称は`ミュージアム`とし、従来の`美術展`データをID・関連・画像・予定ごとそのまま引き継ぐ。対象は美術館・博物館・科学館・資料館の展示／イベント。3列統計は`訪れた館 / 鑑賞イベント / 気になる`とし、訪れた館はPlaceMasterまたは入力済み館名の重複なし件数、鑑賞イベントはVisitがある対象の重複なし件数を数える。文書内に残る`美術展`表記はこの`ミュージアム`の展示・イベント文脈を指す。
- 従来の`おでかけ施設`はトップメニュー上で`テーマパーク / 自然・いきもの / その他・未分類`へ分割する。保存カテゴリキーは順に`theme_park / nature_living / outing_facility`。`ExperienceEvent.subTypeKey`の`テーマパーク・遊園地`はテーマパーク、水族館・動物園・植物園・その他の自然系は自然・いきものへ置き、空値・未知キー・その他施設はその他・未分類に残す。既存カテゴリが表示中だった利用者は初回分割時に前2ジャンルも表示し、その他・未分類は初回移行時に該当する対象または予定がなければ隠す。その後の表示設定は利用者の選択を維持する。編集で施設種別を変えた時はEventと紐づくPlanを移動先カテゴリへ揃える。
- 組み込み`random_goods`ジャンルの利用者向け名称は`ランダムグッズ`とする。シリーズ登録時にグッズ種別、メーカー、発売時期、全種類数、画像を保持し、種類数分のラインナップを先に生成する。シリーズ一覧と詳細は`収集済み種類 / 対象種類`、所持、ダブり、未入手、関連支出を自動表示する。種類別に名前・バリエーション・画像・コンプリート対象可否を編集し、入手・購入・交換・譲渡・売却・紛失等の履歴を追加する。初期UIではトレーディングカードを独立ジャンルとして公開しないが、保存モデルは将来同じ仕組みを再利用できる汎用設計とする。
- 3つの施設ジャンルは当面、パークや水族館の明るい空と水面を想起させるパークブルー`#2F7FB8`を共通利用する。ジャンルトップは白ベースに薄い水色のニュアンス、カード枠・統計・ボタン・MAPマーカーを同じ青系で統一し、色分けは実機確認後に検討する。
- ジャンルトップの`Coming Up / Interests`Heroカードは自由メモを表示しない。総合Homeと同じ248pt共通カード高を使い、観劇・映画・美術展のB判アイキャッチと他ジャンルの情報欄を横ページング枠内で欠けさせない。タイトルは総合Homeと同じ2行分を常に確保し、その下へサブタイトルをセカンダリより少し濃い文字色で表示する。日時、会場を優先し、予定は主催者名、記録は評価が入力済みの場合だけ補足情報として表示する。予定Heroは画像・情報欄・12ptの`予定詳細`を同じ`PlanDetailView`へ接続し、10.5ptの`編集`だけを予定編集へ接続する。両ボタンは高さ34pt、アイコンと文言の間隔3ptで総合Homeと揃える。対象一覧タイルは作品・公演単位の`EventDetailView`へ進む。自由メモはそれぞれの詳細で確認する。

画面情報階層と各画面の責務は [`docs/15-画面情報設計.md`](../docs/15-画面情報設計.md) を正本とする。Mystoriumからはデザインではなく、閲覧/編集分離、対象/体験回分離、Draft保存方式を採用する。Home、ジャンルトップ、対象詳細、記録詳細、登録、対象編集、記録編集、予定編集の責務を混在させない。

初期実装は [HomeView.swift](../favorecoAPP/favorecoAPP/Views/HomeView.swift)。

カレンダーの月見出しは端末言語に左右されない `2026年7月` 形式で表示する。予定・チケット一覧は記録/カレンダー右上から開く。

写真の複数選択は1枚ずつ順次ダウンサンプルし、処理件数付きProgressViewを表示する。入力中サムネイルも縮小生成し、選択枚数分の1600px画像を同時フルデコードしない。カバー写真は選べるが、利用者向けの表示比率選択は入力画面に出さず、ジャンル既定値を使う。無料版だけ1記録10枚で停止し、Pro以上は枚数で停止しない。Pro以上で1記録50枚に達した場合は、取り込み・完全バックアップ・初回同期に時間がかかる可能性と現在の概算容量を入力画面へ表示するが、追加操作は妨げない。入力中に削除した保存済み写真は即座に表示、件数、無料枠計算から外す。

記録写真は`思い出 / チケット / グッズ`の用途を持つ。未設定の既存写真は思い出として扱い、思い出写真は従来どおりHero・アイキャッチ・ギャラリー候補に使う。チケット／グッズ写真は画像ごとにOCR文字列と任意金額を保持し、記録詳細では用途別に分けて表示する。写真由来の金額はVisitの合計金額へ自動加算せず、二重計上を避ける。`PhotoBlob.purpose / ocrText / amount`はCloudKit同期とJSON／完全／自動バックアップの対象で、JSON schema 6では旧バックアップに項目がない場合も空文字・0円へ安全に復元する。

DEBUGビルドの設定 > 開発にはStoreKit購入結果/無料版/Pro/Premium/Premium永久版の権利切替を置く。StoreKit購入結果を選ぶとoverrideを解除する。全データ削除は通常の設定 > データ管理に加え、実機試験用に開発欄からも同じ二段階確認画面へ入れる。ローカルストア復旧画面の診断は実ストアとエラー値を変更せず、次回起動時の通常UI、seed、通知移行、自動バックアップだけを停止する。復旧画面内で診断を終了し、再起動して通常状態へ戻す。

- `ContentView`: 初回ジャンル選択が未完了なら `GenreOnboardingView`、完了済みなら `MainTabView` を表示する入口。新規インストール時は現在バージョンを記録し、既存利用者の保存バージョンとMarketing Versionが異なる場合だけ、更新後の最初の起動で更新概要を1回表示する。永続ローカルストアを開けない場合は `LocalStoreRecoveryView` だけを表示し、通常操作を停止した上で再起動とサポートを案内する。
- `GenreOnboardingView`: 初回起動時に、説明オンボーディング（価値訴求 / 記録できる内容 / ジャンル横断 / 安心材料）を短く表示してから、記録したい標準ジャンルをチェック選択する画面。開始ボタンは1件以上選択時のみ有効。権限要求はこの画面では行わず、写真/通知/同期などは必要な機能を使う時に案内する。
- `MainTabView`: SwiftUI標準`TabView`で`Home / 記録 / 追加 / カレンダー / 統計`の5項目を並べるルート。独自カプセル、影、下端スペーサーを重ねず、標準タブバーの背景にはシステムの`ultraThinMaterial`を明示して、ライト画面でも背後の写真の色と明暗が判別できる透過ブラーにする。iOS 26では本文上へ浮く標準タブバー、iOS 18では従来型の半透明タブバーとして表示する。各タブのルートは下端のcontainer Safe Areaを越えて描画し、スクロール末尾に操作余白を確保することで、本文を標準タブバーの背後へ延長しつつ最後の要素を隠さない。`追加`の選択だけBindingで横取りし、現在タブを維持して目的別の追加メニューを表示する。クイック登録は入力項目を限定したExperienceEventを`気になる`状態で保存する。設定を除く主要4画面は、左に省略しない画面名、右に `お知らせベル / プロフィール` を置く共通固定ヘッダーを使う。Homeのブランド名は先頭を大文字にした `Favoreco` とし、34ptのCormorant Garamond Bold、ライト時はアプリアイコン中央のFに合わせた青墨色 `#192735`、ダーク時は可読性を保つ明るい青灰色とする。ほかの画面名は30pt、ベルは22pt、プロフィールは38ptを基準とし、ヘッダーは安全領域内の基準位置から6pt上へ寄せる。ベルとプロフィールは個別に44ptの押下領域を持つが、外側をピルやカードで囲わない。ベルは申込・入金・発券・会員期限をまとめたお知らせ一覧、プロフィールは設定を開く。予定・チケット入口は記録の検索/フィルター下とカレンダーの表示切替横に置き、共通右上操作へ混在させない。
- Homeの`次にやること`の1行カプセルは`アイコン / 7/18(土)|やること|タイトル`の密度で表示する。曜日は半角括弧かつ括弧前余白なし、区切りは左右間隔を詰めた`|`を使う。
- `QuickRegistrationView`: ジャンル/タイトル必須、アイキャッチ/URL/読み取りメモ/短いメモ任意の簡易フォーム。OCRタイトル候補は先頭行ではなく、文字領域の高さ・面積・認識確度から大きく目立つ行を優先する。同じ領域のVision上位候補が拮抗する場合や目立ち度の首位が明確でない場合は自動候補を出さず、上位文字候補とOCR全行から利用者が選ぶ。候補は利用者が押すまでタイトルへ反映せず、OCR全文は選択・コピー可能な確認欄と対象の読み取りメモへ残す。再読取開始時は前回の候補とOCR全文を破棄し、失敗時に古い読み取りメモを保存しない。アイキャッチ確認はトリミングせず全体を表示する。
- `HomeView`: ブランド名は上部左に固定し、右上の共通ベル/プロフィールと同じ段に置く。トップHeroは`PICK UP`とし、未終了のPlan・今日以降のVisitを日時順に最大5件、その後へ最近の過去Visitを最大5件混在表示する。同じVisitにPlanがある場合はPlanを優先して未来候補を二重表示しない。HeroカードはB判`1:1.414`の観劇・映画・美術展ポスターを正式比率のまま収める248pt共通高とし、画像と右側情報・操作をカード内で欠けさせない。横長・正方形など他ジャンルも同じカード高を使い、右側の余白で情報量の差を吸収して操作ボタンを下端へ揃える。横断ミニ統計の「今後の予定」はHero総数ではなく未来Plan/Visitだけを数える。日付変更だけでPlanは暗黙作成しない。Hero直下の`次にやること`は、進行中TicketAttemptごとに現在の申込・締切・当落・入金・発券アクションを1件だけ抽出し、さらに有効な公演準備リストの未完了・期限付き項目を加える。期限超過を先頭、その後を日時順に並べる。Home上の各項目は高さ40ptのカプセル1行に`アイコン / 日付・やること・タイトル`を表示する。Homeは上位3件まで、4件以上は`ほかN件を見る`から全件シートを開き、各行から予定詳細へ進む。TicketAttemptの状態更新または準備項目の完了後は該当項目を自動で外し、次項目を繰り上げる。続く`Ticket Schedule`は全ジャンルの進行中TicketAttemptを対象に、観劇ジャンルの`Ticket Progress`と同じカード・工程タイムライン・文字サイズを使い、次アクション日時が近い順に表示する。複数件はジャンル名・予定日・会場を含む選択チップで切り替え、選択中ジャンル色をカードへ反映する。その下を`Interesting → Coming Up`とし、`Interesting`は全ジャンルの気になる対象と未整理Inboxを3列ポスター・2列ミニ詳細・横長バナーから切替表示する。一覧全体は横スクロールせず、9/8/6件単位の`さらにn件`で下へ展開する。`Coming Up`は未来Plan/Visitを日時順にまとめ、最初の1件と下方向の追加展開で表示する。続いて最近の思い出、最近の記録、統計サマリを表示し、対応セクションは設定でON/OFF可能。ベル内のお知らせは申込期限、会員期限等を保持し、発売・当落・入金・発券の日時到達後または会員期限が近い場合はベルへ赤いマークを出す。最近の思い出/最近の記録カードでは写真、日付、場所、人物・団体、チケット状態、評価、金額、OCR/詳細オプション有無、短いメモをサマリー表示する。非表示対象に紐づくVisitはHome・記録・カレンダー・統計/レポートから除外する。カテゴリカードからカテゴリトップへ遷移する。
- 総合HomeのHeroを除いた本文は、ジャンルトップと同じ方向限定横スワイプを持つ。左スワイプでは画面上の右隣へ進む`総合 → 先頭ジャンル → … → 最終ジャンル → 総合`、右スワイプではその逆順に循環する。左右24ptの画面端、Hero、チケット等の内側横スクロールは対象外とする。最近の思い出は横スクロールせず、最大8件を2列の縦グリッドで表示する。
- Homeの日本語セクション見出しはNoto Serif JP 17pt Boldとし、左上 `Favoreco` と同じ適応ブランド色で表示する。`PICK UP` / `Ticket Schedule` / `Interesting` / `Coming Up`はCormorant Garamond 22pt Semiboldで観劇の英字見出しと揃える。`Interesting`は見出し右のシェブロンで開閉でき、閉じた時は見出しと件数だけ残して0件の空表示・一覧・表示方式切替を畳む。開閉状態はUserDefaultsに保存し、再起動後も引き継ぐ。共通ヘッダーの区切り線はジャンル列の下へ置く。
- Homeの`PICK UP`カードは、内側の約40%を正式なアイキャッチ比率を保つ画像、約60%を予定または最近記録の情報に使い、画像の成り行き高をカード高の基準とする。複数件は左右の見切れを出さない全幅カード単位でページングし、現在位置は件数分の丸で示して選択中だけテーマ色にする。カード下の余白は丸表示に必要な分まで詰める。候補0件時は同等の高さの枠を確保し、枠内に予定追加ボタンを置く。操作列はジャンルトップHeroと同じ34ptカプセル、主ボタン12pt、編集10.5pt、アイコン間隔3ptとし、画像・情報欄・主ボタンを同じ予定詳細または記録詳細へ接続する。
- `FavoView`: 従来の記録タブを置き換える横断アーカイブ。トップは`MY FAVO / STORIES / コレクション / 探す`の順とし、Homeが担う期限・対応事項・直近予定の一覧は置かない。`MY FAVO`は`FavoPin`で人物・作品/体験・場所を最大4件まで混在固定し、編集画面から追加・解除・ドラッグ並べ替えを行う。人物は`FavoriteProfile`が持つ呼び名・推し色・推してからの日数を使い、作品と場所は推し設定を要求しない。対象カードから同じ詳細へ進み、作品は配下Event、場所は同じ`PlaceMaster`を参照するVisitをUUIDで一意化して全件表示する。各対象の次の予定は詳細だけに1件、写真数とVisitの記録済み支出は集計表示し、支出は初期状態で伏せる。`STORIES`は全記録から重複しない最新・最初・過去の同日を最大3件、`コレクション`は実データを持つ写真、よく行く場所、今年の記録、ジャンル横断件数を保存せず自動集計する。各コレクションカードからその条件に該当する重複なしのVisit一覧へ進み、さらに既存の記録詳細を開ける。写真なし、場所なし、今年0件のカードは出さない。人物・団体、場所、タグ・同行者、作品・体験を`探す`の入口にし、現在の検索UIは`すべての記録を検索`として下層の`RecordsView`へ残す。人物の推し詳細は、推してからの日数、次の予定1件、総回数、作品・公演数、初回・最新記録、保存済み写真数、ジャンル別件数、よく行く場所の上位5件、ジャンル横断タイムライン全件、関連する体験記録の金額合計を表示する。場所は`PlaceMaster`を優先し、未接続時だけ会場名の表記ゆれを正規化して集約する。写真は実データを持つ`photo`だけを数え、支出はVisitの金額ユニットだけを合計してTicketAttemptとの二重計上を避ける。集計は`FavoSnapshot`で1描画につき1回作り、EventリンクとVisitリンクの両方から到達する同じVisitをUUIDで重複排除する。既存の予定・記録・人物モデルへ集計値を保存しない。

- `EventDetailView`: 対象詳細。対象の代表写真、カテゴリ、シリーズ、対象メモ、公式URL、記録数、最新日、平均評価、履歴を表示し、対象編集・回追加・各Visit詳細へ遷移できる。代表写真は対象に紐づく全記録の写真から明示選択でき、未指定または選択写真削除時は最新記録のカバー/先頭写真を自動表示する。対象一覧でも同じ解決規則を使う。メニューの「対象を非表示」は`ExperienceEvent.isArchived`だけを切り替え、履歴・写真・予定を変更しない。履歴行は `VisitSummaryRow` を使い、写真、日付、場所、評価、チケット状態、人物・団体、金額、OCR/詳細オプション有無、短いメモを表示する。
- `EditEventView`: 対象自体のタイトル、シリーズ、公式URL、複数SNSリンク、対象メモを編集するフォーム。
- `EditExperienceView`: 保存済み記録の編集フォーム。既存 `ExperienceEvent` + `Visit` を更新する。記録追加と同じユニット単位アコーディオンを使い、長いジャンル入力に備える。人物・団体ユニットでは保存済みリンクの確認/削除と新規追加、チケット・予定ユニットでは状態/座席メモ、写真ユニットでは保存済み写真の確認/削除と新規写真追加、OCR・取込ユニットでは読み取りテキストの編集/追記、金額ユニットでは合計金額、詳細オプションでは自由項目の編集ができる。
- `ExperienceDetailView`: 保存済みVisitの詳細表示。記録詳細では保存した情報を原則すべて見せる。Home/一覧/サマリーカードではカバー写真、ジャンル色/アイコン、タイトル、日付、場所、人物/団体1〜2件、評価またはステータス、短いメモ1行など厳選した情報だけを表示し、詳細画面で写真、人物、チケット、場所、天気、金額、URL、SNS、メモなどへ深掘りできる。全ジャンル共通で上部を画像主役型Heroとし、標準ナビゲーションバーを完全に隠して、Hero写真上へ独自の戻るボタンとジャンル色の右上メニューだけを重ねる。Hero上部の最大420pt帯へ写真を中央aspect fillで収め、縦長写真は天地を切る。写真比率でHero高を押し広げず、画像の色を残して下端だけをジャンルの正規色へ溶かす。ページ全体はジャンル色から深色へ続く暗色基調とし、観劇は`#8B2F45`のワインレッドを面、ゴールドを操作と装飾の差し色に使う。ステータス、戻る、右上メニュー、Hero文字は白と局所コントラストスクリーンで写真の明暗から判読性を守る。対象名を帯の下端と重なる位置へ上詰めし、その下へ左アイキャッチ、右へ日付・上演時間・観劇スタイル・会場・座席・評価を置く。日付と時間は別行とし、時計行はアイコンと`12:00–15:00`形式の値だけを表示して固定の`上演時間`接頭語は付けない。観劇スタイル行もタグアイコンと選択値だけを表示し、固定の`スタイル`接頭語は付けない。値は`Visit.unitFieldsRaw`の`styleNames`へ複数保存し、スタイルだけを入力した場合もJSONを保持する。旧記録は空配列として互換読込する。追加／編集ではイマーシブ演劇・大衆演劇を含む主要候補の複数選択と自由入力を提供し、`ExperienceEvent.subTypeKey`や全ジャンル共通タグとは分ける。対象名から`EventDetailView`へ進める。背景またはアイキャッチに使った思い出写真は直下の写真一覧で重複表示せず、写真不足時はジャンル色グラデーションとジャンルアイコンへフォールバックする。チケット／グッズ写真はHero候補から除外し、用途別セクションへ画像、任意金額、OCR文字列を表示する。感想、残りの思い出写真、チケット、グッズ、観劇キャスト、費用合計の順に表示する。人物画像がなければ活動タグ由来アイコンを円内へ出し、演出・脚本などは`スタッフ・関係者`として従来の詳細表示へ残す。編集とこの記録だけの削除は全ジャンル共通で右上の`…`へまとめ、削除は既存の確認ダイアログを経由する。写真は`byteCount`で存在判定し、ImageIOサムネイルを画面サイズに応じてバックグラウンド生成・NSCache再利用する。複数写真を`UIImage(data:)`で一括フルデコードしない。`Visit.unitFieldsRaw` にOCRテキストがある場合はOCR・取込セクション、自由項目がある場合は詳細オプションセクション、`Visit.outcomeKey` / `seatText` がある場合はチケット状態と座席メモを表示する。費用合計は確保済みTicketAttempt、グッズ写真、遠征ToDoを正本とし、チケット写真と旧`Visit.amount`は二重計上しない代替値として扱う。`カレンダーに追加` から `EKEventEditViewController` を開き、タイトル/日時/場所/座席/状態/金額/メモ/公式URLを下書きとして外部カレンダーへ手動追加できる。
- 記録詳細のHeroは早見として日付・上演時間・観劇スタイル・会場・座席・評価までに限定する。席種は座席欄へ一体入力し、座席メモ、購入元、申込経路、券面、手数料は下部の`チケット・座席`へ集約する。確保済みTicketAttemptがある場合は当選以降だけを表示し、Visit側の状態はTicketAttemptがない場合だけ表示する。基本情報からチケット状態を外し、下部カードとの重複を作らない。
- 記録詳細のHero直下には公演の公式URL・SNSを常設し、未入力でも`未登録`の枠を残す。SNSは`ExperienceEvent.unitFieldsRaw`の`socialLinks`へ1行1リンクの複数値として保存し、同じ公演の各Visitへ自動反映する。その下へ会場名・住所・地図プレビュー・Apple Maps導線を置き、会場未登録時も空状態を表示する。通常の思い出写真は枚数に関係なく3列正方形グリッドへ整列し、1枚時も全幅へ広げない。

## 5. 重要な実装ルール
<!-- 壊すと怖い部分・触る前に必ず読むべき前提 -->
- タイポグラフィは `FavorecoTypography` を通して指定する。日本語UI本文は `Noto Sans JP`、思い出感のある見出しは `Noto Serif JP`、英字表示は `Cormorant Garamond` を使う。いずれもGoogle Fontsの可変TTFを `Resources/Fonts` に同梱し、`FontRegistrar` で起動時登録する。
- 記録追加/編集フォームのカテゴリ別文言は `CategoryRecordTemplate` を通す。フォーム内では入力中にSwiftDataを書かず、DraftState→Save→Modelを守る。
- 記録追加/編集フォームは、Mystorium同様にユニット単位のアコーディオンUIを基本方針とする。ジャンルによって入力項目が長くなるため、標準ユニットは `基本情報`、`人物・団体`、`チケット・予定`、`写真`、`OCR・取込`、`金額`、`公式情報`、`メモ`、`詳細オプション` を基本形にする。会場・住所・日付・種別・評価はイベント/訪問の核なので `基本情報` に含める。写真と半券/OCRは分け、写真はギャラリー/思い出、OCR・取込は半券/チケット/レシート/リスト画像からの読み取りに寄せる。公式URL、SNS投稿リンク、参考URLは `公式情報` としてメモから分離する。御朱印帳サイズと写真比率の連動は `ExperienceGoshuinBookUnitEditor`、自由なラベル/値の追加・削除は `ExperienceAdvancedUnitEditor` を正本UIとし、新規記録・既存対象への回追加・記録編集で共用する。各ユニットは必須/未入力/入力済み/要確認状態が見出しでわかるようにし、保存後は編集画面ではなく記録詳細画面へ戻る。
- ジャンル別の初期有効ユニットは、観劇/ライブ=`基本情報`、`人物・団体`、`チケット・予定`、`写真`、`OCR・取込`、`金額`、`公式情報`、`メモ`。美術展=`基本情報`、`人物・団体`、`チケット・予定`、`写真`、`OCR・取込`、`公式情報`、`メモ`。映画=`基本情報`、`人物・団体`、`写真`、`OCR・取込`、`公式情報`、`メモ`。本=`基本情報`、`人物・団体`、`写真`、`OCR・取込`、`メモ`。酒/御朱印=`基本情報`、`写真`、`OCR・取込`、`メモ`。テーマパーク/おでかけ施設=`基本情報`、`チケット・予定`、`写真`、`OCR・取込`、`金額`、`公式情報`、`メモ`。
- 人物/団体を記録に紐付ける時は、PersonMaster自体の固定種別だけでなく、紐付け側に関係タイプ/役割を持つ。初期セットは `アーティスト`、`出演`、`主演`、`作家`、`作者`、`監督`、`脚本`、`演出`、`原作`、`音楽`、`演奏`、`翻訳`、`キュレーター`、`主催`、`制作`、`出版社`、`ゲスト`、`その他`。同じ人物/団体でも、作品・公演・本・展示ごとに役割が変わる前提で扱う。
- サマリーカードはHome横断では共通ベースに寄せ、ジャンル内ではジャンル別に最適化する。観劇/ライブはタイトル、日付、会場、出演/アーティスト、座席またはチケット状態、開演時間、カバー写真。映画はタイトル、鑑賞日、鑑賞方法/劇場、監督または出演1名、評価、ポスター/写真。本はタイトル、読了日/購入日、作者、読書状態、評価、表紙写真。美術展は展覧会名、訪問日、会場、作家/主催、会期中/終了、キービジュアル/写真。酒は銘柄、飲んだ日、酒蔵/産地、種別、評価、ラベル写真。御朱印は寺社名、いただいた日、都道府県/場所、印種別、御朱印写真。テーマパーク/おでかけは施設名、訪問日、エリア/場所、予定ステータス、写真、同行者/短いメモ。実装後の画面確認で精査する。
- 既存対象への再訪/再鑑賞/再飲は `AddVisitView` で `Visit` のみ追加し、`ExperienceEvent` を重複作成しない。
- 主要ナビは標準`TabView`の`Home / FAVO / 追加 / カレンダー / 統計`5項目 + 右上プロフィール入口。FAVOは推し専用ではなく、人物・場所・作品・体験を「好き」から横断して振り返る入口とする。下部タブに設定は置かず、設定/マイ領域はプロフィールアイコンから開く。

- 通知全体は初期OFFとし、予定・チケットの個別項目は新規利用時に`おすすめ`（申込締切、当落、入金締切、発券、公演前日/当日、期限付き公演準備）を初期値にする。`少なめ`は申込締切・入金締切・公演、`しっかり`はおすすめへ申込開始を加える。FC・会員期限と思い出リマインダーは初期OFFでプリセット対象外。通知タイプ別ON/OFFとタイプ別時刻は `AppStorageKeys.notification...` 系に保存する。iOS通知許可は起動直後には求めず、通知を有効化する操作やチケット/予定作成時に、用途を説明してから求める。予定通知IDは `plan.<planID>.performance.previousDay/sameDay`、公演準備は`plan.<planID>.preparation.<taskID>.reminder`、チケット通知IDは `ticket.<attemptID>...`、FC・会員期限通知IDは `ticketAccount.<accountID>.expiry...` 形式で管理し、再予約時は古い候補IDを削除してから追加する。締切系の既定は前日と1時間前、公演準備の既定は期限前日20時、FC・会員期限は30日前/7日前/当日朝を予約する。予定・チケット編集、申込単体編集、公演準備更新時も保存後に再予約し、申込削除/予定削除/準備完了・停止時は対象通知をキャンセルする。
- 記録・入力補助の初期値は、デフォルト記録日=今日、デフォルトジャンル=最後に使ったジャンル、記録追加後=詳細を開く、写真追加=カメラを開く、写真圧縮=85%、URL/OCR/Map/天気/入力補助辞書=ONとする。接続済みはデフォルトジャンル、記録追加後、写真追加初期動作、写真圧縮、URL候補、OCR取込、Map検索、天気自動付与、入力補助辞書。写真追加は設定側を主要ボタンとして先に表示し、カメラ/写真ライブラリのもう一方も補助ボタンとして常に選べる。カメラ画像とライブラリ画像は同じ長辺1600px・選択品質・メタデータ非引継ぎ・1記録10枚上限・初回カバー設定を通す。デフォルトジャンルは「最後に保存成功したジャンル」または「Homeから開いた/ジャンル画面で選択中のジャンル」を別々に保持し、中央＋のジャンル候補で既定ジャンルを先頭に明示する。対象ジャンルが非表示なら先頭の有効ジャンルへフォールバックし、他ジャンルも同じダイアログから選べる。記録追加後は新規記録/回追加の保存成功後に保存済みVisit詳細を開き、詳細の「完了」で追加シートを閉じる。「一覧に戻る」選択時は保存後すぐ追加シートを閉じる。編集保存には適用しない。
- 天気自動付与は設定ON時、観劇/美術展/ライブ/おでかけ施設のうち、2021年8月1日以降かつ訪問日が経過した記録へ適用する。登録住所から確定した座標または保存座標をWeatherKitへ渡し、日別履歴のSF Symbols名、最高/最低気温、取得日時、Apple Weather法的情報URLをVisitの補助フィールドへキャッシュする。一覧では日付アイコンを天気へ置換し、詳細では最高/最低気温とApple Weatherリンクを表示する。通信、履歴データ、権限/entitlementの失敗は記録保存を妨げず、次回詳細表示時に再試行する。編集で日付/座標が同じなら保存済み天気を保持し、変わった場合は破棄して再取得する。
- 写真メタデータ削除はON固定とする。保存時に写真ファイル内のGPS、撮影端末、Exif等のメタデータを削除し、訪問日・会場・天気などは写真ファイルではなく `Visit` 側の記録データとして管理する。

- 動画は容量とiCloud同期負荷が大きいため、v1では外部リンク保存、Photos参照、サムネイル表示までを基本にする。動画ファイルをアプリ内へコピーする保存、動画のCloud同期、一括バックアップ同梱は明示ON/容量警告つき、またはv2以降の有料候補とする。カバー写真指定は無料で扱う。

- 写真/カバー比率は `VisitUnitFields.eyecatchAspectRatioKey` に保存し、書籍は作品側の `ExperienceEvent.unitFieldsRaw` にも同じ比率キーを保持する。比率候補は `EyecatchAspectRatio` で、正方形、映画/観劇`1:1.414`、書籍のコミック`112:174`・新書`103:182`・ハードカバー`127:188`・文庫`105:148`・雑誌`182:257`・写真集/画集`210:297`、横長ラベル`4:3`、御朱印帳標準`11:16`を持つ。新規書籍の既定はハードカバー、既存の未選択書籍は従来の`1:1.45`を維持する。比率が決まるHome、一覧、ジャンルトップ、対象/記録詳細、登録・編集、入力サムネイルでは全ジャンルの画像を中央基準のaspect fillで枠いっぱいに表示し、比率差は左右または天地をトリミングする。ジャンルトップのバナー表示だけはジャンル比率に関係なく正方形へ切り抜く。御朱印見開きは`3:2`枠を使い、専用トップでも同じfillルールを適用する。

- Homeは全ジャンル横断ダッシュボード、ジャンルトップは対象ライブラリとして役割を分ける。Homeとジャンルトップは共通ヘッダー直下へ同じ`GenreNavigationStrip`を縦スクロール外で固定し、アイコンなし・明朝体の`総合 / 観劇 / 美術展 / ライブ...`を横1段スクロールで表示する。先頭の`総合`は全件一覧ではなく、予定・チケット・気になる対象・最近の記録をジャンル横断でまとめるHomeを意味する。ジャンル間は細い縦線で区切り、選択中は塗り面ではなくジャンル色の文字と2pt下線で示し、ナビゲーション領域の下端に全幅1pt線を置く。ジャンルトップ中央のジャンル名は見出し表示だけとし、直下の横スクロールメニューと重複する▼・タップ選択・ジャンル一覧ダイアログは持たない。御朱印以外のトップHeroは未来予定を先、同じ対象の重複を除いた興味ありを後へ置き、合計最大10件を同じ候補列で表示する。観劇・美術展・LIVE・映画の`Coming Up`は`Interests`直下、対象一覧前で直近1件を低い横長カード表示し、2件以上は下方向へ展開する。Homeにも横断`Coming Up`を`Interesting`直下へ置き、未来Plan/Visitを同様に1件＋下方向展開で表示する。両方の予定行は左から`24pt月/日＋12pt日本語曜日の2段 / 64pt正方形アイキャッチ / ジャンル・タイトル・会場の間隔2ptの3段`とし、同じ共通Viewを使う。英字見出しは`Interests` / `Interesting` / `Coming Up` / `Productions` / `Ticket Progress` / `Ticket Schedule`でCormorant Garamond 22pt Semiboldへ揃える。Homeの`Ticket Schedule`と各ジャンルの`Ticket Progress`は同じ表示部品を使い、Homeだけ全ジャンル横断、ジャンルトップは現在ジャンル限定とする。ジャンルトップの左右スワイプ認識Viewはジャンル内容の遷移Viewより外側に1個だけ維持し、切替アニメーション中に新旧認識器を重複させない。ジャンルトップの左ブランドロゴはHomeの約80%に当たる27ptとし、中央ジャンル名を主役にする。Home背景は公式サイトに合わせたクリーム基調の淡い桃色/青緑/ベージュと、可読性を落とさない微細な紙粒子を重ねる。青緑のニュアンスは残しつつ桃色・薄赤ベージュをやや優勢にする。ジャンルトップ背景は現在ジャンル色を上部へ薄く反映する。ジャンルトップのミニ統計はジャンル別の対象・体験済み・気になる語彙を使い、有効ユニット数は表示しない。
- ジャンルトップは各ジャンルを連続した「章」として扱う。上部ジャンル列のタップと中央ジャンル名の一覧からは同じ画面内で内容を切り替え、現在の縦位置を維持する。横スワイプ判定範囲はHeroを除いた統計、チケット進捗、対象一覧、章末とする。Hero内のカルーセル操作はHero自身へ渡し、公演選択、チケット情報など横方向に内容を持つ内側ScrollView上でも内側のスクロールを優先する。本文の横操作は、最初の速度が縦より明確に横優勢な場合だけ開始する方向限定パンとし、縦操作では横認識器を開始せずScrollViewへ渡す。横移動が16pt以上になってから本文内のNavigationLink/Buttonを一時無効化し、単純タップや微小な指ブレでは無効化しない。終了100ms後に戻してタイル誤遷移を防ぐ。左右24pt以内から始まる画面端操作もジャンル切替に使わない。方向限定パンは本文マーカーがScrollView階層へ追加された時点で確実に取り付け、`GeometryReader`でマーカーを本文と同じ幅・高さへ明示して開始位置判定が0サイズにならないようにする。ドラッグ量は本文専用コンテナ内で管理し、ドラッグ中に親画面のCategoryTopSnapshotや一覧を毎フレーム作り直さない。切替はページ単位で吸着し、内容と背景色を方向に合わせた約0.22秒の減速型アニメーションで切り替える。表示中ジャンルの180ms後に前後ジャンルの先頭サムネイルだけをバックグラウンド生成し、キャッシュ済み画像は初回描画から同期表示する。ページ末尾には前後ジャンルの写真またはアイコン、ジャンル名、代表対象を示す「別の章へ」カードを置き、章末カードからの移動だけは次ジャンルの先頭へ戻す。大量データは無限スクロールで全件を一度に描画せず、初期件数を絞り、将来のギャラリー/棚も「さらに表示」で段階的に展開する。
- 観劇ジャンルトップは専用の舞台調外観とし、黒へ沈むワインレッド背景、8pt角丸のカード、細いゴールド枠、アイボリー/ゴールド文字を使う。左上の`Favoreco`は金〜銅のグラデーションとする。ポスター/アイキャッチだけは角丸なし、B判相当の`1:1.414`固定枠へ中央基準のaspect fillで表示する。テーマ設定がライト固定でも観劇の章内はこの専用配色を維持し、Homeと他ジャンルには波及させない。
- ライブジャンルトップは中央見出しとジャンル列を`LIVE`表記とし、紺から黒紺へ沈む専用背景、ティールのアクセント、青白い本文色を使う。観劇と同じ3列連結ミニ統計、`Ticket Progress`、`Coming Up`を表示する。気になる対象は`Interests`へ分け、参加または記録済みの公演一覧は`Live History`とする。両一覧は横長バナーへ固定し、切替ボタンを置かず、タイトル・シリーズ/ツアー・日時・会場・出演者または主催を優先表示する。ライト固定でもライブの章内は専用配色を維持する。
- 美術展・LIVE・おでかけ施設のジャンルトップは対象一覧の後、章末カードの前へ訪問場所MAPを表示する。Visit自身の座標を優先し、なければ紐づくPlaceMasterの座標を使う。同じPlaceMaster、または同一座標・施設名の記録を1地点へ集約し、訪問回数が多いほどジャンル色のマーカーを濃く大きくする。マーカーをタップすると対象座標を中央へ移して市街地相当までズームし、選択リングを表示する。MAP上部の`全国 / 北海道 / 東北 / 関東 / 中部 / 近畿 / 中国 / 四国 / 九州・沖縄`は単一選択のチェック式簡易フィルターとし、選択地域へカメラを移動する。詳細検索では施設名・会場名・住所・メモ等の自由ワードと都道府県を指定できる。位置未設定の記録はMAPへ出さず件数を案内し、MAPのパン・ズームと地域フィルターの横操作はジャンル切替スワイプより優先する。MAPはVisit座標だけでも表示できるが、個人PlaceMasterの新規生成は名称に加えて住所から都道府県を特定できる場合だけとし、既存の座標のみ記録を自動でマスター化・書換しない。
- ジャンルトップのミニ統計は表示部品と集計定義を分離する。アイコン左・情報右の低い横型は再利用可能な`MiniStatisticsBlock`とし、`threeColumns / fourColumns`の2フォーマットを持つ。3列はアイコンと数値を一段階広く大きく、4列は3桁の件数が収まるコンパクト値を使う。各ジャンル側がフォーマットを選び、タイトル・値・単位・SF Symbolを持つ`MiniStatisticsItem`を組み立てて渡す。映画は4列で`総鑑賞数 / 年間数 / 観たい / レビュー`を表示し、レビュー値は評点済み鑑賞記録の全期間平均とする。既存の連結パネルも隣接項目間を細い縦線で区切る。観劇は1枚の3列連結枠を維持した専用`TheaterStatisticsPanel`を使い、アイコンを出さず、各列を`12pt基準の名称 / 33pt基準の数値＋単位 / 短い補足`の3段で組む。補足は`総作品数 / 総観劇数 / 観劇予定`。名称と数値の間は1pt、数値と補足の間は2pt、列の上下余白は10ptを基準とする。背景は上側のわずかに明るいワインから下側の黒に近いワインへ、色差を抑えた縦グラデーションで移し、右上の淡い金光を重ねる。0.7ptの金グラデーション枠と区切り線、アイボリー文字で描く。観劇のチケット進捗見出しは`Ticket Progress`、作品・公演一覧見出しは`Productions`とし、22pt基準の英字ディスプレイ明朝で揃える。その他は現在ジャンル色で枠と区切り線を描く。本=`総冊数`、おでかけ施設=`総来園数`とし、英語の`トータル`表記は使わない。

- **ライフサイクル状態・予定・申込・記録を混ぜない**。クイック登録は`気になる`状態の `Event`、手動のあとで整理する項目は `InboxItem`、予定/公演回は `Plan`、申込1件は `TicketAttempt`、実体験は `Visit`、記録下書きは `MemoryDraft`/`VisitDraft` として責務分離する。既存の `Visit.outcomeKey` / `seatText` は暫定互換として残すが、複数先行・落選履歴・名義別当選率・通知更新の正本は `TicketAttempt` に移す。
- **Apple Kit依存は境界で閉じ込める**。SwiftUI/SwiftData/CloudKit/MapKit/WeatherKit/EventKit/Vision/StoreKit はiOS実装で使ってよいが、ドメインモデル・状態遷移・Smart Add解析結果・同期DTO・写真/OCR/カレンダー/通知/課金の抽象インターフェースはApple APIへ直結させない。将来Androidを完全に捨てないため、UI/OS連携以外の移植可能性を保つ。
- **スキーマはCloudKit互換3条件で設計する**（①全プロパティにデフォルト値 ②unique禁止・fetch-first upsertで代替 ③リレーション全optional）。同期公開はv2だが構造はv1から `docs/07-CloudKit同期設計リファレンス.md` に従う。違反すると同期導入時にスキーマ再設計になる
- **写真はPhotoBlob（externalStorage）方式**（`docs/09-CloudKit写真ストレージ仕様.md` が正本）。画面はrelativePath文字列だけ扱う・動画は `video_` プレフィックスでファイル保存・zipバックアップは `_SUPPORT` 同梱＋再起動必須・static ユーティリティは `nonisolated` 明示（Swift 6）
- 端末内のデータは絶対に消えない設計を守る（同期・バックアップのどの失敗ケースでも壊れる方向に倒さない）
- **サブ種別レイヤーの下地を最初から通す**（酒の種別=日本酒/ビール/ウィスキー…）。〈カテゴリ→サブ種別→軸/スペック/OCR辞書〉の2階層マスターで、種別ごとにプリセットが切り替わる仕組みをv1で実装（中身は日本酒のみ）。後付けすると既存記録の移行が発生し重いリファクタになる（Mystorium enum変更クラッシュの教訓）。詳細: docs/08 §4-6★★
