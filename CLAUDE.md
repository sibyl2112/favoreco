# グローバル運用ガイド

> **役割**: このファイルはプロジェクト横断の運用ルール。特定プロジェクトの実装仕様は各プロジェクトの `CLAUDE.md` を参照する。
> **最終更新**: （新規アプリ作成時に更新）

---

## 1. ドキュメント構成

| ファイル | 役割 |
|---------|------|
| `CLAUDE.md`（このファイル） | プロジェクト横断の運用ルール・作業前提 |
| `APP_NAME/CLAUDE.md` | 各プロジェクトの実装仕様の正本 |
| `docs/新規アプリ立ち上げ.md` | 新規アプリの立ち上げ（アイデア精査→コンセプト確定→MVP範囲確定）の手順書・対話台本 |
| `docs/開発プロセス.md` | アプリ開発の進め方（要件→仕様→指示書→実装→レビュー→記録）の手順書 |
| `docs/レビュー観点.md` | 事前レビュー（精査）用チェックリスト |
| `docs/project-log.md` | 変更履歴・意思決定ログ |
| `docs/_templates/` | コピーして使う各種テンプレート雛形 |

**原則**: 実装仕様はプロジェクトの `CLAUDE.md` に書く。ここには実装詳細を重複して書かない。

---

## 2. 更新ルール

- コード・仕様・UI・保存ルールに変更を入れた時は、同じ作業内で `docs/project-log.md` に追記する
- `docs/project-log.md` には以下を残す
  - 日付
  - 変更概要
  - 変更意図
  - 主な変更ファイル
  - 確認結果（実機 / ビルド）
  - 残課題
- プロジェクトの `CLAUDE.md` は「現在どうなっているか」の正本として保つ
- `docs/project-log.md` は「なぜそうしたか / どう遷移したか」を残す

## 3. 作業の原則

### 3.1 作る前に合意する

変更に着手する前に「何を作るか」「完成の定義」を明確にする。曖昧なまま実装を始めない。

- 複数ステップの作業は、最初にやることリストを提示してユーザーと合意してから進める
- 「完成」の基準が曖昧な場合は、先に確認する
- 各変更に対して「これが満たされたらOK」という検証可能な条件を先に書き出す

> 例：「ギャラリーの空白タイルを修正する」→ 完成条件：画像のない項目がグリッドに枠を残さず、前後のタイルが詰まって表示される

### 3.2 実装と検証を分離する

自分が書いたコードを自分で「OK」にしない。作成後に必ず別の視点で検証する。

**変更報告に必ず含めること：**
- 変更したファイルの一覧
- 各ファイルで何を変えたか（1行要約）
- その変更が影響する画面・機能の一覧
- 既知のリスクや副作用の可能性

**検証として行うこと：**
- コード変更後、変更箇所を読み直して意図通りか確認する
- UIフレームワーク固有の落とし穴を確認する（例: SwiftUI なら型の不一致・ビュー階層への影響）
- グリッド・リスト系の変更：空データ、0件、1件、大量データの各パターンを想定する
- アニメーション・ジェスチャー系の変更：他のジェスチャーやシステム動作（画面端スワイプ等）との干渉を確認する

### 3.3 段階的に作る

大きな変更を一度にやらない。機能単位・画面単位で区切って、確認を挟みながら進める。

- 1つの変更が3ファイル以上に及ぶ場合は、先に影響範囲を一覧で提示する
- 5ファイル以上に及ぶ場合は、作業を分割してステップごとに確認を挟む
- デザイン系は選択肢を出す → 選んでもらう → 実装、の順序を守る
- 途中経過を見せて方向修正できるようにする

### 3.4 評価基準を具体化する

「良い」「悪い」ではなく、具体的な基準で判断する。

- 「見やすいか」→「フォントサイズ・コントラスト比・余白は基準を満たしているか」
- 「使いやすいか」→「主要アクションに2タップ以内で到達できるか」
- 「直ったか」→「再現手順をなぞって問題が起きないか」
- 主観的な判断が必要な場面では、選択肢を提示してユーザーに決めてもらう

### 3.5 現行仕様を確認してから着手する

各プロジェクトの `CLAUDE.md` で現在の実装状態を確認してからコードに触る。古い前提や記憶に基づいて実装しない。

- 対象ファイルの現在のコードを読んでから変更する（前回のセッションの記憶に頼らない）
- 関連するモデル定義・View構造を確認してから手を動かす

## 4. 品質チェックリスト

変更を完了とする前に、以下を確認する。

- [ ] 変更意図と実際の変更が一致しているか
- [ ] 変更したファイル・関数を読み直して、論理的な誤りがないか
- [ ] 影響を受ける他の画面・機能に副作用がないか（影響範囲を列挙したか）
- [ ] エッジケースを考慮したか（空データ、nil、0件、型の不一致）
- [ ] プロジェクトの `CLAUDE.md` の記述と矛盾していないか
- [ ] `docs/project-log.md` に記録したか
- [ ] ユーザーが検証できる形で結果を提示したか（変更ファイル一覧＋確認手順）

---

## 5. NGが出た時のルール

ユーザーが問題を報告した場合：

1. **まず原因を特定する** — 「直す」前に「なぜそうなったか」を明らかにする
2. **原因を報告してから修正する** — 「直しました」ではなく「原因は○○でした。△△に変更します」の順序
3. **同種の問題を他にも探す** — 1箇所で見つかった問題は、同じパターンが他にもないか確認する
4. **再発防止を考える** — 同じ失敗を繰り返さないための知見を `docs/project-log.md` に残す

## 7. 作業環境

- **ユーザーの Mac 作業ディレクトリ**: `/Users/<ユーザー>/Documents/APP_NAME`
- Claude は Linux 環境でコードを編集し、GitHub ブランチにプッシュする
- ユーザーは Mac 側でプルしてビルドする

### コード反映手順（毎回ユーザーに提示すること）

変更をプッシュしたら、必ず以下のコマンドをそのまま提示する：

```bash
cd /Users/<ユーザー>/Documents/APP_NAME
git pull origin <ブランチ名>
```

- `cd` を省略しない（ホームディレクトリで実行するとエラーになる）
- ブランチ名は作業中のブランチを明記する
